不同浏览器中 URL 的最大长度是多少？

Arya

2023-02-16 18:37

问题 1： 每个浏览器的 URL 的最大长度是多少？

问题 2： 最大 URL 长度是 HTTP 规范的一部分吗？



解答

简短回答 - 实际上限制为 2000 个字符

如果您将 URL 保持在 2000 个字符以内，它们几乎可以在客户端和服务器软件的任何组合中使用。

如果您的目标是特定浏览器，请参阅下文了解有关特定限制的更多详细信息。

更长的答案 - 首先，标准......

RFC 2616（超文本传输​​协议 HTTP/1.1）第 3.2.1 节说

HTTP 协议不对 URI 的长度设置任何先验限制。服务器必须能够处理它们所服务的任何资源的 URI，并且如果它们提供可以生成此类 URI 的基于 GET 的形式，则应该能够处理无限长度的 URI。如果 URI 比服务器可以处理的长（请参阅第 10.4.15 节），服务器应该返回 414（请求 URI 太长）状态。

该 RFC 已被RFC7230废弃，RFC7230是 HTTP/1.1 规范的更新。它包含类似的语言，但也继续暗示这一点：

在实践中发现了对请求行长度的各种特殊限制。建议所有 HTTP 发送方和接收方至少支持 8000 个八位字节的请求行长度。

...和现实

标准 就是这么说的。实际上， boutell.com上有一篇文章（链接转到 Internet Archive backup）讨论了哪些浏览器和服务器实现将支持 。 执行摘要是：

极长的 URL 通常是错误的。 超过 2,000 个字符的 URL 将无法在大多数流行的网络浏览器中使用。 如果您想让您的站点为大多数 Internet 用户服务，请不要使用它们。

（注意：这是引用自2006 年 撰写的一篇文章，但在 2015 年 IE 的使用率下降意味着较长的 URL 确实 适用于大多数人。但是，IE 仍然存在局限性......）

Internet Explorer 的限制...

IE8 的最大 URL 长度是 2083 个字符，看来IE9 也有类似的限制。

我测试过 IE10，地址栏只接受 2083 个字符。你可以 点击 一个比这个长的网址，但是地址栏仍然只会显示这个链接的2083个字符。

IE Internals 博客上有一篇很好的文章，其中介绍了一些背景知识。

有混合报告 IE11 支持更长的 URL - 请参阅下面的评论。鉴于一些人报告了问题，一般建议仍然有效。

搜索引擎喜欢 URL < 2048 个字符...

请注意，允许站点将可用页面通知搜索引擎的站点地图协议在 URL 中有 2048 个字符的限制。如果您打算使用站点地图，我们已经为您确定了一个限制！（请参阅下面Calin-Andrei Burloiu 的回答）

2010 年还有一些关于搜索引擎抓取和索引的最大 URL 长度的研究。他们发现限制是 2047 个字符，这似乎与站点地图协议规范有关。但是，他们还发现 Google SERP工具无法处理超过 1855 个字符的 URL。

CDN 有限制

CDN 还对 URI 长度施加了限制，并且在达到这些限制时将返回一个414 Too long request，例如：

  • 快8Kb
  • 云端8Kb
  • 云火炬16Kb

（感谢 timrs2998 在评论中提供该信息）

这些信息是最新的吗？

这是一个很受欢迎的问题，由于最初的研究已有 14 年的历史，我会尽量保持最新：截至 2021 年 1 月 ，该建议仍然有效。尽管 IE11 可能会接受更长的 URL，但旧 IE 安装的普遍存在加上搜索引擎的限制意味着保持在 2000 个字符以下是最好的一般策略。

在 JavaScript 比较中应该使用哪个等号运算符（== vs ===）？

Arya

2023-02-16 18:37

这个问题的答案是 社区的努力。编辑现有答案以改进这篇文章。它目前不接受新的答案或互动。

我正在使用JSLint来遍历 JavaScript，它会返回许多建议，以便在执行诸如比较内部的操作时将==（两个等号）替换为===（三个等号）idSele_UNVEHtype.value.length == 0``if。

==替换为有性能优势吗===？

任何性能改进都将受到欢迎，因为存在许多比较运算符。

如果不进行类型转换，性能会比吗==？



解答

严格相等运算符 ( ===) 的行为与抽象相等运算符 (== ) 相同，只是不进行类型转换，并且类型必须相同才能被视为相等。

参考：Javascript 教程：比较运算符

运算符将 在进行任何必要的类型转换后==比较是否相等。运算符不会 进行 转换，因此如果两个值的类型不同，将简单地返回 __=== ****===``false。两者都同样快。

引用 Douglas Crockford 的优秀JavaScript: The Good Parts，

JavaScript 有两组相等运算符：===and !==，以及它们的邪恶双胞胎==and !=。好的方法会按照您期望的方式工作。如果两个操作数的类型相同且值相同，则===producestrue和!==produces false。当操作数属于同一类型时，邪恶双胞胎会做正确的事情，但如果它们属于不同类型，它们就会尝试强制转换值。他们这样做的规则是复杂且难以记忆的。这些是一些有趣的案例：

'' == '0'           // false
0 == ''             // true
0 == '0'            // true

false == 'false'    // false
false == '0'        // true

false == undefined  // false
false == null       // false
null == undefined   // true

' \t\r\n ' == 0     // true

传递性的缺乏令人担忧。我的建议是永远不要使用邪恶的双胞胎。相反，始终使用===和!==。刚才显示的所有比较都是false由===操作员生成的。

更新：

@Casebash在评论和@Phillipe Laybaert 关于对象的回答中提出了一个很好的观点。对于对象，==并且===行为一致（特殊情况除外）。

var a = [1,2,3];
var b = [1,2,3];

var c = { x: 1, y: 2 };
var d = { x: 1, y: 2 };

var e = "text";
var f = "te" + "xt";

a == b            // false
a === b           // false

c == d            // false
c === d           // false

e == f            // true
e === f           // true

特殊情况是当您将原语与计算结果相同的对象进行比较时，由于其toStringorvalueOf方法。例如，考虑将字符串基元与使用String构造函数创建的字符串对象进行比较。

"abc" == new String("abc")    // true
"abc" === new String("abc")   // false

这里==运算符正在检查两个对象的值并返回true，但===看到它们不是同一类型并返回false。哪一个是正确的？这实际上取决于您要比较的内容。我的建议是完全绕过这个问题，只是不要使用String构造函数从字符串文字创建字符串对象。

参考
http://www.ecma-international.org/ecma-262/5.1/#sec-11.9.3


更多建议

使用==运算符（ 相等 ）

true == 1; //true, because 'true' is converted to 1 and then compared
"2" == 2;  //true, because "2" is converted to 2 and then compared

使用===运算符（ 身份 ）

true === 1; //false
"2" === 2;  //false

这是因为 相等运算符==执行类型强制转换，这意味着解释器会在比较之前隐式尝试转换值。

另一方面， 身份运算符===不进行类型强制转换，因此在比较时不转换值。

如何更改远程 Git 存储库的 URI (URL)？

Arya

2023-02-16 18:36

我在我的硬盘驱动器（本地）上克隆的 USB 密钥上有一个 repo（来源）。我将“起源”移动到 NAS 并成功地测试了从这里克隆它。

我想知道我是否可以在“本地”设置中更改“来源”的 URI，以便它现在将从 NAS 中提取，而不是从 USB 密钥中提取。

目前，我可以看到两种解决方案：

  • 将所有内容推送到 USB 来源，然后再次将其复制到 NAS（由于新提交到 NAS 来源意味着大量工作）；

  • 添加一个新的遥控器到“本地”并删除旧的（我担心我会打破我的历史）。



解答

你可以

git remote set-url origin new.git.url/here

看git help remote。您还可以.git/config在那里编辑和更改 URL。

除非你做了一些非常愚蠢的事情，否则你不会有丢失历史记录的危险（如果你担心，只需复制你的回购协议，因为你的回购协议 就是 你的历史记录。）


更多建议

git remote -v
# View existing remotes
# origin  https://github.com/user/repo.git (fetch)
# origin  https://github.com/user/repo.git (push)

git remote set-url origin https://github.com/user/repo2.git
# Change the 'origin' remote's URL

git remote -v
# Verify new remote URL
# origin  https://github.com/user/repo2.git (fetch)
# origin  https://github.com/user/repo2.git (push)
