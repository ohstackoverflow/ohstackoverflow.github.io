如何检查 Bash 脚本中的程序是否存在？

Arya

2023-02-16 20:20

我将如何以返回错误并退出或继续执行脚本的方式验证程序是否存在？

看起来应该很容易，但它一直难倒我。



解答

回答

POSIX 兼容：

command -v <the_command>

使用示例：

if ! command -v <the_command> &> /dev/null
then
    echo "<the_command> could not be found"
    exit
fi

对于 Bash 特定环境：

hash <the_command> # For regular commands. Or...
type <the_command> # To check built-ins and keywords

解释

避免which。它不仅是一个你启动的外部进程，因为它做的很少（意味着内置函数hash，type或者command更便宜），你还可以依靠内置函数来实际做你想做的事情，而外部命令的效果很容易从系统到系统。

为什么要关心？

  • 许多操作系统都有一个which甚至 没有设置退出状态的 a ，这意味着它if which foo甚至不会在那里工作并且 总是 报告foo存在，即使它不存在（请注意一些 POSIX shell 似乎也这样做hash）。
  • 许多操作系统会做一些which自定义的和邪恶的事情，比如改变输出，甚至挂钩到包管理器中。

所以，不要使用which. 而是使用其中之一：

command -v foo >/dev/null 2>&1 || { echo >&2 "I require foo but it's not installed.  Aborting."; exit 1; }



type foo >/dev/null 2>&1 || { echo >&2 "I require foo but it's not installed.  Aborting."; exit 1; }



hash foo 2>/dev/null || { echo >&2 "I require foo but it's not installed.  Aborting."; exit 1; }

（次要注意事项：有些人会建议2>&-是相同的2>/dev/null但更短 - 这是不正确的 。2>&-关闭 FD 2 会在程序尝试写入 stderr 时导致程序 出错 ，这与成功写入它并丢弃输出有很大不同（而且很危险！））

如果你的 hash bang 是/bin/sh那么你应该关心 POSIX 说的是什么。typeandhash的退出代码没有被 POSIX 很好地定义，并且hash当命令不存在时被认为成功退出（还没有看到这个type）。command的退出状态由 POSIX 明确定义，因此使用起来可能是最安全的。

但是，如果您的脚本使用了POSIX 规则bash，那么 POSIX 规则就不再重要了，而且type使用hash起来非常安全。type现在只有 a-P可以搜索PATH并且hash有副作用，命令的位置将被散列（以便下次使用它时更快地查找），这通常是一件好事，因为您可能检查它是否存在以便实际使用它.

作为一个简单的例子，这里有一个函数，gdate如果存在则运行，否则date：

gnudate() {
    if hash gdate 2>/dev/null; then
        gdate "$@"
    else
        date "$@"
    fi
}

具有完整功能集的替代方案

您可以使用scripts-common来满足您的需要。

要检查是否安装了某些东西，您可以执行以下操作：

checkBin <the_command> || errorMessage "This tool requires <the_command>. Install it please, and then run this tool again."


更多建议

以下是检查命令是否存在$PATH 以及 是否可执行的可移植方法：

[ -x "$(command -v foo)" ]

例子：

if ! [ -x "$(command -v git)" ]; then
  echo 'Error: git is not installed.' >&2
  exit 1
fi

需要进行可执行检查，因为如果在$PATH.

另请注意，如果与可执行文件同名的不可执行文件较早存在于 中$PATH，破折号将返回前者，即使后者将被执行。这是一个错误，违反了 POSIX 标准。[错误报告] [标准]
编辑 ：这似乎从 dash 0.5.11 (Debian 11) 开始修复。

此外，如果您正在寻找的命令已被定义为别名，这将失败。

在 JavaScript 中是否有标准函数来检查 null、undefined 或空白变量？

Arya

2023-02-16 20:20

是否有一个通用的 JavaScript 函数来检查变量是否具有值并确保它不是undefined或null？我有这段代码，但我不确定它是否涵盖所有情况：

function isEmpty(val){
    return (val === undefined || val == null || val.length <= 0) ? true : false;
}



解答

您可以只检查 变量 是否有truthy值。这意味着

if (value) {
    // do something..
}

将 评估true如果value不是： ****

  • 无效的
  • 不明确的
  • 钠
  • 空字符串 ("")
  • 0
  • 错误的

上面的列表代表了falsyECMA-/Javascript 中所有可能的值。在部分的规范中找到它ToBoolean。

此外，如果您不 知道 变量是否存在（也就是说，如果它被 声明 ），您应该与操作员核实typeof。例如

if (typeof foo !== 'undefined') {
    // foo could get resolved and it's defined
}

如果你能确定至少声明了一个 变量truthy，你应该直接检查它是否有如上所示的值。


更多建议

检查值是否未定义或为空的详细方法是：

return value === undefined || value === null;

您也可以使用==运算符，但这需要知道所有规则：

return value == null; // also returns true if value is undefined

为什么打印“B”比打印“#”慢得多？

Arya

2023-02-16 20:19

我生成了1000x的两个矩阵1000：

第一个矩阵：O和#。
第二个矩阵：O和B。

使用以下代码，第一个矩阵需要 8.52 秒才能完成：

Random r = new Random();
for (int i = 0; i < 1000; i++) {
    for (int j = 0; j < 1000; j++) {
        if(r.nextInt(4) == 0) {
            System.out.print("O");
        } else {
            System.out.print("#");
        }
    }

   System.out.println("");
 }

使用此代码，第二个矩阵需要 259.152 秒才能完成：

Random r = new Random();
for (int i = 0; i < 1000; i++) {
    for (int j = 0; j < 1000; j++) {
        if(r.nextInt(4) == 0) {
            System.out.print("O");
        } else {
            System.out.print("B"); //only line changed
        }
    }

    System.out.println("");
}

截然不同的运行时间背后的原因是什么？

正如评论中所建议的那样，打印只System.out.print("#");需要7.8871几秒钟，而System.out.print("B");给出still printing....

正如其他人指出它对他们正常工作一样，我尝试了Ideone.com，例如，两段代码以相同的速度执行。

测试条件：

  • 我从 Netbeans 7.2 运行了这个测试，输出到它的控制台
  • 我用来System.nanoTime()测量



解答

纯粹的猜测 是您使用的终端尝试进行文字换行而不是字符换行，并将其B视为单词字符而不#是非单词字符。因此，当它到达一行的末尾并寻找一个可以断行的地方时，它#几乎会立即看到并愉快地在那里断行；而对于B，它必须继续搜索更长时间，并且可能有更多文本要换行（这在某些终端上可能很昂贵，例如，输出退格键，然后输出空格以覆盖被换行的字母）。

但这纯粹是猜测。


更多建议

我在 Eclipse 和 Netbeans 8.0.2 上进行了测试，两者都使用 Java 版本 1.8；我用来System.nanoTime()测量。

蚀：

我 在这两种情况下得到了相同的时间 ——大约 1.564 秒 。

网豆：

  • 使用“#”： 1.536 秒
  • 使用“B”： 44.164 秒

因此，看起来 Netbeans 在打印到控制台时性能不佳。

经过更多研究，我意识到问题是Netbeans 的最大缓冲区的换行（它不限于System.out.println命令），由这段代码演示：

for (int i = 0; i < 1000; i++) {
    long t1 = System.nanoTime();
    System.out.print("BBB......BBB"); \\<-contain 1000 "B"
    long t2 = System.nanoTime();
    System.out.println(t2-t1);
    System.out.println("");
}

每次迭代的时间结果小于 1 毫秒，但 每五次迭代 除外，当时时间结果约为 225 毫秒。类似（以纳秒为单位）：

BBB...31744
BBB...31744
BBB...31744
BBB...31744
BBB...226365807
BBB...31744
BBB...31744
BBB...31744
BBB...31744
BBB...226365807
.
.
.

等等..

概括：

  1. Eclipse 与“B”完美配合
  2. Netbeans有个换行问题可以解决（因为eclipse中不会出现这个问题）（不用在B("B")后面加空格）。
