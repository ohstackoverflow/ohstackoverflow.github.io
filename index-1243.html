关于设计模式：什么时候应该使用单例？

Arya

2023-02-18 04:56

荣耀的全局变量 - 成为荣耀的全局类。有人说打破了面向对象的设计。

给我场景，而不是使用单例的好旧记录器。



解答

在寻求真相的过程中，我发现实际上很少有使用单例的“可接受”理由。

在 Internet 上反复出现的一个原因是“日志记录”类（您提到过）。在这种情况下，可以使用 Singleton 而不是类的单个实例，因为日志类通常需要被项目中的每个类一遍又一遍地重复使用。如果每个类都使用这个日志类，依赖注入就会变得很麻烦。

日志记录是“可接受的”单例的一个具体示例，因为它不会影响代码的执行。禁用日志记录，代码执行保持不变。启用它，一样。Misko 在Singletons 的根本原因中以下列方式表示，“这里的信息以一种方式流动：从您的应用程序到记录器。即使记录器是全局状态，因为没有信息从记录器流到您的应用程序，所以记录器是可以接受的。”

我敢肯定还有其他正当理由。Alex Miller 在“我讨厌的模式”中谈到服务定位器和客户端 UI 也可能是“可接受的”选择。

在 Singleton 阅读更多信息 我爱你，但你让我失望了。


更多建议

单身人士候选人必须满足三个要求：

  • 控制对共享资源的并发访问。
  • 系统的多个不同部分将请求访问资源。
  • 只能有一个对象。

如果您提议的 Singleton 只有这些要求中的一两个，那么重新设计几乎总是正确的选择。

例如，打印机后台处理程序不太可能从多个地方（打印菜单）调用，因此您可以使用互斥体来解决并发访问问题。

一个简单的记录器是可能有效的单例的最明显的例子，但这可能会随着更复杂的日志方案而改变。

python中字典的深拷贝

Arya

2023-02-18 04:56

我想dict在 python 中制作 a 的深层副本。不幸的是，该.deepcopy()方法不存在于dict. 我怎么做？

>>> my_dict = {'a': [1, 2, 3], 'b': [4, 5, 6]}
>>> my_copy = my_dict.deepcopy()
Traceback (most recent calll last):
  File "<stdin>", line 1, in <module>
AttributeError: 'dict' object has no attribute 'deepcopy'
>>> my_copy = my_dict.copy()
>>> my_dict['a'][2] = 7
>>> my_copy['a'][2]
7

最后一行应该是3。

我希望修改my_dict不会影响快照my_copy。

我怎么做？该解决方案应与 Python 3.x 兼容。



解答

怎么样：

import copy
d = { ... }
d2 = copy.deepcopy(d)

Python 2 或 3：

Python 3.2 (r32:88445, Feb 20 2011, 21:30:00) [MSC v.1500 64 bit (AMD64)] on win32
Type "help", "copyright", "credits" or "license" for more information.
>>> import copy
>>> my_dict = {'a': [1, 2, 3], 'b': [4, 5, 6]}
>>> my_copy = copy.deepcopy(my_dict)
>>> my_dict['a'][2] = 7
>>> my_copy['a'][2]
3
>>>


更多建议

dict.copy() 是字典的浅拷贝函数
id 是内置函数，可以为您提供变量的地址

首先，您需要了解“为什么会出现这个特定问题？”

In [1]: my_dict = {'a': [1, 2, 3], 'b': [4, 5, 6]}

In [2]: my_copy = my_dict.copy()

In [3]: id(my_dict)
Out[3]: 140190444167808

In [4]: id(my_copy)
Out[4]: 140190444170328

In [5]: id(my_copy['a'])
Out[5]: 140190444024104

In [6]: id(my_dict['a'])
Out[6]: 140190444024104

键“a”的两个字典中存在的列表地址指向同一位置。
因此，当您更改 my_dict 中列表的值时，my_copy 中的列表也会更改。

问题中提到的数据结构的解决方案：

In [7]: my_copy = {key: value[:] for key, value in my_dict.items()}

In [8]: id(my_copy['a'])
Out[8]: 140190444024176

或者您可以使用上面提到的 deepcopy。

Android drawable目录可以包含子目录吗？

Arya

2023-02-18 04:55

在 Android SDK 文档中，所有与 @drawable/my_image xml 语法一起使用的示例都直接处理存储在我项目的 res/drawable 目录中的图像。

我想知道在 drawable 目录中创建子目录是否明确不行。

例如，如果我有以下目录布局：

res/drawable
-- sandwiches
  -- tunaOnRye.png
  -- hamAndSwiss.png
-- drinks
  -- coldOne.png
  -- hotTea.png

我可以将金枪鱼沙拉三明治的图像引用为 @drawable/sandwiches/tunaOnRye

或者我是否必须在 drawable 目录中保持层次结构平坦。



解答

不，资源机制不支持 drawable 目录中的子文件夹，所以是的 - 您需要保持该层次结构平坦。

您显示的目录布局将导致没有图像可用。

从我自己的实验来看，似乎在文件夹中有一个包含任何项目的子文件夹res/drawable将导致资源编译器失败——阻止文件R.java被正确生成。


更多建议

我正在使用的解决方法（Android 本身似乎喜欢的解决方法）基本上是用下划线代替正斜杠，因此您的结构看起来像这样：

sandwich_tunaOnRye.png
sandwich_hamAndSwiss.png
drink_coldOne.png
drink_hotTea.png

该方法要求您在命名时一丝不苟，并且不会让文件本身更容易争论（如果您决定饮料和三明治真的应该全部是“” food，您必须进行大量重命名而不是简单地重命名将它们移动到目录）；但是与等效的文件夹结构相比，您的编程逻辑的复杂性并没有受到 太大影响。

这种情况确实很糟糕。Android 是一个混合了美妙和糟糕的设计决策的包。我们只能希望后一部分能尽快被淘汰 :)
