JavaScript 中的“导出默认值”是什么？

Arya

2023-02-17 11:45

文件：SafeString.js

// Build out our basic SafeString type
function SafeString(string) {
  this.string = string;
}

SafeString.prototype.toString = function() {
  return "" + this.string;
};

export default SafeString;

export default我以前从未见过。是否有任何等效的东西export default可以更容易理解？



解答

它是 ES6 模块系统的一部分，在此处进行了描述。该文档中有一个有用的示例，还有：

如果模块定义了默认导出：

// foo.js
export default function() { console.log("hello!") }

然后您可以通过省略大括号来导入该默认导出：

import foo from "foo";
foo(); // hello!

更新：截至 2015 年 6 月，模块系统在 ECMAScript 2015 规范的 §15.2中定义，尤其是语法在ECMAScript 2015 规范的§15.2.3export中定义。


更多建议

export default用于从脚本文件导出单个类、函数或原语。

出口也可以写成

export default function SafeString(string) {
  this.string = string;
}

SafeString.prototype.toString = function() {
  return "" + this.string;
};

这用于在另一个脚本文件中导入此函数

在app.js 中说，你可以

import SafeString from './handlebars/safe-string';

关于出口的一点

顾名思义，它用于从脚本文件或模块中导出函数、对象、类或表达式

Utiliites.js

export function cube(x) {
  return x * x * x;
}
export const foo = Math.PI + Math.SQRT2;

这可以导入并用作

应用程序.js

import { cube, foo } from 'Utilities';
console.log(cube(3)); // 27
console.log(foo);    // 4.555806215962888

或者

import * as utilities from 'Utilities';
console.log(utilities.cube(3)); // 27
console.log(utilities.foo);    // 4.555806215962888

当使用 export default 时，这就简单多了。脚本文件只导出一件事。 立方体.js

export default function cube(x) {
  return x * x * x;
};

并用作 App.js

import Cube from 'cube';
console.log(Cube(3)); // 27

Android getResources().getDrawable() 弃用 API 22

Arya

2023-02-17 11:44

使用新的 android API 22getResources().getDrawable()现已弃用。现在最好的方法是只使用getDrawable().

什么改变了？



解答

你有一些选择来以正确的（和 未来的证明 ）方式处理这种弃用，这取决于你加载的是哪种可绘制对象：

A） 具有 主题属性的可绘制对象

ContextCompat.getDrawable(getActivity(), R.drawable.name);

您将按照 Activity 主题的指示获得样式化的 Drawable。这可能就是您所需要的。

B) 没有 主题属性的可绘制对象

ResourcesCompat.getDrawable(getResources(), R.drawable.name, null);

您将以旧方式获得无样式的可绘制对象。请注意：ResourcesCompat.getDrawable()未 弃用 ！

EXTRA) 具有来自 另一个 主题的主题属性的可绘制对象

ResourcesCompat.getDrawable(getResources(), R.drawable.name, anotherTheme);


更多建议

编辑：请参阅我关于该主题的博客文章以获得更完整的解释

您应该改用支持库中的以下代码：

ContextCompat.getDrawable(context, R.drawable.***)

使用此方法等同于调用：

if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.LOLLIPOP) {
    return resources.getDrawable(id, context.getTheme());
} else {
    return resources.getDrawable(id);
}

从 API 21 开始，您应该使用getDrawable(int, Theme)方法而不是getDrawable(int)，因为它允许您获取与给定屏幕密度/主题的特定资源 ID 关联的可绘制对象。调用已弃用的getDrawable(int)方法等同于调用getDrawable(int, null).

如何修复 Node.js 中的“ReferenceError: primordials is not defined”

Arya

2023-02-17 11:44

我已经通过“npm install”安装了 Node.js 模块，然后尝试gulp sass-watch在命令提示符下安装。之后，我得到了以下回复。

[18:18:32] Requiring external module babel-register
fs.js:27
const { Math, Object, Reflect } = primordials;
                                  ^

ReferenceError: primordials is not defined

我以前试过这个gulp sass-watch：

npm -g install gulp-cli



解答

我遇到了同样的错误。我怀疑您使用的是 Node.js 12 和 Gulp.js 3。该组合不起作用： _Gulp.js 3 在 Node.js 12 上损坏

2324_

1 月的先前解决方法也不起作用： 更新到 Node.js 11.0.0 后运行 Gulp.js 退出并显示“ReferenceError：internalBinding is not defined”#2246

解决方案：升级到 Gulp.js 4 或降级到 Node.js 的早期版本。


更多建议

我们在更新依赖于 Node.js 12+ 的遗留项目时遇到了同样的问题gulp@3.9.1。

gulp@3.9.1这些修复使您能够通过覆盖graceful-fs版本来使用 Node.js 12+ ^4.2.10。

如果你正在使用 pnpm

pnpm支持覆盖一些依赖版本。为此，您应该pnpm在 package.json 文件中添加一个部分：

{
  "pnpm": {
    "overrides": {
      "graceful-fs": "^4.2.10"
    }
  }
}

如果您使用的是纱线 v1

Yarn v1支持将包解析为定义的版本。您需要将一个resolutions部分添加到您的package.json：

{
  "resolutions": {
    "graceful-fs": "^4.2.10"
  }
}

感谢@jazd以这种方式解决问题。

如果你正在使用 npm

运行此命令以了解您使用的是哪个版本的 Node.js：

node -v

它将返回一个版本号，<major>.<minor>.<patch>例如18.11.0.

如果您的版本是 v16.14.0 或更高版本，那么您可以通过在 package.json 文件中graceful- fs添加一个overrides部分来覆盖版本：

{
  "overrides": {
    "graceful-fs": "^4.2.10"
  }
}

否则，您需要用作npm-force-resolutions预安装脚本，以便能够graceful-fs通过更改您的 package.json 文件来覆盖的版本，如下所示：

{
  "scripts": {
    "preinstall": "npx npm-force-resolutions"
  },
  "resolutions": {
    "graceful-fs": "^4.2.10"
  }
}

npm-force-resolutions将在完成之前更改package-lock.json文件以设置graceful- fs为所需的版本。install

如果您在项目中使用自定义.npmrc文件并且它包含代理或自定义注册表，您可能需要更改npx npm-force- resolutions为，因为默认情况下npx --userconfig .npmrc npm-force- resolutions某些版本npx不使用当前文件夹文件。.npmrc

问题的根源

gulp@3.9.1 这个问题源于依赖于graceful-fs@^3.0.0哪个 monkeypatches Node.js模块的事实fs。

在 11.15 版本之前，它一直与 Node.js 一起使用（这是一个来自开发分支的版本，不应在生产中使用）。

graceful-fs@^4.0.0不再使用 monkeypatch Node.jsfs模块，这使其与 Node.js > 11.15 兼容（已测试并可与版本 12、14 和 16 一起使用）。

请注意，这不是一个长期的解决方案，但当您没有时间更新到gulp@^4.0.0.
