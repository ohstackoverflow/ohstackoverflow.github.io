如何恢复隐藏的未提交更改

Arya

2023-02-17 08:36

我的开发分支中有一些未提交的更改，我使用 隐藏了它们git stash，但是在这些隐藏的更改中有一些非常重要的更改。有什么办法可以取回这些更改吗？

此外，从那以后，我对隐藏的代码文件进行了一些更改。

如果可能的话，我是否有机会将隐藏的更改检索到新分支？



解答

简单问题的简单答案是git stash apply

只需检查您想要更改的分支，然后git stash apply. 然后使用git diff看看效果。

完成所有更改后 -apply看起来不错并且您确定不再需要藏匿处 - 然后 用git stash drop它来摆脱它。

我总是建议使用git stash apply而不是git stash pop. 不同之处在于，apply将 stash 留在周围以便于重试apply，或查看等。如果pop能够提取 stash，它会立即提取drop它，如果您随后意识到要将它提取到其他地方（在不同的分支中），或 with --index，或类似的，这不是那么容易。如果你apply， 你 可以选择什么时候去drop。

不管怎样，这都是很小的问题，对于 Git 的新手来说，应该是差不多的。（你可以跳过所有其余部分！）

如果您正在做更高级或更复杂的事情怎么办？

至少有三四种不同的“使用 git stash 的方式”。以上是“方法 1”，“简单方法”：

  1. 您从一个干净的分支开始，正在进行一些更改，然后意识到您在错误的分支中进行更改。您只想将现在所做的更改“移动”到另一个分支。

这是上面描述的简单情况。运行git stash save（或普通的git stash，同样的事情）。检查其他分支并使用git stash apply. 这使 Git 可以使用 Git 相当强大的合并机制合并您之前的更改。 仔细检查结果 （使用git diff），看看您是否喜欢它们，如果喜欢，请使用git stash dropdrop the stash。你完成了！

  1. 您开始了一些更改并将它们存储起来。然后你切换到另一个分支并开始更多更改，忘记你有隐藏的。

现在您想保留甚至移动 这些 更改， 并 应用您的存储。

实际上git stash save，您可以再次git stash进行“堆栈”更改。如果你这样做，你有两个 stashes，一个只是被称为stash— 但你也可以写stash@{0}— 和一个拼写的stash@{1}。使用git stash list（随时）查看所有内容。最新的总是编号最小的。当你git stash drop，它会丢弃最新的，而那个是stash@{1}移动到堆栈的顶部。如果你有，甚至更多，原来的那个stash@{2}变成了stash@{1}，依此类推。

您apply也可以，然后drop是特定的存储：git stash apply stash@{2}，依此类推。删除特定的藏品只会重新编号较高的藏品。同样，没有数字的那个也是stash@{0}。

如果你堆积了很多藏品，它会变得相当混乱（是我想要的藏品stash@{7}还是它stash@{4}？等等，我刚刚推了另一个，现在他们是 8 和 5？）。我个人更喜欢将这些更改转移到新的分支，因为分支有名称，而且cleanup-attempt-in- December对我来说比stash@{12}. （该git stash命令采用可选的保存消息，这些可以提供帮助，但不知何故，我所有的藏品都以 named 结尾WIP on branch。）

  1. （超高级）git stash save -p您在运行git add. 您在隐藏索引/暂存区中有一个版本，在工作树中有另一个（不同的）版本。你想保留这一切。所以现在你使用, 有时会失败：git rm``git stash save``git stash apply --index

    Conflicts in index.  Try without --index.

  2. 您正在使用git stash save --keep-index以测试“将提交的内容”。这个超出了这个答案的范围；请参阅其他 StackOverflow 答案。

对于复杂的情况，我建议首先从“干净”的工作树开始，提交您现在所做的任何更改（如果您愿意，可以在新分支上）。这样，您正在应用它们的“某处”就没有其他内容了，您将只是尝试隐藏的更改：

git status               # see if there's anything you need to commit
                         # uh oh, there is - let's put it on a new temp branch
git checkout -b temp     # create new temp branch to save stuff
git add ...              # add (and/or remove) stuff as needed
git commit               # save first set of changes

现在你处于一个“干净”的起点。或者它可能更像这样：

git status               # see if there's anything you need to commit
                         # status says "nothing to commit"
git checkout -b temp     # optional: create a new branch for "apply"
git stash apply          # apply stashed changes; see below about --index

要记住的主要事情是“隐藏” 是 一个提交，它只是一个有点“有趣/奇怪”的提交，而不是“在一个分支上”。该apply操作查看提交更改的内容，并尝试在您现在所在的任何地方重复它。藏匿处仍会在那里（apply保留它），因此您可以多看它，或者认为这是错误的地方apply并以不同的方式重试，或其他任何方式。

任何时候你有一个藏匿处，你都可以使用它git stash show -p来查看藏匿处内容的简化版本。（此简化版本仅查看“最终工作树”更改， 而不 查看单独恢复的已保存索引更改。）没有--index命令的命令现在只是尝试在您的工作树中进行 相同的更改。git stash apply``--index __

即使您已经进行了一些更改，也是如此。该apply命令很乐意将存储应用于 修改后的 工作树（或者至少尝试应用它）。例如，您可以这样做：

git stash apply stash      # apply top of stash stack
git stash apply stash@{1}  # and mix in next stash stack entry too

您可以在此处选择“应用”顺序，选择特定的存储以按特定顺序应用。但是请注意，每次您基本上都在执行“git merge”，并且正如合并文档警告的那样：

不鼓励运行带有重要的未提交更改的 git merge：虽然可能，但它可能会使您处于一种在发生冲突时难以退出的状态。

如果您从一棵干净的树开始 并且只进行了几个git apply操作，则很容易退出：使用git reset --hard返回到干净状态，并更改您的apply操作。（这就是为什么对于这些​​复杂的情况，我建议首先从一个干净的工作树开始。）

那最坏的情况呢？

假设你正在做很多高级 Git 的东西，你已经做了一个存储，并且想要git stash apply --index，但是不再可能应用保存的存储，--index因为自从你保存它以来分支已经分叉太多。

这是git stash branch为了什么。

如果你：

  1. 检查您在执行原始操作时所进行的 确切提交stash，然后
  2. 创建一个新分支，最后
  3. git stash apply --index

重新创建更改的尝试肯定 会 奏效。这是做什么的。（然后它会删除存储，因为它已成功应用。）git stash branch _newbranch_

关于--index（这到底是什么？）

所做的--index解释起来很简单，但内部有点复杂：

  • 当您有更改时，您必须在 -inggit add之前（或“暂存”）它们commit。
  • 因此，当您运行 时git stash，您 可能 同时编辑了文件foo和zorg，但只暂存了其中一个。
  • 因此，当您要求取回藏匿物时，如果它git add是added东西而不是 未 git add添加的东西，那可能会很好。也就是说，如果您在执行之前被add-edfoo但没有返回，那么拥有完全相同的设置可能会很好。上演过的，还要再上演；修改但未上演的内容应再次修改但未上演。zorg``stash

标志--indextoapply试图以这种方式进行设置。如果你的工作树是干净的，这通常就可以了。但是，如果您的工作树已经有 stuff add-ed，您可以看到这里可能存在一些问题。如果您省略--index，则该apply操作不会尝试保留整个暂存/未暂存设置。相反，它只是调用 Git 的合并机制，使用“存储包”中的工作树提交。如果您不关心保留已上演/未上演的内容，那么省略--index它可以更容易地git stash apply完成它的工作。

git checkout -b newbranch

jQuery 禁用/启用提交按钮

Arya

2023-02-17 08:35

我有这个 HTML：

<input type="text" name="textField" />
<input type="submit" value="send" />

我怎样才能做这样的事情：

  • 当文本字段为空时，应该禁用提交 (disabled="disabled")。
  • 当在文本字段中键入内容以删除禁用的属性时。
  • 如果文本字段再次变空（文本被删除），则应再次禁用提交按钮。

我试过这样的事情：

$(document).ready(function(){
    $('input[type="submit"]').attr('disabled','disabled');
    $('input[type="text"]').change(function(){
        if($(this).val != ''){
            $('input[type="submit"]').removeAttr('disabled');
        }
    });
});

…但它不起作用。有任何想法吗？



解答

问题在于只有当焦点从输入移开时才会触发更改事件（例如，有人单击输入或从中切换）。尝试使用 keyup 代替：

$(document).ready(function() {
     $(':input[type="submit"]').prop('disabled', true);
     $('input[type="text"]').keyup(function() {
        if($(this).val() != '') {
           $(':input[type="submit"]').prop('disabled', false);
        }
     });
 });


更多建议

$(function() {
  $(":text").keypress(check_submit).each(function() {
    check_submit();
  });
});

function check_submit() {
  if ($(this).val().length == 0) {
    $(":submit").attr("disabled", true);
  } else {
    $(":submit").removeAttr("disabled");
  }
}

灵活的盒子布局

随着CSS Flexible Box的出现，许多网页设计师的噩梦1都得到了解决。最hacky的之一，垂直对齐。 现在即使在未知的高度 也是可能的。

“二十年的布局黑客即将结束。也许不是明天，但很快，我们的余生。”

— CSS 传奇人物Eric Meyer在W3Conf 2013

Flexible Box（或简称Flexbox），是一种专门为布局目的而设计的新型布局系统。规范指出：

Flex 布局表面上类似于块布局。它缺少许多可用于块布局的更复杂的以文本或文档为中心的属性，例如浮动和列。作为回报，它获得了简单而强大的工具，用于以网络应用程序和复杂网页通常需要的方式分配空间和对齐内容。

在这种情况下它有什么帮助？好吧，走着瞧。

垂直对齐的列

使用 Twitter Bootstrap 我们有.rows 有一些.col-*s。我们需要做的就是将想要的.row2显示为一个flex 容器框，然后将其所有的flex item（列）按align-items属性垂直对齐。

此处示例 （请仔细阅读评论）

<div class="container">
    <div class="row vertical-align"> <!--
                    ^--  Additional class -->
        <div class="col-xs-6"> ... </div>
        <div class="col-xs-6"> ... </div>
    </div>
</div>



.vertical-align {
    display: flex;
    align-items: center;
}

澄清align-items: center

8.3 横轴对齐：align-items属性

弹性项目可以在弹性容器当前行的交叉轴上对齐，类似于justify-content但在垂直方向。align- items设置所有弹性容器项目的默认对齐方式，包括匿名弹性项目。

align-items: center;通过 center 值，flex item的 margin box 在该行内的交叉轴上居中。

大警报

重要说明 #1： Twitter Bootstrap 不会指定width超小型设备中的列，除非您将其中一个.col-xs-#类指定给列。

因此，在这个特定的演示中，我使用了.col-xs-*类以便在移动模式下正确显示列，因为它width明确指定了列的。

但也可以通过更改为特定屏幕尺寸 来关闭 Flexbox 布局。例如：display: flex;``display: block;

/* Extra small devices (767px and down) */
@media (max-width: 767px) {
    .row.vertical-align {
        display: block; /* Turn off the flexible box layout */
    }
}

或者您可以 仅 在特定屏幕尺寸上指定，.vertical-align ****如下所示：

/* Small devices (tablets, 768px and up) */
@media (min-width: 768px) {
    .row.vertical-align {
        display: flex;
        align-items: center;
    }
}

在那种情况下，我会采用@KevinNelson的方法。

重要说明 #2： 为简洁起见省略了供应商前缀。Flexbox 语法在此期间发生了变化。新的书面语法不适用于旧版本的网络浏览器（但不像 Internet Explorer 9 那样旧！Internet Explorer 10 及更高版本支持 Flexbox）。

这意味着您还应该display: -webkit-box在生产模式下使用供应商前缀的属性等。

如果您在演示中单击 “切换编译视图” ，您将看到 CSS 声明的前缀版本（感谢Autoprefixer）。

垂直对齐内容的全高列

正如您在前面的演示中看到的那样，列（弹性项目）不再与其容器（弹性容器框。即元素.row）一样高。

这是因为使用属性center值align-items。默认值是stretch项目可以填满父元素的整个高度。

为了解决这个问题，您display: flex;也可以添加到列中：

此处示例 （再次注意评论）

.vertical-align {
  display: flex;
  flex-direction: row;
}

.vertical-align > [class^="col-"],
.vertical-align > [class*=" col-"] {
  display: flex;
  align-items: center;     /* Align the flex-items vertically */
  justify-content: center; /* Optional, to align inner flex-items
                              horizontally within the column  */
}
