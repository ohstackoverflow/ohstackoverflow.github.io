如何在 node.js 中发出 HTTP POST 请求？

Arya

2023-02-17 05:48

如何在 node.js 中发出带有数据的出站 HTTP POST 请求？



解答

下面是使用 node.js 向 Google Compiler API 发出 POST 请求的示例：

// We need this to build our post string
var querystring = require('querystring');
var http = require('http');
var fs = require('fs');

function PostCode(codestring) {
  // Build the post string from an object
  var post_data = querystring.stringify({
      'compilation_level' : 'ADVANCED_OPTIMIZATIONS',
      'output_format': 'json',
      'output_info': 'compiled_code',
        'warning_level' : 'QUIET',
        'js_code' : codestring
  });

  // An object of options to indicate where to post to
  var post_options = {
      host: 'closure-compiler.appspot.com',
      port: '80',
      path: '/compile',
      method: 'POST',
      headers: {
          'Content-Type': 'application/x-www-form-urlencoded',
          'Content-Length': Buffer.byteLength(post_data)
      }
  };

  // Set up the request
  var post_req = http.request(post_options, function(res) {
      res.setEncoding('utf8');
      res.on('data', function (chunk) {
          console.log('Response: ' + chunk);
      });
  });

  // post the data
  post_req.write(post_data);
  post_req.end();

}

// This is an async file read
fs.readFile('LinkedList.js', 'utf-8', function (err, data) {
  if (err) {
    // If this were just a small part of the application, you would
    // want to handle this differently, maybe throwing an exception
    // for the caller to handle. Since the file is absolutely essential
    // to the program's functionality, we're going to exit with a fatal
    // error instead.
    console.log("FATAL An error occurred trying to read in the file: " + err);
    process.exit(-2);
  }
  // Make sure there's data before we post it
  if(data) {
    PostCode(data);
  }
  else {
    console.log("No data to post");
    process.exit(-1);
  }
});

我更新了代码以显示如何从文件而不是硬编码字符串发布数据。它使用异步fs.readFile命令来实现这一点，在成功读取后发布实际代码。如果有错误，则抛出错误，如果没有数据，则进程以负值退出以指示失败。

Git：找到两个分支的最近共同祖先

Arya

2023-02-17 05:48

如何找到两个 Git 分支的最近共同祖先？



解答

你正在寻找git merge-base。用法：

$ git merge-base branch2 branch3
050dc022f3a65bdc78d97e2b1ac9b595a924c3f2


更多建议

git diff master...feature

显示当前（可能是多次提交）功能分支的所有新提交。

man git-diff记录：

git diff A...B

是相同的：

git diff $(git merge-base A B) B

但更...容易输入和记忆。

正如Dave 所HEAD提到的，可以省略的特例。所以：

git diff master...HEAD

是相同的：

git diff master...

如果当前分支是，这就足够了feature。

最后，请记住顺序很重要！这样做git diff feature...master会显示 on masternot on 的变化feature。

我希望更多的 git 命令能够支持这种语法，但我认为它们不会。有些甚至有不同的语义...：Git 提交范围中双点“..”和三点“...”有什么区别？

解答

当您想将 lambda 表达式视为表达式树并查看它们的内部而不是执行它们时。例如，LINQ to SQL 获取表达式并将其转换为等效的 SQL 语句并将其提交给服务器（而不是执行 lambda）。

从概念上讲， 完全Expression<Func<T>>不同于. 表示一个几乎是指向方法的指针，并表示lambda 表达式的 树数据结构。 此树结构 描述了 lambda 表达式的作用 ，而不是实际执行的操作。它基本上保存有关表达式、变量、方法调用等组成的数据（例如，它保存诸如此 lambda 是某个常量 + 某个参数的信息）。您可以使用此描述将其转换为实际方法（使用）或使用它执行其他操作（如 LINQ to SQL 示例）。将 lambda 表达式视为匿名方法和表达式树的行为纯粹是编译时的事情。 Func<T>``Func<T>``delegate``Expression<Func<T>> ****Expression.Compile

将有效地编译为一个 IL 方法，该方法什么都不获取并返回 10。

我添加了一个新手答案，因为这些答案似乎让我难以理解，直到我意识到它是多么简单。有时，您的期望是它很复杂，以至于您无法“全神贯注”。

在我尝试一般地使用 LINQ-to-SQL 时遇到一个非常烦人的“错误”之前，我不需要了解其中的区别：

public IEnumerable<T> Get(Func<T, bool> conditionLambda){
  using(var db = new DbContext()){
    return db.Set<T>.Where(conditionLambda);
  }
}

在我开始在更大的数据集上出现 OutofMemoryExceptions 之前，这一直很有效。在 lambda 内部设置断点让我意识到它正在逐行遍历我表中的每一行，以寻找与我的 lambda 条件匹配的项。这让我困惑了一会儿，因为到底为什么要把我的数据表当作一个巨大的 IEnumerable 而不是像它应该的那样执行 LINQ-to-SQL？它也在我的 LINQ-to-MongoDb 对应物中做完全相同的事情。

修复只是变成Func<T, bool>，Expression<Func<T, bool>>所以我用谷歌搜索为什么它需要一个Expression而不是Func，最后在这里。

表达式只是将委托转换为关于自身的数据。 所以a => a + 1变成类似“在左侧有一个int a。在右侧你加 1 到它。” 就是这样。 你现在可以回家了。它显然比那更结构化，但这基本上就是一个表达式树的真正含义——没有什么可以让你费解的。

理解了这一点，就会清楚为什么 LINQ-to-SQL 需要Expression, 而 aFunc是不够的。Func它没有一种进入自身的方法，无法了解如何将其转换为 SQL/MongoDb/其他查询的本质。你看不到它是在做加法、乘法还是减法。您所能做的就是运行它。Expression，另一方面，允许您查看委托内部并查看它想做的所有事情。这使您能够将委托转换为您想要的任何内容，例如 SQL 查询。Func没有用，因为我的 DbContext 看不到 lambda 表达式的内容。因此，它无法将 lambda 表达式转换为 SQL；但是，它做了下一个最好的事情，并在我的表中的每一行中迭代了条件。

编辑：应 John Peter 的要求阐述我的最后一句话：

IQueryable 扩展了 IEnumerable，因此 IEnumerable 的方法类似于Where()获取接受Expression. 当您将 an 传递Expression给它时，您会保留 IQueryable 作为结果，但是当您传递 a 时Func，您将退回到基础 IEnumerable 并且您将得到一个 IEnumerable 作为结果。换句话说，在没有注意到的情况下，您已经将数据集变成了要迭代的列表，而不是要查询的内容。除非您真正深入了解签名，否则很难注意到差异。
