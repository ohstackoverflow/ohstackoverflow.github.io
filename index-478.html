代表数字 7 的 8 位看起来像这样：

00000111

设置了三个位。

确定 32 位整数中设置位数的算法是什么？



解答

这被称为“汉明重量”、“流行计数”或“横向加法”。

一些 CPU 有一条内置指令来执行此操作，而另一些 CPU 则有作用于位向量的并行指令。像 x86 这样的指令popcnt（在支持它的 CPU 上）对于单个整数几乎肯定是最快的。其他一些体系结构可能有一个慢速指令，它通过微编码循环实现，每个循环测试一个位（ 需要引用 - 如果它存在，硬件 popcount 通常很快。）。

“最佳”算法实际上取决于您使用的是哪个 CPU 以及您的使用模式。

您的编译器可能知道如何做一些对您正在编译的特定 CPU 有益的事情，例如C++20std::popcount()或 C++ std::bitset<32>::count()，作为访问内置/内部函数的可移植方式（请参阅此问题的另一个答案）。但是您的编译器为没有硬件 popcnt 的目标 CPU 选择的回退可能不是您的用例的最佳选择。或者您的语言（例如 C）可能不会公开任何可以使用特定于 CPU 的 popcount 的可移植函数（如果有的话）。

不需要（或受益于）任何硬件支持的便携式算法

如果您的 CPU 有很大的缓存并且您在紧密循环中执行大量这些操作，则预填充表查找方法可以非常快。然而，由于“缓存未命中”的代价，它可能会受到影响，CPU 必须从主内存中获取一些表。（分别查找每个字节以保持表格较小。）如果您希望对连续范围的数字进行 popcount，对于 256 个数字的组，只有低字节会发生变化，这非常好。

如果您知道您的字节大部分是 0 或大部分是 1，那么对于这些​​场景有有效的算法，例如，在循环中使用 bithack 清除最低集，直到它变为零。

我相信下面是一个非常好的通用算法，称为“并行”或“可变精度 SWAR 算法”。我已经用类似 C 的伪语言表达了这一点，您可能需要调整它以适用于特定语言（例如，在 C++ 中使用 uint32_t，在 Java 中使用 >>>）：

GCC10 和 clang 10.0 可以识别这种模式/习惯用法，并在可用时将其编译为硬件 popcnt 或等效指令，让您两全其美。( https://godbolt.org/z/qGdh1dvKK )

int numberOfSetBits(uint32_t i)
{
     // Java: use int, and use >>> instead of >>. Or use Integer.bitCount()
     // C or C++: use uint32_t
     i = i - ((i >> 1) & 0x55555555);        // add pairs of bits
     i = (i & 0x33333333) + ((i >> 2) & 0x33333333);  // quads
     i = (i + (i >> 4)) & 0x0F0F0F0F;        // groups of 8
     return (i * 0x01010101) >> 24;          // horizontal sum of bytes
}

对于 JavaScript：强制转换为整数以|0提高性能：将第一行更改为i = (i|0) - ((i >> 1) & 0x55555555);

这具有所讨论的任何算法中最好的最坏情况行为，因此将有效地处理您抛给它的任何使用模式或值。（它的性能不依赖于普通 CPU 的数据，在普通 CPU 中，包括乘法在内的所有整数运算都是恒定时间的。使用“简单”输入它不会变得更快，但它仍然相当不错。）

这个 SWAR bithack 是如何工作的：

i = i - ((i >> 1) & 0x55555555);

第一步是优化版本的掩码以隔离奇数/偶数位，移动以将它们对齐并添加。这有效地在 2 位累加器中执行了 16 个单独的加法（SWAR = SIMD Within A Register）。喜欢(i & 0x55555555) + ((i>>1) & 0x55555555)。

下一步采用那些 16x 2 位累加器中的奇数/偶数 8 并再次相加，产生 8x 4 位和。i - ...这次优化是不可能的，所以它只是在移动之前/之后进行屏蔽。0x33...在为需要在寄存器中单独构造 32 位常量的 ISA 进行编译时，两次使用相同的常量而不是0xccc...在移位之前使用相同的常量是一件好事。

最后的移位加法步骤(i + (i >> 4)) & 0x0F0F0F0F扩大到 4 个 8 位累加器。它 在添加之后4而不是之前进行屏蔽，因为如果相应输入位的所有 4 位都已设置，任何 4 位累加器中的最大值都是。4+4 = 8 仍然适合 4 位，因此在半字节元素之间进位是不可能的i + (i >> 4)。

到目前为止，这只是相当正常的 SIMD，使用 SWAR 技术并进行了一些巧妙的优化。继续使用相同的模式再执行 2 个步骤可以扩大到 2x 16 位，然后是 1x 32 位计数。但是在具有快速硬件乘法的机器上有一种更有效的方法：

一旦我们有足够少的“元素”， 乘以一个神奇的常量就可以将所有元素相加到 top element 中 。在本例中为字节元素。乘法是通过左移和加法完成的，因此 乘法x * 0x01010101结果为x + (x<<8) + (x<<16) + (x<<24). 我们的 8 位元素足够宽（并且包含足够小的计数），因此不会产生进位 到 前 8 位。

这个的 64 位版本 可以用 0x0101010101010101 乘数在 64 位整数中做 8x 8 位元素，并用>>56. 所以它不需要任何额外的步骤，只是更宽的常量。__builtin_popcountll当硬件popcnt指令未启用时，这就是 GCC 在 x86 系统上的用途。如果您可以为此使用内置函数或内在函数，那么这样做可以让编译器有机会进行特定于目标的优化。

具有用于更宽向量的完整 SIMD（例如计算整个数组）

这种按位 SWAR 算法可以并行化以同时在多个向量元素中完成，而不是在单个整数寄存器中完成，以便在具有 SIMD 但没有可用的 popcount 指令的 CPU 上加速。（例如，必须在任何 CPU 上运行的 x86-64 代码，而不仅仅是 Nehalem 或更高版本。）

但是，使用矢量指令进行 popcount 的最佳方法通常是使用变量洗牌在每个字节的时间并行执行 4 位的表查找。（4 位索引保存在向量寄存器中的 16 条目表）。

在 Intel CPU 上，硬件 64 位 popcnt 指令的性能比SSSE3PSHUFB位并行实现高出大约 2 倍，但前提是您的编译器恰到好处。否则 SSE 可以显着领先。较新的编译器版本可以识别Intel 上的popcnt false 依赖 问题。

  • https://github.com/WojciechMula/sse-popcount用于 SSSE3、AVX2、AVX512BW、AVX512VBMI 或 AVX512 VPOPCNT 的最先进的 x86 SIMD popcount。跨向量使用 Harley-Seal 来延迟元素内的 popcount。（还有 ARM 霓虹灯）
  • 使用 AVX-512 或 AVX-2 在大数据上计算 1 位（人口计数）
  • 相关： https: //github.com/mklarqvist/positional-popcount - 对多个 8、16、32 或 64 位整数的每个位位置进行单独计数。（同样，包含 AVX-512 的 x86 SIMD 非常擅长于此，让vpternlogdHarley-Seal 非常 出色。）


更多建议

一些语言以一种可以 使用有效硬件支持（如果可用）的方式可移植地公开操作，否则一些库回退可能是不错的。

例如（来自语言表）：

  • C++ 有std::bitset<>::count(), 或C++20std::popcount(T x)
  • Java 有java.lang.Integer.bitCount()（也适用于 Long 或 BigInteger）
  • C#有System.Numerics.BitOperations.PopCount()
  • Python 有int.bit_count()（自 3.10 起）

但是，并不是所有的编译器/库都在可用时真正设法使用硬件支持。（值得注意的是 MSVC，即使使用使 std::popcount 内联为 x86 popcnt 的选项，它的 std::bitset::count 仍然总是使用查找表。这有望在未来的版本中改变。）

当可移植语言没有这种基本位操作时，还要考虑编译器的内置函数。例如在 GNU C 中：

int __builtin_popcount (unsigned int x);
int __builtin_popcountll (unsigned long long x);

在最坏的情况下（没有单指令硬件支持），编译器将生成对函数的调用（在当前的 GCC 中，它使用像这个答案这样的移位/和位破解，至少对于 x86）。在最好的情况下，编译器将发出一条 cpu 指令来完成这项工作。（就像*or/运算符一样——如果可用，GCC 将使用硬件乘法或除法指令，否则将调用 libgcc 辅助函数。）或者甚至更好，如果操作数是内联后的编译时常量，它可以进行常量传播获得编译时常量 popcount 结果。

GCC 内置函数甚至可以跨多个平台工作。Popcount 几乎已成为 x86 架构的主流，因此现在开​​始使用内置指令是有意义的，这样您就可以重新编译以让它在编译时内联硬件指令或包含该指令的内容（-mpopcnt例如https://godbolt.org/ z/Ma5e5a）。其他架构已经流行多年，但在 x86 世界中，仍有一些古老的 Core 2 和类似的老式 AMD CPU 在使用。

在 x86 上，您可以告诉编译器它可以假定支持popcntwith 指令-mpopcnt（也暗示为-msse4.2）。请参阅GCC x86 选项。-march=nehalem -mtune=skylake（或者-march=您希望代码采用和调整的任何 CPU）可能是一个不错的选择。在较旧的 CPU 上运行生成的二进制文件将导致非法指令错误。

要使二进制文件针对您构建它们的机器进行优化， 请使用-march=native（与 gcc、clang 或 ICC）。

popcntMSVC 为 x86指令提供了内在指令，但与 gcc 不同的是，它实际上是硬件指令的内在指令，需要硬件支持。

使用std::bitset<>::count()而不是内置

理论上，任何知道如何为目标 CPU 有效地弹出计数的编译器都应该通过 ISO C++ 公开该功能std::bitset<>。实际上，在某些情况下，对于某些目标 CPU，使用 bit-hack AND/shift/ADD 可能会更好。

对于硬件 popcount 是可选扩展的目标体系结构（如 x86），并非所有编译器都具有std::bitset在可用时利用它的功能。例如，MSVC 没有办法popcnt在编译时启用支持，它std::bitset<>::count总是使用表查找，即使使用/Ox /arch:AVX（这意味着 SSE4.2，这又意味着 popcnt 功能。）（更新：见下文；这 确实 得到了 MSVC 的C++20std::popcount使用 x86 popcnt，但仍然不是它的 bitset<>::count。MSVC 可以通过更新他们的标准库头来解决这个问题，以便在可用时使用 std::popcount。）

但至少你得到了可以在任何地方工作的便携的东西，并且使用带有正确目标选项的 gcc/clang，你可以获得支持它的架构的硬件 popcount。

#include <bitset>
#include <limits>
#include <type_traits>

template<typename T>
//static inline  // static if you want to compile with -mpopcnt in one compilation unit but not others
typename std::enable_if<std::is_integral<T>::value,  unsigned >::type 
popcount(T x)
{
    static_assert(std::numeric_limits<T>::radix == 2, "non-binary type");

    // sizeof(x)*CHAR_BIT
    constexpr int bitwidth = std::numeric_limits<T>::digits + std::numeric_limits<T>::is_signed;
    // std::bitset constructor was only unsigned long before C++11.  Beware if porting to C++03
    static_assert(bitwidth <= std::numeric_limits<unsigned long long>::digits, "arg too wide for std::bitset() constructor");

    typedef typename std::make_unsigned<T>::type UT;        // probably not needed, bitset width chops after sign-extension

    std::bitset<bitwidth> bs( static_cast<UT>(x) );
    return bs.count();
}

  {
    var topicId = xmlhttp.responseText;

  }

function postinsql(topicId)
{
  //alert(topicId);
}

我收到一个未定义的错误topicId在我使用该函数之前一切正常setTimeout()。

我希望我的postinsql(topicId)函数在一段时间后被调用。我应该怎么办？



解答

setTimeout(function() {
    postinsql(topicId);
}, 4000)

您需要提供一个匿名函数作为参数而不是字符串，后一种方法甚至不应该根据 ECMAScript 规范工作，但浏览器很宽容。setTimeout()这是正确的解决方案，在使用or时永远不要依赖将字符串作为“函数”传递setInterval()，它速度较慢，因为它必须被评估而且它是不正确的。

更新：

正如 Hobblin 在他对问题的评论中所说，现在您可以使用Function.prototype.bind().

例子：

setTimeout(postinsql.bind(null, topicId), 4000);

从 master 更新 Git 分支

Arya

2023-02-17 07:57

我有四个分支（master、b1、b2 和 b3）。在我处理 b1-b3 之后，我意识到我在分支 master 上需要更改一些应该在所有其他分支中的内容。我改变了我需要的东西master......这是我的问题：

如何使用master分支代码更新所有其他分支？



解答

你有两个选择：

第一个是合并，但这会为合并创建一个额外的提交。

检查每个分支：

git checkout b1

然后合并：

git merge origin/master

然后推：

git push origin b1

或者，你可以做一个变基：

git fetch
git rebase origin/master


更多建议

你基本上有两个选择：

  1. 你合并。这实际上非常简单，并且是一个完美的本地操作：
    git checkout b1

    git merge master

    repeat for b2 and b3

这让历史完全原样：你从 master 分支出来，你对所有分支进行了更改，最后你将来自 master 的更改合并到所有三个分支中。

git可以很好地处理这种情况，它是为同时在各个方向发生的合并而设计的。您可以相信它能够正确地将所有线程组合在一起。它根本不关心 branch b1mergesmaster还是mastermerges b1，合并提交对 git 来说看起来都是一样的。唯一的区别是，哪个分支最终指向这个合并提交。

  1. 你变基了。具有 SVN 或类似背景的人会发现这更直观。这些命令类似于合并案例：
    git checkout b1

    git rebase master

    repeat for b2 and b3

人们喜欢这种方法，因为它在所有分支中都保留了线性历史记录。然而，这种线性历史是一个谎言，你应该意识到它是。考虑这个提交图：

    A --- B --- C --- D <-- master
 \
  \-- E --- F --- G <-- b1

合并产生了真实的历史：

    A --- B --- C --- D <-- master
 \                 \
  \-- E --- F --- G +-- H <-- b1

然而，rebase 为您提供了以下历史记录：

    A --- B --- C --- D <-- master
                   \
                    \-- E' --- F' --- G' <-- b1

关键是，提交E'、F'和G'从未真正存在过，并且可能从未经过测试。他们甚至可能无法编译。通过 rebase 创建无意义的提交实际上很容易，尤其是master当b1.

这样做的结果可能是，您无法区分三个提交中的哪一个E，实际上引入了回归，减少了的值。F``G``git bisect

我并不是说你不应该使用git rebase. 它有它的用途。但是无论何时你使用它，你都需要意识到你在历史上撒谎。你至少应该编译测试新的提交。
