为什么 iostream::eof 在循环条件（即 `while (!stream.eof())`）内被认为是错误的？

Arya

2023-02-17 19:12

我刚刚在这个答案中发现一条评论说iostream::eof在循环条件中使用“几乎肯定是错误的”。我通常使用类似的东西while(cin>>n)- 我猜它会隐式检查 EOF。

为什么明确使用错误检查eof while (!cin.eof())？

scanf("...",...)!=EOF它与在 C 中使用（我经常毫无问题地使用）有何不同？



解答

因为只有读到流结束 后iostream::eof才会返回。它并不 表示 下一次读取将是流的末尾。true ____

考虑一下（并假设下一次读取将在流的末尾）：

while(!inStream.eof()){
  int data;
  // yay, not end of stream yet, now read ...
  inStream >> data;
  // oh crap, now we read the end and *only* now the eof bit will be set (as well as the fail bit)
  // do stuff with (now uninitialized) data
}

反对这个：

int data;
while(inStream >> data){
  // when we land here, we can be sure that the read was successful.
  // if it wasn't, the returned stream from operator>> would be converted to false
  // and the loop wouldn't even be entered
  // do stuff with correctly initialized data (hopefully)
}

关于你的第二个问题：因为

if(scanf("...",...)!=EOF)

是相同的

if(!(inStream >> data).eof())

和 不 一样

if(!inStream.eof())
    inFile >> data


更多建议

Bottom-line top： 通过正确处理空白，eof可以使用以下方法（甚至比fail()错误检查更可靠）：

while( !(in>>std::ws).eof() ) {
   int data;
   in >> data;
   if ( in.fail() ) /* Handle with 'break' or 'throw' */;
   // Now use data
}

（ 感谢 Tony D 提出的突出答案的建议。请参阅下面他的评论以了解为什么这更可靠的示例。 ）

反对 using 的主要论据eof()似乎遗漏了一个关于空白角色的重要微妙之处。我的主张是，eof()明确检查不仅不是“ 总是错误的 ”——这似乎是这个问题和类似的 Stack Overflow 问题中的压倒一切的意见——而且通过正确处理空白，它提供了一个更清晰、更可靠的错误处理，并且 始终 是正确的解决方案（尽管不一定是最简洁的）。

总结一下建议的“正确”终止和读取顺序如下：

int data;
while(in >> data) {  /* ... */ }

// Which is equivalent to
while( !(in >> data).fail() )  {  /* ... */ }

由于超出eof 的读取尝试失败被视为终止条件。这意味着没有简单的方法来区分成功的流和真正因 eof 以外的原因而失败的流。采用以下流：

  • 1 2 3 4 5<eof>
  • 1 2 a 3 4 5<eof>
  • a<eof>

while(in>>data) 以所有failbit三个输入的一组结束。在第一和第三，也是集。因此，在循环之后，需要非常丑陋的额外逻辑来区分正确的输入（第一个）和不正确的输入（第二个和第三个）。 __eofbit

鉴于以下内容：

while( !in.eof() )
{
   int data;
   in >> data;
   if ( in.fail() ) /* Handle with break or throw */;
   // Now use data
}

这里，in.fail()验证只要有读到的就是正确的。它的目的不仅仅是一个 while 循环终止符。

到目前为止一切顺利，但是如果流中有尾随空格会发生什么——听起来像eof()终结者的主要关注点是什么？

我们不需要放弃我们的错误处理；只吃掉空白：

while( !in.eof() )
{
   int data;
   in >> data >> ws; // Eat white space with 'std::ws'
   if ( in.fail() ) /* Handle with 'break' or 'throw' */;
   // Now use data
}

std::ws在设置eofbit, 而 不是failbit. 因此，in.fail()只要至少有一个数据要读取，就可以按预期工作。如果全空白流也可以接受，那么正确的形式是：

while( !(in>>ws).eof() )
{
   int data;
   in >> data;
   if ( in.fail() ) /* Handle with 'break' or 'throw' */;
   /* This will never fire if the eof is reached cleanly */
   // Now use data
}

简介： 正确构造while(!eof)不仅可能而且不会出错，而且它允许数据在范围内本地化，并提供更清晰的错误检查与常规业务的分离。话虽如此，while(!fail)它无疑是一种更常见和简洁的习惯用法，并且在简单（每次读取类型的单个数据）场景中可能更受欢迎。

为什么 AngularJS 在选择中包含一个空选项？

Arya

2023-02-17 19:12

过去几周我一直在使用 AngularJS，真正困扰我的一件事是，即使在尝试了所有排列或规范中定义的配置之后，http: //docs.angularjs.org/api/ng .directive:select，我仍然得到一个空选项作为 select 元素的第一个子元素。

这是翡翠：

select.span9(ng-model='form.type', required, ng-options='option.value as option.name for option in typeOptions');

这里的控制器：

$scope.typeOptions = [
    { name: 'Feature', value: 'feature' },
    { name: 'Bug', value: 'bug' },
    { name: 'Enhancement', value: 'enhancement' }
];

最后，这是生成的 HTML：

<select ng-model="form.type" required="required" ng-options="option.value as option.name for option in typeOptions" class="span9 ng-pristine ng-invalid ng-invalid-required">
    <option value="?" selected="selected"></option>
    <option value="0">Feature</option>
    <option value="1">Bug</option>
    <option value="2">Enhancement</option>
</select>

我需要做什么才能摆脱它？

PS：没有这个也能正常工作，但如果你使用 select2 而没有多选，它看起来很奇怪。



解答

当传递给 的一组选项中不存在引用的option值时，将生成空值。这样做是为了防止意外的模型选择：AngularJS 可以看到初始模型未定义或不在选项集中，并且不想自己决定模型值。ng-model``ng-options

如果您想摆脱空选项，只需在控制器中选择一个初始值，例如：

$scope.form.type = $scope.typeOptions[0].value;

这是 jsFiddle： http: //jsfiddle.net/MTfRD/3/

简而言之：空选项意味着没有选择有效模型（有效我的意思是：来自选项集）。您需要选择一个有效的模型值来摆脱这个空选项。


更多建议

如果你想要一个初始值，请参阅@pkozlowski.opensource 的答案，仅供参考，也可以使用 ng-init 在视图（而不是控制器）中实现：

<select ng-model="form.type" required="required" ng-init="form.type='bug'"
  ng-options="option.value as option.name for option in typeOptions" >
</select>

如果您不想要初始值，“可以将值设置为空字符串的单个硬编码元素嵌套到元素中。然后该元素将表示 null 或“未选择”选项”：

<select ng-model="form.type" required="required"
  ng-options="option.value as option.name for option in typeOptions" >
    <option style="display:none" value="">select a type</option>
</select>

如何将大文本文件拆分为行数相等的小文件？

Arya

2023-02-17 19:11

我有一个大的（按行数）纯文本文件，我想将其拆分成较小的文件，也按行数。因此，如果我的文件有大约 200 万行，我想将它分成 10 个包含 200k 行的文件，或 100 个包含 20k 行的文件（加上一个文件和其余部分；均匀分割并不重要）。

我可以在 Python 中相当轻松地做到这一点，但我想知道是否有任何一种忍者方法可以使用 Bash 和 Unix 实用程序来做到这一点（而不是手动循环和计数/分区行）。



解答

看一下拆分命令：

$ split --help
Usage: split [OPTION] [INPUT [PREFIX]]
Output fixed-size pieces of INPUT to PREFIXaa, PREFIXab, ...; default
size is 1000 lines, and default PREFIX is `x'.  With no INPUT, or when INPUT
is -, read standard input.

Mandatory arguments to long options are mandatory for short options too.
  -a, --suffix-length=N   use suffixes of length N (default 2)
  -b, --bytes=SIZE        put SIZE bytes per output file
  -C, --line-bytes=SIZE   put at most SIZE bytes of lines per output file
  -d, --numeric-suffixes  use numeric suffixes instead of alphabetic
  -l, --lines=NUMBER      put NUMBER lines per output file
      --verbose           print a diagnostic to standard error just
                            before each output file is opened
      --help     display this help and exit
      --version  output version information and exit

你可以这样做：

split -l 200000 filename

这将创建文件，每个文件包含 200000 行，名为xaa xab xac...

另一种选择，按输出文件的大小拆分（仍按换行符拆分）：

 split -C 20m --numeric-suffixes input_filename output_prefix

output_prefix01 output_prefix02 output_prefix03 ...创建每个最大大小为 20 兆字节的文件。


更多建议

使用拆分命令：

split -l 200000 mybigfile.txt
