为什么 C 预处理器将单词“linux”解释为常量“1”？

Arya

2023-02-17 06:00

为什么 GCC 中的 C 预处理器将单词（小写字母）解释为常量？ ****linux``1

测试.c：

#include <stdio.h>
int main(void)
{       
    int linux = 5;
    return 0;
}

结果$ gcc -E test.c（预处理阶段后停止）：

....
int main(void)
{
    int 1 = 5;
    return 0;
}

这当然会产生错误。

#define linux（顺便说一句：文件中没有stdio.h。）



解答

在过去（ANSI 之前），预定义诸如unix和 之类的符号vax是一种允许代码在编译时检测它正在为哪个系统编译的方法。当时没有官方语言标准（除了第一版 K&R 后面的参考资料），任何复杂的 C 代码通常都是#ifdefs 的复杂迷宫，以允许系统之间的差异。这些宏定义一般是由编译器自己设置的，而不是在库头文件中定义的。由于没有关于哪些标识符可以由实现使用以及哪些标识符保留给程序员的真正规则，编译器编写者可以随意使用简单的名称，例如unix并假设程序员会简单地避免将这些名称用于他们自己的目的。

1989 年的 ANSI C 标准引入了限制实现可以合法预定义哪些符号的规则。编译器预定义的宏的名称只能以两个下划线开头，或以一个下划线后跟一个大写字母开头，从而使程序员可以自由使用与该模式不匹配且未在标准库中使用的标识符。

因此，任何预定义unix或linux不符合规范的编译器都无法编译使用int linux = 5;.

碰巧的是，默认情况下 gcc 是不符合规范的——但可以使它符合（合理地）符合正确的命令行选项：

gcc -std=c90 -pedantic ... # or -std=c89 or -ansi
gcc -std=c99 -pedantic
gcc -std=c11 -pedantic

有关详细信息，请参阅gcc 手册。

gcc 将在未来的版本中逐步淘汰这些定义，因此您不应编写依赖于它们的代码。如果您的程序需要知道它是否正在为 Linux 目标编译，它可以检查是否已__linux__定义（假设您使用的是 gcc 或与其兼容的编译器）。有关详细信息，请参阅GNU C 预处理器手册。

一个基本无关紧要的旁白：David Korn（是的，Korn Shell 的作者）的 1987 年国际混淆 C 代码竞赛的“最佳单行代码”获胜者利用了预定义的宏unix：

main() { printf(&unix["\021%six\012\0"],(unix)["have"]+"fun"-0x60);}

它打印"unix"，但出于与宏名称的拼写完全无关的原因。

我不想在这里发布任何剧透，我鼓励任何阅读本文的人首先尝试自己理解该代码。但是如果你真的想放弃，我在这里发布了一个解释： https: //gist.github.com/Keith-S-Thompson/6920347


更多建议

这似乎是一个（未记录的）“GNU 扩展”：[ 更正 ：我终于在文档中找到了提及。见下文。]

以下命令使用该-dM选项打印所有预处理器定义；由于输入“文件”是空的，它准确地显示了预定义的宏。它在标准的 ubuntu 安装上使用 gcc-4.7.3 运行。您可以看到预处理器是标准感知的。总共有 243 个宏-std=gnu99和 240 个宏-std=c99；我过滤了输出的相关性。

$ cpp --std=c89 -dM < /dev/null | grep linux
#define __linux 1
#define __linux__ 1
#define __gnu_linux__ 1

$ cpp --std=gnu89 -dM < /dev/null | grep linux
#define __linux 1
#define __linux__ 1
#define __gnu_linux__ 1
#define linux 1

$ cpp --std=c99 -dM < /dev/null | grep linux
#define __linux 1
#define __linux__ 1
#define __gnu_linux__ 1

$ cpp --std=gnu99 -dM < /dev/null | grep linux
#define __linux 1
#define __linux__ 1
#define __gnu_linux__ 1
#define linux 1

“gnu 标准”版本也是#define unix。（使用c11和gnu11产生相同的结果。）

我想他们有他们的理由，但在我看来，gcc 的默认安装（除非另有-std=gnu89说明，否则编译 C 代码）不符合要求，并且 - 就像这个问题一样 - 令人惊讶。在一致的实现中，不允许使用名称不以下划线开头的宏来污染全局命名空间。（6.8.10p2：“任何其他预定义的宏名称应以前导下划线开头，后跟大写字母或第二个下划线，”但是，如附录 J.5（可移植性问题）中所述，此类名称通常是预定义的。）

当我最初写这个答案时，我无法在 gcc 中找到关于这个问题的任何文档，但我终于发现了它，不是在 C实现定义的行为中，也不是在C 扩展中，而是在手册第 3.7.3cpp节中，其中它指出：

我们正在慢慢淘汰保留命名空间之外的所有预定义宏。你不应该在新程序中使用它们......

我正在尝试学习如何使用 Git，并创建了一个包含 HTML、CSS 和 JavaScript 文件的小项目。我从我基本上是空的项目创建了一个分支，然后对我的代码进行了一些更改。我尝试暂存更改，但收到以下错误消息：

Another git process seems to be running in this repository, e.g.
an editor opened by 'git commit'. Please make sure all processes
are terminated then try again. If it still fails, a git process
may have crashed in this repository earlier:
remove the file manually to continue.

诚然，我确实遇到了问题，试图更早地提交我的空项目并退出Git Bash，因为我不知道如何摆脱我不知何故的地方。

有没有办法解决这个问题，或者我应该开始一个新的存储库？



解答

如果您在工作树中，请尝试删除目录或其中一个 工作树index.lock中的文件。.git ****.git/worktrees/*/index.lock

rm -f .git/index.lock

当同时执行两个git命令时，通常会出现此类问题；也许一个来自命令提示符，另一个来自 IDE。


更多建议

在应用程序的根目录中使用以下命令。这将删除 index.lock 文件并释放活动锁。

rm .git/index.lock

什么时候应该使用花括号导入 ES6？

Arya

2023-02-17 05:59

这似乎很明显，但我发现自己对何时使用大括号在 ES6 中导入单个模块感到有点困惑。例如，在我正在处理的 React-Native 项目中，我有以下文件及其内容：

文件 initialState.js 

var initialState = {
    todo: {
        todos: [
            {id: 1, task: 'Finish Coding', completed: false},
            {id: 2, task: 'Do Laundry', completed: false},
            {id: 2, task: 'Shopping Groceries', completed: false},
        ]
    }
};

export default initialState;

在 TodoReducer.js 中，我必须在不使用大括号的情况下导入它：

import initialState from './todoInitialState';

如果我将initialState大括号括起来，我会收到以下代码行的以下错误：

无法读取未定义的属性 todo

文件 TodoReducer.js ：

export default function todos(state = initialState.todo, action) {
    // ...
}

类似的错误也发生在我的带有花括号的组件上。我想知道什么时候我应该对单个导入使用花括号，因为很明显，当导入多个组件/模块时，你必须将它们括在花括号中，我知道这一点。

这里的 Stack Overflow 帖子没有回答我的问题，而是问我 什么时候 应该或不应该使用花括号导入 单个 模块，或者我永远不应该使用花括号导入 ES6 中的单个模块（这显然不是这种情况，正如我所看到的需要花括号的单一导入）。



解答

这是 默认导入 ：

// B.js
import A from './A'

A它仅在具有 默认导出 时才有效：

// A.js
export default 42

在这种情况下，导入时分配给它的名称无关紧要：

// B.js
import A from './A'
import MyA from './A'
import Something from './A'

因为它总是会 解析 为A.

这是一个 名为的命名导入A：

import { A } from './A'

它仅在A包含 名为的命名导出A时才有效：

export const A = 42

在这种情况下，名称很重要，因为您要 通过其导出名称导入特定的东西 ：

// B.js
import { A } from './A'
import { myA } from './A' // Doesn't work!
import { Something } from './A' // Doesn't work!

要使这些工作正常，您需要将 相应的命名导出 添加到A：

// A.js
export const A = 42
export const myA = 43
export const Something = 44

一个模块只能有 一个默认导出 ，但 可以有任意多个命名导出 （零个、一个、两个或多个）。您可以将它们全部导入：

// B.js
import A, { myA, Something } from './A'

在这里，我们将默认导出导入为A，并分别将导出命名为myA和Something。

// A.js
export default 42
export const myA = 43
export const Something = 44

我们还可以在导入时为它们分配不同的名称：

// B.js
import X, { myA as myX, Something as XSomething } from './A'

默认导出往往用于您通常希望从模块中获得的任何内容。命名导出往往用于可能方便但并非总是必需的实用程序。然而，您可以选择如何导出内容：例如，一个模块可能根本没有默认导出。

这是一个很好的 ES 模块指南，解释了默认导出和命名导出之间的区别。


更多建议

我想说importES6 关键字也有一个星号值得一提。

如果您尝试控制台日志 Mix：

import * as Mix from "./A";
console.log(Mix);

你会得到：

什么时候应该使用花括号导入 ES6？

当你只需要模块中的特定组件时，括号是金色的，这使得像 webpack 这样的打包器占用空间更小。
