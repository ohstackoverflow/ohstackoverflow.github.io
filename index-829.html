Java Persistence API 中 FetchType LAZY 和 EAGER 的区别？

Arya

2023-02-17 17:14

FetchType.LAZYJava Persistence API和Java Persistence API之间有什么区别FetchType.EAGER？



解答

有时您有两个实体，并且它们之间存在关系。例如，您可能有一个名为的实体University和另一个名为的实体Student，一所大学可能有很多学生：

University 实体可能有一些基本属性，如 id、name、address 等，以及一个名为 students 的集合属性，它返回给定大学的学生列表：

public class University {
   private String id;
   private String name;
   private String address;
   private List<Student> students;

   // setters and getters
}

现在，当您从数据库加载一所大学时，JPA 会为您加载它的 id、名称和地址字段。但是对于如何加载学生，您有两种选择：

  1. 将它与其余字段一起加载（即急切地），或者
  2. 当您调用大学的方法时按需（即延迟）加载它getStudents()。

当一所大学有很多学生时，将所有学生一起加载是没有效率的，特别是当不需要他们时，在这种情况下，您可以声明您希望在实际需要时加载学生。这称为延迟加载。

这是一个示例，其中students明确标记为急切加载：

@Entity
public class University {

    @Id
    private String id;

    private String name;

    private String address;

    @OneToMany(fetch = FetchType.EAGER)
    private List<Student> students;

    // etc.    
}

students这是一个明确标记为延迟加载的示例：

@Entity
public class University {

    @Id
    private String id;

    private String name;

    private String address;

    @OneToMany(fetch = FetchType.LAZY)
    private List<Student> students;

    // etc.
}


更多建议

基本上，

LAZY = fetch when needed
EAGER = fetch immediately

Docker 中的“公开”和“发布”有什么区别？

Arya

2023-02-17 17:13

我正在试验 Dockerfiles，我想我理解大部分逻辑。但是，在这种情况下，我看不出“公开”和“发布”端口之间的区别。

我看到的所有教程首先包括EXPOSEDockerfile中的命令：

...
EXPOSE 8080
...

然后他们从这个 Dockerfile 构建一个镜像：

$ docker build -t an_image - < Dockerfile

然后在运行图像时 发布与上面相同的端口：

$ docker run -d -p 8080 an_image

或使用发布所有端口

$ docker run -d -P an_image

如果无论如何都会发布，那么在 Dockerfile 中公开一个端口有什么意义呢？是否需要先公开端口，然后 再 发布？实际上，我想在创建映像时指定我将在 Dockerfile 中使用的所有端口，然后不再理会它们，只需使用以下命令运行它们：

$ docker run -d an_image

这可能吗？



解答

基本上，您有三（四）种选择：

  1. 既不指定EXPOSE也不-p
  2. 仅指定EXPOSE
  3. 指定EXPOSE和-p

  4. 仅指定-p隐式执行的操作EXPOSE

  5. 如果既不指定EXPOSE也不指定-p，则容器中的服务只能从容器本身 内部访问。

  6. 如果你是EXPOSE一个端口，容器中的服务不能从 Docker 外部访问，而是从其他 Docker 容器内部访问。所以这有利于容器间通信。

  7. 如果你EXPOSE和-p一个端口，容器中的服务可以从任何地方访问，甚至是在 Docker 之外。

  8. 如果您这样做-p，但没有这样做EXPOSE，Docker 会隐式执行EXPOSE。这是因为如果一个端口向公众开放，它也会自动向其他 Docker 容器开放。因此-p包括EXPOSE. 这实际上与 3) 相同。

两者分开的原因是恕我直言，因为：

  • 选择主机端口取决于主机，因此不属于 Dockerfile（否则将取决于主机），
  • 如果容器中的服务可以从其他容器访问，通常就足够了。

该文档明确指出：

该EXPOSE指令公开了在链接中使用的端口。

它还指出了如何链接容器，这基本上就是我谈到的容器间通信。

暴露和发布端口

在 Docker 网络中，有两种不同的机制直接涉及网络端口：暴露和发布端口。这适用于默认桥接网络和用户定义的桥接网络。

  • 您使用EXPOSEDockerfile 中的关键字或--expose标志将端口公开给 docker run。 公开端口是记录 使用了哪些端口的一种方式， 但实际上并不映射或打开任何端口 。公开端口是可选的。

  • 您使用--publishor--publish-all标志将端口发布到docker run. 这告诉 Docker 在容器的网络接口上打开哪些端口。30000发布端口时，它会映射到主机上可用的高阶端口（高于），除非您在运行时指定要映射到主机上的端口。在构建镜像时（在 Dockerfile 中），您不能指定要映射到主机上的端口，因为无法保证 该端口在运行镜像的主机上可用 。

来自： Docker 容器网络

在@Golo Roden 的回答中指出：

“如果你不指定其中任何一个，容器中的服务将无法从容器本身内部以外的任何地方访问。”

也许在写答案时就是这种情况，但现在看来，即使您不使用EXPOSEor --publish，同一网络的hostand other 也containers将能够访问您可能在该容器内启动的服务。

如何测试这个：

我使用了以下内容Dockerfile。基本上，我从 ubuntu 开始并安装一个小型网络服务器：

FROM ubuntu
RUN apt-get update && apt-get install -y mini-httpd

我将build图像作为“testexpose”和run一个新容器：

docker run --rm -it testexpose bash

在容器内，我启动了几个实例mini-httpd：

root@fb8f7dd1322d:/# mini_httpd -p 80
root@fb8f7dd1322d:/# mini_httpd -p 8080

然后我就可以curl从主机或其他容器中使用来获取主页mini-httpd。
