TINYTEXT、TEXT、MEDIUMTEXT 和 LONGTEXT 最大存储大小

Arya

2023-02-17 09:40

根据MySQL 文档，有四种 TEXT 类型：

  1. 小文字
  2. 文本
  3. 中文本
  4. 长文

假设字符编码为 UTF-8，我可以在每种数据类型的列中存储的最大长度是多少？



解答

从文档（MySQL 8）：

      类型 | 最大长度
----------+------------------------------------
  小文 | 255 (2 8 −1) 字节
      文字 | 65,535 (2 16 −1) 字节 = 64 KiB
中文 | 16,777,215 (2 24 −1) 字节 = 16 MiB
  长文 | 4,294,967,295 (2 32 −1) 字节 = 4 GiB

请注意，可以存储在您的列中的字符 数将取决于 字符编码 。

  2. 正如第 (1) 点所述，应始终使用 VARCHAR 而不是 TINYTEXT。但是，当使用 VARCHAR 时，最大行大小不应超过 65535 字节。
  3. 如此处所述http://dev.mysql.com/doc/refman/5.0/en/charset-unicode-utf8.html，utf-8 最多 3 个字节。

这是一个用于快速决策的粗略估算表！

  2. 假设英语平均每个单词有 4.5 个字母
  3. x 是分配的字节数

“=”和“<-”赋值运算符有什么区别？

Arya

2023-02-17 09:39

=赋值运算符和R中的区别是什么<-？

我知道运算符略有不同，如本例所示

x <- y <- 5
x = y = 5
x = y <- 5
x <- y = 5
# Error in (x <- y) = 5 : could not find function "<-<-"

但这是唯一的区别吗？



解答

=赋值运算符和R中的区别是什么<-？

如您的示例所示，=并且<-运算符优先级略有不同（当它们混合在同一表达式中时，这决定了评估的顺序）。实际上，?Syntax在 R 中给出了如下运算符优先级表，从高到低：

…
‘-> ->>’           rightwards assignment
‘<- <<-’           assignment (right to left)
‘=’                assignment (right to left)
…

但这是唯一的区别吗？

由于您询问的是 赋值运算符 ：是的，这是唯一的区别。但是，您不这样认为是可以原谅的。?assignOps甚至声称存在更多差异的R 文档：

运算符<-可以在任何地方使用，而运算符=只允许在顶层使用（例如，在命令提示符下键入的完整表达式中）或作为括号表达式列表中的子表达式之一。

让我们不要过分强调它： R 文档是错误的 。这很容易证明：我们只需要找到一个运算符的反例，=它不是 (a) 在顶层，也不是 (b) 花括号表达式列表中的子表达式（即{…; …}）。- 无需再费周折：

x
# Error: object 'x' not found
sum((x = 1), 2)
# [1] 3
x
# [1] 1

=显然，我们在上下文 (a) 和 (b) 之外使用 执行了一项任务。那么，为什么核心 R 语言功能的文档几十年来一直是错误的？

这是因为在 R 的语法中，符号=有两个不同的含义，经常被混淆（即使是专家，包括在上面引用的文档中）：

  1. 第一个意思是作为 赋值运算符 。到目前为止，这就是我们所讨论的全部内容。
  2. 第二个含义不是运算符，而是表示在函数调用中 传递命名参数的 语法标记。 与运算 符 不同，它在运行时不执行任何操作，它仅更改表达式的解析方式。 =

那么 R 如何决定给定的用法是=指运算符还是命名参数传递？让我们来看看。

在任何一般形式的代码中......

‹function_name›(‹argname› = ‹value›, …)
‹function_name›(‹args›, ‹argname› = ‹value›, …)

… the = 是定义命名参数传递的标记：它不是 赋值 运算符。此外，在某些句法上下文中=是完全 禁止的：

if (‹var› = ‹value›) …
while (‹var› = ‹value›) …
for (‹var› = ‹value› in ‹value2›) …
for (‹var1› in ‹var2› = ‹value›) …

这些中的任何一个都会引发错误“在‹bla›中出现意外的'='”。

在任何其他上下文中，=指的是赋值运算符调用。特别是，仅在子表达式周围加上括号就可以使上述 (a) 中的任何一个有效，并且 (b) 成为一个 赋值 。例如，以下执行分配：

median((x = 1 : 10))

但是也：

if (! (nf = length(from))) return()

现在您可能会反对这样的代码是残酷的（您可能是对的）。但我从函数中提取了这段代码base::file.copy（替换<-为=）——它是许多核心 R 代码库中的普遍模式。

R 文档可能基于John Chambers 的原始解释实际上正确地解释了这一点：

[=赋值]只允许在语法中的两个地方：在顶层（作为一个完整的程序或用户输入的表达式）；当与周围的逻辑结构隔离时，通过大括号或一对额外的括号。

总之，默认情况下运算符<-和=做同样的事情。但是它们中的任何一个都可以被单独 覆盖 以改变它的行为。相比之下，<-and ->（从左到右的赋值）虽然在语法上不同，但总是调用 相同的 函数。覆盖一个也会覆盖另一个。知道这一点很少有实际意义，但 可以 用于一些有趣的恶作剧。

text 和 varchar 的区别（字符变化）

Arya

2023-02-17 09:39

text数据类型和character varying( ) 数据类型有什么区别varchar？

根据文档

如果在没有长度说明符的情况下使用字符变化，则该类型接受任何大小的字符串。后者是 PostgreSQL 扩展。

和

另外，PostgreSQL 提供了文本类型，可以存储任意长度的字符串。虽然类型文本不在 SQL 标准中，但其他几个 SQL 数据库管理系统也有它。

那么有什么区别呢？



解答

没有区别，在引擎盖下就是全部varlena（可变长度数组）。

查看 Depesz 的这篇文章：http://www.depesz.com/index.php/2010/03/02/charx-vs-varcharx- vs-varchar-vs-text/

几个亮点：

总结一下：

  • nchar(n) – 在处理短于（将它们填充到）的值时占用太多空间n，并且由于添加尾随空格而可能导致细微错误，而且更改限制是有问题的
  • varchar(n) – 在实时环境中更改限制是有问题的（更改表时需要独占锁）
  • varchar——就像文本一样
  • text——对我来说是赢家 ——胜过 (n) 数据类型，因为它没有它们的问题，胜过 varchar——因为它有不同的名字

文章做了详细的测试，表明4种数据类型的insert和select性能都差不多。它还详细介绍了在需要时限制长度的替代方法。基于函数的约束或域提供了立即增加长度约束的优势，并且基于减少字符串长度约束的情况很少见，depesz 得出结论，其中之一通常是长度限制的最佳选择。


更多建议

正如文档中的“字符类型varchar(n)”所指出的， 、char(n)和text都以相同的方式存储。唯一的区别是需要额外的周期来检查长度（如果给定的话），以及如果需要填充则需要额外的空间和时间char(n)。

然而，当您只需要存储单个字符时，使用特殊类型"char"（保留双引号——它们是类型名称的一部分）会带来轻微的性能优势。您可以更快地访问该字段，并且没有存储长度的开销。

"char"我刚刚制作了一张从小写字母表中随机选择的 1,000,000 的表格。获取频率分布 ( select count(*), field ... group by field) 的查询大约需要 650 毫秒，而使用text字段查询相同数据大约需要 760 毫秒。
