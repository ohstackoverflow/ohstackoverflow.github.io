有没有一种简单的方法可以按值删除列表元素？

Arya

2023-02-17 05:04

我想从列表中删除一个值，如果它存在于列表中（它可能不存在）。

a = [1, 2, 3, 4]
b = a.index(6)

del a[b]
print(a)

以上给出了错误：

ValueError: list.index(x): x not in list

所以我必须这样做：

a = [1, 2, 3, 4]

try:
    b = a.index(6)
    del a[b]
except:
    pass

print(a)

但是有没有更简单的方法来做到这一点？



解答

要删除元素的 第一次出现list.remove，请使用：

>>> xs = ['a', 'b', 'c', 'd']
>>> xs.remove('b')
>>> print(xs)
['a', 'c', 'd']

要删除 所有出现 的元素，请使用列表理解：

>>> xs = ['a', 'b', 'c', 'd', 'b', 'b', 'b', 'b']
>>> xs = [x for x in xs if x != 'b']
>>> print(xs)
['a', 'c', 'd']


更多建议

通常，如果你告诉 Python 做一些它不能做的事情，Python 会抛出一个异常，所以你必须做以下任一事情：

if c in a:
    a.remove(c)

或者：

try:
    a.remove(c)
except ValueError:
    pass

Exception 不一定是坏事，只要它是您期望并正确处理的异常。

.zip是一种存档格式，通常使用Deflate 压缩方法。.gzgzip 格式适用于单个文件，也使用 Deflate 压缩方法。通常 gzip 与 tar 结合使用以制作压缩存档格式，.tar.gz. zlib库提供了供 zip、gzip、png（在 deflate 数据上使用zlib 包装器）和许多其他应用程序使用的 Deflate 压缩和解压缩代码。

ZIP格式由 Phil Katz 开发为具有开放规范的开放格式，他的实现 PKZIP 是共享软件。它是一种存储文件及其目录结构的存档格式，其中每个文件都被单独压缩。文件类型是.zip. 可以选择对文件以及目录结构进行加密。

ZIP 格式支持多种压缩方法：

    0 - The file is stored (no compression)
    1 - The file is Shrunk
    2 - The file is Reduced with compression factor 1
    3 - The file is Reduced with compression factor 2
    4 - The file is Reduced with compression factor 3

    7 - Reserved for Tokenizing compression algorithm
    8 - The file is Deflated
    9 - Enhanced Deflating using Deflate64(tm)
   10 - PKWARE Data Compression Library Imploding (old IBM TERSE)
   11 - Reserved by PKWARE

   13 - Reserved by PKWARE

方法 1 到 7 是历史的，没有被使用。方法 9 到 98 是相对较新的添加，并且使用量不一。在 ZIP 格式中真正广泛使用的唯一方法是方法 8，Deflate，以及某种程度上完全没有压缩的方法 0。.zip实际上，您在野外遇到的每个文件都将专门使用方法 8 和 0，可能只是方法 8。（方法 8 也有一种有效存储数据的方法，无需压缩和相对较少的扩展，并且方法 0 无法流式传输而方法 8 可以。）

文件容器的ISO /IEC 21320-1:2015 标准是一种受限制的 zip 格式，例如用于 Java 存档文件 (.jar)、Office Open XML 文件（Microsoft Office .docx、.xlsx、.pptx）、Office Document格式文件（.odt、.ods、.odp）和 EPUB 文件 (.epub)。该标准将压缩方法限制为 0 和 8，以及其他限制，例如不加密或签名。

1990 年左右，Info-ZIP 小组编写了可移植的、免费的、开源的实现zip和unzip实用程序，支持使用 Deflate 格式进行压缩，以及对该格式和更早格式的解压缩。这极大地扩展了格式的使用范围.zip。

在 20 世纪 90 年代初期，gzip 格式被开发为Unixcompress实用程序的替代品，它源自 Info-ZIP 实用程序中的 Deflate 代码。Unixcompress旨在压缩单个文件或流，将 a 附加.Z到文件名。compress使用LZW 压缩算法，该算法当时处于专利状态，其免费使用受到专利持有人的争议。虽然 Deflate 的某些特定实现已获得 Phil Katz 的专利，但格式并未获得专利，因此可以编写不侵犯任何专利的 Deflate 实现。在过去的 20 多年里，这种实施并没有受到如此大的挑战。Unixgzip实用程序旨在作为compress，并且实际上能够解压缩compress- 压缩数据（假设您能够解析该句子）。gzip将 a 附加.gz到文件名。gzip使用 Deflate 压缩数据格式，压缩比 Unix 好很多compress，解压速度非常快，并添加了 CRC-32 作为数据的完整性检查。头部格式还允许存储比compress格式允许的更多信息，例如原始文件名和文件修改时间。

虽然compress只压缩单个文件，但通常使用该tar实用程序将文件、它们的属性和目录结构创建到一个.tar文件中，然后将其压缩成compress一个.tar.Z文件。事实上，该tar实用程序曾经并且仍然可以选择同时进行压缩，而不必将 的输出通过管道传输tar到compress. 这一切都继承到了 gzip 格式，并且tar有一个选项可以直接压缩到该.tar.gz格式。该tar.gz格式比方法压缩得更好.zip，因为 a 的压缩.tar可以利用跨文件的冗余，尤其是许多小文件。.tar.gz是 Unix 上最常用的归档格式，因为它具有非常高的可移植性，但也有更有效的压缩方法在使用，所以你会经常看到和.tar.bz2归档.tar.xz。

在 gzip 引入后不久，大约在 1990 年代中期，同样的专利纠纷对图像格式的免费使用提出了质疑，该图像格式.gif在公告牌和万维网上非常广泛使用（在当时是一个新事物）。因此，一个小组创建了 PNG 无损压缩图像格式，文件类型为.png，以取代.gif. 该格式还使用 Deflate 格式进行压缩，在图像数据的过滤器暴露更多冗余后应用。为了促进 PNG 格式的广泛使用，创建了两个免费代码库。libpng和zlib. libpng 处理了 PNG 格式的所有特性，而 zlib 提供了压缩和解压缩代码供 libpng 以及其他应用程序使用。zlib 改编自gzip代码。

如何在 30 分钟后使 PHP 会话过期？

Arya

2023-02-17 05:03

我需要让会话保持 30 分钟，然后销毁它。



解答

您应该实现自己的会话超时。其他人提到的两个选项（ session.gc_maxlifetime 和 session.cookie_lifetime ）都不可靠。我会解释其中的原因。

第一的：

session.gc_maxlifetime
session.gc_maxlifetime 指定数据将被视为“垃圾”并清理之前的秒数。垃圾收集发生在会话启动期间。

但是垃圾收集器仅以 session.gc_probability 除以 session.gc_divisor 的概率启动。并使用这些选项的默认值（分别为 1 和 100），机会仅为 1%。

好吧，您可以简单地调整这些值，以便更频繁地启动垃圾收集器。但是当垃圾收集器启动时，它会检查每个注册会话的有效性。这是成本密集型的​​。

此外，当使用 PHP 的默认 session.save_handler 文件时，会话数据存储在 session.save_path 中指定路径的文件中。使用该会话处理程序，会话数据的年龄是根据文件的最后修改日期而不是最后访问日期计算的：

注意： 如果您使用默认的基于文件的会话处理程序，您的文件系统必须跟踪访问时间 (atime)。Windows FAT 没有，因此如果您受困于 FAT 文件系统或时间跟踪不可用的任何其他文件系统，您将不得不想出另一种方法来处理垃圾收集您的会话。从 PHP 4.2.3 开始，它使用 mtime（修改日期）代替 atime。因此，您不会遇到时间跟踪不可用的文件系统的问题。

因此，还可能会发生会话数据文件被删除而会话本身仍被视为有效的情况，因为会话数据最近未更新。

第二：

session.cookie_lifetime
session.cookie_lifetime 指定发送到浏览器的 cookie 的生命周期（以秒为单位）。[…]

恩，那就对了。这只会影响 cookie 的生命周期，会话本身可能仍然有效。但使会话无效是服务器的任务，而不是客户端的任务。所以这没有任何帮助。事实上，将 session.cookie_lifetime 设置为0将使会话的 cookie 成为真正的会话 cookie，它仅在浏览器关闭之前有效。

结论/最佳解决方案：

最好的解决方案是实现您自己的会话超时。使用一个简单的时间戳来表示最后一次活动（即请求）的时间，并在每次请求时更新它：

if (isset($_SESSION['LAST_ACTIVITY']) && (time() - $_SESSION['LAST_ACTIVITY'] > 1800)) {
    // last request was more than 30 minutes ago
    session_unset();     // unset $_SESSION variable for the run-time 
    session_destroy();   // destroy session data in storage
}
$_SESSION['LAST_ACTIVITY'] = time(); // update last activity time stamp

使用每个请求更新会话数据还会更改会话文件的修改日期，以便垃圾收集器不会过早删除会话。

您还可以使用额外的时间戳定期重新生成会话 ID，以避免对会话进行攻击，例如会话固定：

if (!isset($_SESSION['CREATED'])) {
    $_SESSION['CREATED'] = time();
} else if (time() - $_SESSION['CREATED'] > 1800) {
    // session started more than 30 minutes ago
    session_regenerate_id(true);    // change session ID for the current session and invalidate old session ID
    $_SESSION['CREATED'] = time();  // update creation time
}

笔记：

  • session.gc_maxlifetime应至少等于此自定义过期处理程序的生命周期（本例中为 1800）；
  • 如果您想在活动 30 分钟后 而 不是在 开始后 30 分钟后使会话过期，您还需要使用setcookiewith expire oftime()+60*30来保持会话 cookie 处于活动状态。
