如何将空目录添加到 Git 存储库？

Arya

2023-02-16 18:47

如何将空目录（不包含任何文件）添加到 Git 存储库？



解答

使目录（在存储库中）保持（几乎）为空的另一种方法是在该目录中创建一个.gitignore包含以下四行的文件：

# Ignore everything in this directory
*
# Except this file
!.gitignore

这样您就不必像在 m104 的解决方案中那样正确地获得订单。

这也带来了好处，即当您执行 git status 时，该目录中的文件不会显示为“未跟踪”。

使@GreenAsJade的评论持久化：

我认为值得注意的是，这个解决方案正是按照问题的要求进行的，但也许并不是许多正在寻找这个问题的人一直在寻找的。此解决方案保证目录保持为空。它说“我真的不想在这里签入文件”。与“我这里没有任何文件要签入，但我需要这里的目录，文件可能会在稍后出现”相反。

如何创建 GUID / UUID？

Arya

2023-02-16 18:47

如何在 JavaScript 中创建 GUID（全局唯一标识符）？GUID / UUID 应至少为 32 个字符，并且应保持在 ASCII 范围内以避免在传递它们时出现问题。

我不确定所有浏览器上都有哪些例程，内置随机数生成器的“随机”和播种方式等。



解答

根据RFC 4122， UUID（通用唯一标识符）也称为 GUID（全局唯一标识符），是旨在提供某些唯一性保证的标识符。

虽然可以在几行 JavaScript 代码中实现符合 RFC 的 UUID（例如，请参阅下面的@broofa 的回答），但有几个常见的陷阱：

  • 无效的 ID 格式（UUID 必须采用“ xxxxxxxx-xxxx-Mxxx-Nxxx-xxxxxxxxxxxx”形式，其中 x 是 [0-9, af] 之一 M 是 [1-5] 之一， N 是 [8, 9, a, or b]
  • 使用低质量的随机源（例如Math.random）

因此，鼓励为生产环境编写代码的开发人员使用严格、维护良好的实现，例如uuid模块。

“INNER JOIN”和“OUTER JOIN”有什么区别？

Arya

2023-02-16 18:46

LEFT OUTER JOIN另外， , RIGHT OUTER JOIN, 和如何FULL OUTER JOIN适应？



解答

假设您要加入没有重复的列，这是一种非常常见的情况：

  • A 和 B 的内部连接给出 A 与 B 相交的结果，即维恩图相交的内部部分。

  • A 和 B 的外连接给出 A 并集 B 的结果，即维恩图并集的外部部分。

例子

假设你有两个表，每个表只有一列，数据如下：

A    B
-    -
1    3
2    4
3    5
4    6

请注意，(1,2) 是 A 独有的，(3,4) 是通用的，而 (5,6) 是 B 独有的。

内部联接

使用任一等效查询的内部联接给出两个表的交集，即它们共有的两行。

select * from a INNER JOIN b on a.a = b.b;
select a.*, b.*  from a,b where a.a = b.b;

a | b
--+--
3 | 3
4 | 4

左外连接

左外连接将给出 A 中的所有行，以及 B 中的所有公共行。

select * from a LEFT OUTER JOIN b on a.a = b.b;
select a.*, b.*  from a,b where a.a = b.b(+);

a |  b
--+-----
1 | null
2 | null
3 |    3
4 |    4

右外连接

右外连接将给出 B 中的所有行，以及 A 中的任何公共行。

select * from a RIGHT OUTER JOIN b on a.a = b.b;
select a.*, b.*  from a,b where a.a(+) = b.b;

a    |  b
-----+----
3    |  3
4    |  4
null |  5
null |  6

全外连接

完全外部联接将为您提供 A 和 B 的并集，即 A 中的所有行和 B 中的所有行。如果 A 中的某些内容在 B 中没有相应的数据，则 B 部分为空，反之亦然反之亦然。

select * from a FULL OUTER JOIN b on a.a = b.b;

 a   |  b
-----+-----
   1 | null
   2 | null
   3 |    3
   4 |    4
null |    6
null |    5


更多建议

维恩图并不能真正为我做这件事。

例如，它们不显示交叉连接和内部连接之间的任何区别，或者更一般地说，不显示不同类型的连接谓词之间的任何区别，或者提供一个框架来推理它们将如何运行。

理解逻辑处理是无可替代的，而且掌握起来也相对简单。

  1. 想象一下交叉连接。
  2. 针对步骤 1 中的所有行评估on子句，保留谓词评估为的那些行true
  3. （仅适用于外部连接）添加回第 2 步中丢失的任何外部行。

（注意：在实践中，查询优化器可能会找到比上面的纯逻辑描述更有效的查询执行方式，但最终结果必须相同）

我将从完整外部联接 的动画版本开始。进一步的解释如下。

解释

源表

首先从（又名笛卡尔积）开始CROSS JOIN。这没有子句ON，只是返回两个表中行的每个组合。

从交叉连接 B 中选择 A.颜色、B.颜色

内连接和外连接有一个“ON”子句谓词。

  • 内部联接。 评估交叉连接结果中所有行的“ON”子句中的条件。如果为真，则返回连接的行。否则丢弃它。
  • 左外连接。 与 inner join 相同，然后对于左表中不匹配任何输出的任何行，右表列输出这些行的 NULL 值。
  • 右外连接。 与 inner join 相同，然后对于右表中不匹配任何输出的任何行，这些行与左表列的 NULL 值一起输出。
  • 全外连接。 与内连接相同，然后像左外连接一样保留左非匹配行，按照右外连接保留右非匹配行。

一些例子

SELECT A.Colour，B.Colour FROM A INNER JOIN B ON A.Colour = B.Colour

以上是经典的 equi join。

动画版

SELECT A.Colour, B.Colour FROM A INNER JOIN B ON A.Colour NOT IN

('Green','Blue')

内部连接条件不一定是相等条件，也不需要引用来自两个（甚至任何一个）表的列。A.Colour NOT IN ('Green','Blue')对交叉连接返回的每一行进行评估。

选择 A.Colour, B.Colour FROM A INNER JOIN B ON 1 =1

对于交叉连接结果中的所有行，连接条件的计算结果为真，因此这与交叉连接相同。16行的图我就不重复了。

SELECT A.Colour，B.Colour FROM A LEFT OUTER JOIN B ON A.Colour = B.Colour

外连接在逻辑上的计算方式与内连接相同，不同之处在于，如果左表中的一行（对于左连接）根本不与右表中的任何行连接，则它会在结果中保留NULL为右手列。

SELECT A.Colour, B.Colour FROM A LEFT OUTER JOIN B ON A.Colour = B.Colour

WHERE B.Colour IS NULL 中

这只是将先前的结果限制为仅返回行 where B.Colour IS NULL。在这种特殊情况下，这些将是保留的行，因为它们在右侧表中没有匹配项，并且查询返回表中不匹配的单个红色行B。这称为反半连接。

重要的是为测试选择一个列，IS NULL该列要么不可为空，要么连接条件确保NULL排除任何值，以便此模式正常工作，并避免只返回恰好具有NULL该值的行除了不匹配的行之外的列。

从 A.Colour = B.Colour 上的右外连接 B 中选择 A.Colour, B.Colour

右外连接的行为与左外连接类似，只是它们保留右表中的非匹配行并且空扩展左手列。

从 A.Colour = B.Colour 上的完整外部连接 B 中选择 A.Colour、B.Colour

全外连接结合了左连接和右连接的行为，并保留了左右表中的非匹配行。

SELECT A.Colour，B.Colour FROM A FULL OUTER JOIN B ON 1 = 0

交叉连接中没有行与1=0谓词匹配。两侧的所有行都使用正常的外部连接规则保留，另一侧表的列中为 NULL。

SELECT COALESCE(A.Colour, B.Colour) AS Color FROM A FULL OUTER JOIN B ON 1

= 0 颜色

对前面的查询稍作修改，就可以模拟UNION ALL两个表中的一个。

SELECT A.Colour，B.Colour FROM A LEFT OUTER JOIN B ON A.Colour = B.Colour

WHERE B.Colour = 'Green'

请注意，该WHERE子句（如果存在）逻辑上在连接之后运行。一个常见的错误是执行左外连接，然后在右表中包含一个带有条件的 WHERE 子句，最终排除了不匹配的行。以上最终执行了外部连接......

... And then the "Where" clause runs. NULL= 'Green' does not evaluate to true so the row preserved by the outer join ends up discarded (along with the blue one) effectively converting the join back to an inner one.

If the intention was to include only rows from B where Colour is Green and all rows from A regardless the correct syntax would be

SELECT A.Colour, B.Colour FROM A LEFT OUTER JOIN B ON A.Colour = B.Colour

AND B.Colour = 'Green'
