有效电子邮件地址的最大长度是多少？

Arya

2023-02-17 04:51

有效电子邮件地址的最大长度是多少？它是由任何标准定义的吗？



解答

电子邮件地址不得超过 254 个 字符。

这在提交勘误表后被 IETF 接受。任何给定地址的完整诊断都可以在线获得。RFC 3696 的原始版本将 320 描述为最大长度，但 John Klensin 随后接受了一个不正确的值，因为 Path 被定义为

Path = "<" [ A-d-l ":" ] Mailbox ">"

因此 Mailbox 元素（即电子邮件地址）周围有尖括号以形成 Path，其最大长度为 254 个字符，以将 Path 长度限制为 256 个字符或更少。

RFC 5321规定的最大长度规定：

反向路径或正向路径的最大总长度为 256 个字符。

RFC 3696 已在此处更正。

人们应该特别注意针对 RFC 3696 的勘误表。三个规范示例实际上是无效地址。

我整理了几百个测试地址，您可以在http://www.dominicsayers.com/isemail找到这些地址


更多建议

320

这些片段看起来像这样

{64}@{255}

64 + 1 + 255 = 320

如果您正在验证电子邮件，您还应该阅读此内容：在阅读 RFC 之前，我知道如何验证电子邮件地址

全局 Git 忽略

Arya

2023-02-17 04:51

我想将 Git 设置为全局忽略某些文件。

.gitignore我在我的主目录 ( ) 中添加了一个文件/Users/me/，并向其中添加了以下行：

*.tmproj

但它并没有忽略这种类型的文件，知道我做错了什么吗？



解答

您需要设置全局core.excludesfile配置文件以指向此全局忽略文件，例如：

  • nix 或 Windows git bash:

    git config --global core.excludesFile '~/.gitignore'

窗口命令：

git config --global core.excludesFile "%USERPROFILE%\.gitignore"

Windows PowerShell：

git config --global core.excludesFile "$Env:USERPROFILE\.gitignore"

对于 Windows，它设置为位置C:\Users\%username%\.gitignore。您可以通过执行以下操作来验证配置值是否正确：

git config --global core.excludesFile

结果应该是您的用户配置文件的扩展路径.gitignore。确保该值不包含未展开的%USERPROFILE%字符串。

重要提示 ：以上命令只会设置 git 将使用的忽略文件的位置。该文件仍必须在该位置手动创建并填充忽略列表。（来自muruge的评论）

您可以在https://docs.github.com/en/get-started/getting-started-with-git/ignoring- files#configuring-ignored-files-for-all-repositories-on-your阅读该命令-电脑


更多建议

尽管其他答案是正确的，但它们正在设置全局配置值，而全局 git 忽略文件有一个默认的 git 位置。

您可以通过以下方式检查全局 gitignore 是否存在：

git config --get core.excludesfile

通常，它位于不同操作系统的以下位置：

*尼克斯：

~/.config/git/ignore

视窗：

%USERPROFILE%\.config\git\ignore

您可能需要创建git目录和ignore文件并通过 告诉 git 位置git config --global core.excludesfile ~/.config/git/ignore，然后您可以将全局忽略放入该文件中，仅此而已！

为什么不使用 Double 或 Float 来表示货币呢？

Arya

2023-02-17 04:51

我一直被告知 永远不要 用doubleorfloat类型来表示金钱，这次我向你提出问题：为什么？

我确定有一个很好的理由，我只是不知道它是什么。



解答

因为 float 和 double 不能准确表示我们用于货币的以 10 为底的倍数。这个问题不仅适用于 Java，它适用于任何使用 base 2 浮点类型的编程语言。

在基数 10 中，您可以将 10.25 写为 1025 * 10 -2（整数乘以 10 的幂）。IEEE-754 浮点数不同，但考虑它们的一种非常简单的方法是乘以 2 的幂。例如，您可以查看 164 * 2 -4（整数乘以 2 的幂），它也等于 10.25。这不是数字在内存中的表示方式，但数学含义是相同的。

即使以 10 为基数，此表示法也无法准确表示大多数简单分数。例如，您不能表示 1/3：小数表示是重复的 (0.3333...)，因此没有有限整数可以乘以 10 的幂得到 1/3。您可以确定一长串 3 和一个小指数，例如 333333333 * 10 -10，但它不准确：如果您将其乘以 3，则不会得到 1。

但是，为了数钱，至少对于货币价值在美元一个数量级内的国家来说，通常你需要的只是能够存储 10 -2 的倍数，所以这并不重要那1/3不能代表。

float 和 double 的问题在于 绝大多数类似 金钱的数字都没有精确表示为整数乘以 2 的幂。事实上，0 和 1 之间的 0.01 的唯一倍数（这在交易时很重要有钱，因为它们是整数美分）可以精确表示为 IEEE-754 二进制浮点数的是 0、0.25、0.5、0.75 和 1。所有其他的都略有偏差。与 0.333333 示例类比，如果将浮点值 0.01 乘以 10，则不会得到 0.1。相反，你会得到类似 0.099999999786 的东西......

将钱表示为doubleorfloat可能一开始看起来不错，因为软件会舍去微小的错误，但是当您对不精确的数字执行更多的加法、减法、乘法和除法时，错误会加剧，您最终会得到明显可见的值不准确。这使得 float 和 double 不足以处理货币，因为货币需要以 10 为底的幂的倍数的完美精度。

几乎适用于任何语言的解决方案是使用整数代替，并计算美分。例如，1025 将是 10.25 美元。一些语言也有内置的类型来处理金钱。其中，Java 有类BigDecimal，Rust 有rust_decimal箱子，C# 有类型decimal。


更多建议

来自 Bloch, J., Effective Java，（第 2 版，第 48 项。第 3 版，第 60 项）：

floatand类型double特别不适合货币计算，因为不可能将 0.1（或任何其他 10 的负幂）精确地表示为 afloat或double。

例如，假设您有 1.03 美元并且花费了 42c。你还剩多少钱？

System.out.println(1.03 - .42);

打印出来0.6100000000000001。

解决此问题的正确方法是使用BigDecimal,int或long进行货币计算。

虽然BigDecimal有一些警告（请参阅当前接受的答案）。
