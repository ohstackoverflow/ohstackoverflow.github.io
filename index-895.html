将 const std::string & 作为参数传递的时代结束了吗？

Arya

2023-02-17 19:03

std::vector我最近听到 Herb Sutter 的一次演讲，他说过去std::string的理由const &已经基本消失了。他建议现在最好编写如下函数：

std::string do_something ( std::string inval )
{
   std::string return_val;
   // ... do stuff ...
   return return_val;
}

我知道return_val函数返回时将是一个右值，因此可以使用非常便宜的移动语义返回。但是，inval仍然比引用（通常作为指针实现）的大小大得多。这是因为 a具有各种组件，包括指向堆的指针和用于短字符串优化的std::string成员。char[]所以在我看来，通过引用传递仍然是一个好主意。

谁能解释为什么 Herb 会这样说？



解答

赫布之所以说出他的话，就是因为这样的案例。

假设我有A调用函数的函数B，调用函数的函数C。并将A字符串传递B到C. A不知道或不关心C；都A知道是B。也就是说，C是 的一个实现细节B。

假设 A 定义如下：

void A()
{
  B("value");
}

如果 B 和 C 采用字符串 by const&，那么它看起来像这样：

void B(const std::string &str)
{
  C(str);
}

void C(const std::string &str)
{
  //Do something with `str`. Does not store it.
}

一切都很好。你只是传递指点，没有复制，没有移动，每个人都很开心。C需要 aconst&因为它不存储字符串。它只是使用它。

现在，我想做一个简单的更改：C需要将字符串存储在某处。

void C(const std::string &str)
{
  //Do something with `str`.
  m_str = str;
}

你好，复制构造函数和潜在的内存分配（忽略短字符串优化（SSO））。C++11 的移动语义应该可以消除不必要的复制构造，对吧？并A通过了一个临时的；没有理由C必须复制 数据 。它应该带着给它的东西潜逃。

除了它不能。因为它需要一个const&.

如果我更改C为按值获取其参数，那只会导致B将副本复制到该参数中；我一无所获。

因此，如果我只是str通过所有函数按值传递，依赖于std::move随机排列数据，我们就不会有这个问题。如果有人想坚持下去，他们可以。如果他们不这样做，哦，好吧。

它更贵吗？是的; 移动到一个值比使用引用更昂贵。它比副本便宜吗？不适用于具有 SSO 的小字符串。值得做吗？

这取决于您的用例。你有多讨厌内存分配？

不 。许多人（包括 Dave Abrahams）在它适用的领域之外采纳了这个建议，并将其简化以适用于 所有 std::string参数—— 始终 按值传递std::string对于任何和所有任意参数和应用程序来说都不是“最佳实践”，因为优化这些会谈/文章侧重于 仅适用于一组受限制的案例 。

如果您要返回一个值、改变参数或获取值，那么按值传递可以节省昂贵的复制并提供语法上的便利。

与以往一样，当您不需要 copy 时， 通过 const 引用传递可以节省大量复制。

函数式编程中如何存在时间函数？

Arya

2023-02-17 19:02

我不得不承认我对函数式编程知之甚少。我从各处读到它，因此了解到在函数式编程中，无论函数被调用多少次，函数都会为相同的输入返回相同的输出。它就像一个数学函数，对于函数表达式中涉及的输入参数的相同值，计算出相同的输出。

例如，考虑一下：

f(x,y) = x*x + y; // It is a mathematical function

无论您使用多少次f(10,4)，它的价值始终存在104。因此，无论您在何处编写f(10,4)，都可以将其替换为104，而不会改变整个表达式的值。此属性称为表达式的引用透明性。

正如维基百科所说（链接），

相反，在函数式代码中，函数的输出值仅取决于函数的输入参数，因此使用相同的参数 x 值调用函数 f 两次将产生相同的结果 f(x) 两次。

函数式编程中可以存在时间函数（返回 当前时间）吗？

  • 如果是，那它怎么可能存在？不违反函数式编程的原则吗？它特别违反了函数式编程的属性之一的引用透明性（如果我理解正确的话）。

  • 或者如果不是，那么如何知道函数式编程中的当前时间？



解答

另一种解释方式是：没有 函数 可以获取当前时间（因为它一直在变化），但是一个 动作 可以获取当前时间。假设这getClockTime是一个常量（如果你愿意，也可以是零元函数），它代表获取当前时间的 动作。 这个 动作 不管什么时候用，每次都是一样的，所以是实实在在的常量。

同样，假设print是一个函数，它需要一些时间表示并将其打印到控制台。由于函数调用在纯函数式语言中不会产生副作用，因此我们假设它是一个函数，它接受时间戳并将打印它的 操作 返回到控制台。同样，这是一个真正的函数，因为如果你给它相同的时间戳，它每次都会返回相同的打印 操作。

现在，如何将当前时间打印到控制台？嗯，你必须把这两个动作结合起来。那我们该怎么做呢？我们不能只传递getClockTime给print，因为 print 需要时间戳，而不是动作。但我们可以想象有一个运算符 ，>>=它 结合了 两个动作，一个获取时间戳，另一个将一个作为参数并打印出来。将其应用于前面提到的操作，结果是……tadaaa……一个获取当前时间并打印它的新操作。顺便说一句，这正是 Haskell 中的做法。

Prelude> System.Time.getClockTime >>= print
Fri Sep  2 01:13:23 東京 (標準時) 2011

因此，从概念上讲，您可以这样看待它：一个纯函数式程序不执行任何 I/O，它定义了一个 动作 ，然后由运行时系统执行。 每次的动作 都是一样的，但是执行的结果取决于执行时的情况。

我不知道这是否比其他解释更清楚，但它有时会帮助我这样想。

InvariantCulture 和 Ordinal 字符串比较的区别

Arya

2023-02-17 19:02

在 c# 中比较两个字符串是否相等时，InvariantCulture 和 Ordinal 比较有什么区别？



解答

不变文化

使用一组“标准”字符顺序（a、b、c、...等）。这与某些特定的语言环境形成对比，这些语言环境可能以不同的顺序对字符进行排序（'a-with-acute' 可能在 'a' 之前 或 之后，具体取决于语言环境，等等）。

序数

另一方面，纯粹查看表示字符的原始字节的值。

http://msdn.microsoft.com/en-us/library/e6883c06.aspx上有一个很好的示例，它显示了各种 StringComparison 值的结果。一路走到最后，显示（节选）：

StringComparison.InvariantCulture:
LATIN SMALL LETTER I (U+0069) is less than LATIN SMALL LETTER DOTLESS I (U+0131)
LATIN SMALL LETTER I (U+0069) is less than LATIN CAPITAL LETTER I (U+0049)
LATIN SMALL LETTER DOTLESS I (U+0131) is greater than LATIN CAPITAL LETTER I (U+0049)

StringComparison.Ordinal:
LATIN SMALL LETTER I (U+0069) is less than LATIN SMALL LETTER DOTLESS I (U+0131)
LATIN SMALL LETTER I (U+0069) is greater than LATIN CAPITAL LETTER I (U+0049)
LATIN SMALL LETTER DOTLESS I (U+0131) is greater than LATIN CAPITAL LETTER I (U+0049)

您可以看到 InvariantCulture 产生 (U+0069, U+0049, U+00131)，Ordinal 产生 (U+0049, U+0069, U+00131)。


更多建议

它确实很重要，例如 - 有一种叫做字符扩展的东西

var s1 = "Strasse";
var s2 = "Straße";

s1.Equals(s2, StringComparison.Ordinal);           //false
s1.Equals(s2, StringComparison.InvariantCulture);  //true

随着InvariantCultureß 字符被扩展为 ss。
