将开发分支与 master 合并

Arya

2023-02-17 09:48

我有两个分支，即master在developmentGitHub 存储库中。如图所示，我正在开发分支中进行所有开发。

git branch development
git add *
git commit -m "My initial commit message"
git push -u origin development

现在我想将分支上的所有更改合并development到master. 我目前的做法是：

git checkout master 
git merge development
git push -u origin master

如果我遵循的程序正确，请告诉我。



解答

我通常喜欢合并master到development第一个，这样如果有任何冲突，我可以在development分支本身解决并且我master保持干净。

(on branch development)$ git merge master
(resolve any merge conflicts if there are any)
git checkout master
git merge development (there won't be any conflicts now)

这两种方法没有太大区别，但我注意到有时我不想将分支合并到合并master它们之后，或者在合并它们之前还有更多工作要做，所以我倾向于在最后的东西之前保持master不变。

编辑：来自评论

如果您想跟踪谁进行了合并以及何时进行了合并，您可以--no- ff在合并时使用标志来执行此操作。这通常仅在合并development到master（最后一步）时有用，因为您可能需要在工作流中多次合并master到（第一步），并且为这些创建提交节点可能不是很有用。development

git merge --no-ff development


更多建议

就个人而言，我的方法与您的方法相似，多了几个分支，并且在它们返回到 master 时压缩了一些提交。

我的一位同事不喜欢如此频繁地切换分支，而是留在开发分支上，并在开发分支上执行类似于以下内容的所有操作。

git fetch origin master    
git merge master    
git push origin development:master

第一行确保他有自上次更新他的本地存储库以来已经对 master 进行的任何上游提交。

第二个将这些更改（如果有的话）从 master 拉到开发中

第三个将开发分支（现在与 master 完全合并）向上推到 origin/master。

我可能对他的基本工作流程有一点误解，但这是它的主要要点。

malloc 和 calloc 的区别？

Arya

2023-02-17 09:47

做之间有什么区别：

ptr = malloc(MAXELEMS * sizeof(char *));

和：

ptr = calloc(MAXELEMS, sizeof(char*));

什么时候使用callocovermalloc或反之亦然？



解答

calloc()为您提供一个零初始化的缓冲区，同时malloc()保留未初始化的内存。

对于大型分配，主流操作系统下的大多数calloc实现将从操作系统（例如通过 POSIXmmap(MAP_ANONYMOUS)或 Windows VirtualAlloc）获取已知归零页面，因此不需要将它们写入用户空间。这也是正常malloc从操作系统获取更多页面的方式；calloc只是利用操作系统的保证。

这意味着calloc内存仍然可以是“干净的”和延迟分配的，并且写时复制映射到系统范围内共享的零物理页。（假设系统具有虚拟内存。）例如，通过Linux 上的性能实验可以看到效果。

有些编译器甚至可以为您将 malloc + memset(0) 优化为 calloc，但如果您想要清零内存，最好只在源代码中使用 calloc。（或者，如果您试图预先对其进行故障处理以避免以后发生页面故障，那么该优化将使您的尝试失败。）

如果您在写入内存之前不打算读取内存，请使用malloc它（可能）从其内部空闲列表中为您提供脏内存，而不是从操作系统获取新页面。（或者不是将空闲列表中的一块内存归零以进行小分配）。

如果没有操作系统，或者它不是一个花哨的多用户操作系统，它可以将页面归零以阻止进程之间的信息泄漏，那么嵌入式实现可能会将它自己留给零内存calloc。calloc

在嵌入式 Linux 上，malloc 可以mmap(MAP_UNINITIALIZED|MAP_ANONYMOUS)，它只对某些嵌入式内核启用，因为它在多用户系统上不安全。


更多建议

一个鲜为人知的区别是，在具有乐观内存分配的操作系统中，如 Linux，返回的指针在malloc程序实际接触它之前不受实际内存的支持。

calloc确实触及了内存（它在上面写了零），因此您可以确定操作系统正在支持实际 RAM（或交换）的分配。这也是它比 malloc 慢的原因（不仅必须将其归零，操作系统还必须通过可能换出其他进程来找到合适的内存区域）
