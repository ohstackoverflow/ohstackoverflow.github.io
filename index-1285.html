target="_blank" 与 target="_new"

Arya

2023-02-18 05:57

如果我只想在新选项卡/窗口中打开链接，我应该使用 和 之间有<a target="_new">什么区别？<a target="_blank">



解答

使用“_空白”

根据HTML5 规范：

有效的浏览上下文名称 是至少有一个不以 U+005F LOW LINE 字符开头的字符的任何字符串。（以下划线开头的名称是为特殊关键字保留的。）

有效 的浏览上下文名称或关键字 是任何字符串，它要么是有效的浏览上下文名称，要么是不区分大小写的 ASCII 匹配项：_blank、_self、_parent 或 _top。”-来源

这意味着在 HTML5 中没有这样的关键字_new，在 HTML4（以及因此的 XHTML）中也没有。这意味着，如果您将其用作目标属性的值，将不会有任何一致的行为。

安全建议

正如丹尼尔和迈克尔在评论中指出的那样，当使用目标_blank指向不受信任的网站时，您还应该设置rel="noopener". 这可以防止打开站点通过 JavaScript 干扰打开器。有关更多信息，请参阅这篇文章。


更多建议

target="_blank"当用户单击链接时，using将指示浏览器创建一个新的浏览器选项卡或窗口。

根据规范，使用target="_new"在技术上是无效的，但据我所知，每个浏览器的行为方式都相同：

  • 它将搜索上下文名称为“_new”的选项卡或窗口
  • 如果找到“_new”选项卡/窗口，则将 URL 加载到其中
  • 如果未找到，将创建一个新的选项卡/窗口，上下文名称为“_new”，并将 URL 加载到其中

Note 的target="_new"行为与 完全相同target="new"，后者是有效的 HTML，而前者是无效的 HTML。

更让人困惑的是，在 HTML4 中该target属性已被弃用。在 HTML5 中，这个决定被逆转，它再次成为规范的官方部分。target无论您使用什么版本的 HTML，所有浏览器都支持，但如果您的文档类型是 HTML4，一些验证器会将使用标记为已弃用。

哪个更快：堆栈分配或堆分配

Arya

2023-02-18 05:56

这个问题听起来可能相当初级，但这是我与另一位合作的开发人员进行的辩论。

我正在注意尽可能地堆栈分配东西，而不是堆分配它们。他在和我说话并看着我的肩膀并评论说没有必要，因为他们在表现方面是一样的。

我一直认为堆栈的增长是恒定的时间，堆分配的性能取决于分配（找到合适大小的空洞）和取消分配（折叠空洞以减少碎片，如如果我没记错的话，许多标准库实现在删除期间需要时间来执行此操作）。

这让我印象深刻，因为它可能非常依赖编译器。特别是对于这个项目，我正在为PPC架构使用Metrowerks编译器。了解这种组合最有帮助，但一般来说，对于 GCC 和 MSVC++，情况如何？堆分配的性能不如堆栈分配吗？没有区别吗？或者差异如此之小以至于变成毫无意义的微优化。



解答

堆栈分配要快得多，因为它真正做的只是移动堆栈指针。使用内存池，您可以从堆分配中获得可比的性能，但这会稍微增加复杂性并带来一些麻烦。

此外，堆栈与堆不仅仅是性能方面的考虑；它还会告诉您很多有关对象的预期寿命的信息。


更多建议

堆栈要快得多。它实际上只在大多数架构上使用一条指令，在大多数情况下，例如在 x86 上：

sub esp, 0x10

（这会将堆栈指针向下移动 0x10 个字节，从而“分配”这些字节供变量使用。）

当然，堆栈的大小非常非常有限，因为如果过度使用堆栈分配或尝试进行递归，您很快就会发现:-)

此外，没有理由优化不需要它的代码的性能，例如通过分析证明的。“过早的优化”通常会导致比其价值更多的问题。

我的经验法则：如果我知道我将 在编译时 需要一些数据，并且它的大小在几百字节以下，我会对其进行堆栈分配。否则我堆分配它。

以编程方式查找机器上的内核数

Arya

2023-02-18 05:56

有没有办法以独立于平台的方式确定一台机器有多少个来自 C/C++ 的内核？如果不存在这样的东西，那么如何确定每个平台（Windows/*nix/Mac）呢？



解答

C++11

#include <thread>

//may return 0 when not able to detect
const auto processor_count = std::thread::hardware_concurrency();

参考：std::thread::hardware_concurrency

在 C++11 之前的 C++ 中，没有可移植的方法。相反，您需要使用以下一种或多种方法（由适当的#ifdef行保护）：

  • Win32

    SYSTEM_INFO sysinfo;

    GetSystemInfo(&sysinfo); int numCPU = sysinfo.dwNumberOfProcessors;

  • Linux、Solaris、AIX 和 Mac OS X >=10.4（即 Tiger 以上版本）

    int numCPU = sysconf(_SC_NPROCESSORS_ONLN);

  • FreeBSD、MacOS X、NetBSD、OpenBSD 等。

    int mib[4];

    int numCPU; std::size_t len = sizeof(numCPU);

    / set the mib for hw.ncpu / mib[0] = CTL_HW; mib[1] = HW_AVAILCPU; // alternatively, try HW_NCPU;

    / get the number of CPUs from the system / sysctl(mib, 2, &numCPU, &len, NULL, 0);

    if (numCPU < 1) { mib[1] = HW_NCPU; sysctl(mib, 2, &numCPU, &len, NULL, 0); if (numCPU < 1) numCPU = 1; }

  • HPUX

    int numCPU = mpctl(MPC_GETNUMSPUS, NULL, NULL);

  • 爱丽克斯

    int numCPU = sysconf(_SC_NPROC_ONLN);

  • Objective-C（Mac OS X >=10.5 或 iOS）

    NSUInteger a = [[NSProcessInfo processInfo] processorCount];

    NSUInteger b = [[NSProcessInfo processInfo] activeProcessorCount];


更多建议

此功能是 C++11 标准的一部分。

#include <thread>

unsigned int nthreads = std::thread::hardware_concurrency();

对于较旧的编译器，您可以使用Boost.Thread库。

#include <boost/thread.hpp>

unsigned int nthreads = boost::thread::hardware_concurrency();

在任何一种情况下，hardware_concurrency()都会返回硬件能够根据 CPU 内核和超线程单元的数量并发执行的线程数。
