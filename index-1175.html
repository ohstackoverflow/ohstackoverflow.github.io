IBOutlets 在 ARC 下应该强还是弱？

Arya

2023-02-18 03:15

我正在使用 ARC 专门为 iOS 5 开发。IBOutlets 到UIViews（和子类）应该是strong还是weak？

下列：

@property (nonatomic, weak) IBOutlet UIButton *button;

会摆脱所有这些：

- (void)viewDidUnload
{
    // ...
    self.button = nil;
    // ...
}

这样做有什么问题吗？模板使用的strong是从“Interface Builder”编辑器直接连接到标题时创建的自动生成的属性，但为什么呢？已经UIViewController有对其子视图的strong引用。view



解答

Apple 目前推荐的最佳实践是 IBOutlets 是 强大的 ，除非特别需要 弱化 来避免保留周期。正如 Johannes 上面提到的，这是在 WWDC 2015 的“在 Interface Builder 中实施 UI 设计”会议中评论的，一位 Apple 工程师说：

我要指出的最后一个选项是存储类型，它可以是强的也可以是弱的。一般来说，你应该让你的 outlet 更强大，尤其是当你将一个 outlet 连接到一个子视图或一个并不总是被视图层次结构保留的约束时。唯一真正需要使出口变弱的情况是，如果您有一个自定义视图，该视图引用了视图层次结构中的某些内容，并且通常不推荐这样做。

我在 Twitter 上向 IB 团队的一位工程师询问了这个问题，他确认 strong 应该是默认设置，并且正在更新开发人员文档。

Chrome 发送请求错误：TypeError: Converting circular structure to JSON

Arya

2023-02-18 03:15

我有以下...

chrome.extension.sendRequest({
  req: "getDocument",
  docu: pagedoc,
  name: 'name'
}, function(response){
  var efjs = response.reply;
});

调用以下..

case "getBrowserForDocumentAttribute":
  alert("ZOMG HERE");
  sendResponse({
    reply: getBrowserForDocumentAttribute(request.docu,request.name)
  });
  break;

但是，我的代码永远不会到达“ZOMG HERE”，而是在运行时抛出以下错误chrome.extension.sendRequest

 Uncaught TypeError: Converting circular structure to JSON
 chromeHidden.JSON.stringify
 chrome.Port.postMessage
 chrome.initExtension.chrome.extension.sendRequest
 suggestQuery

有谁知道是什么原因造成的？



解答

这意味着您在请求中传递的对象（我猜是pagedoc）具有循环引用，类似于：

var a = {};
a.b = a;

JSON.stringify不能像这样转换结构。

注意 ：DOM 节点就是这种情况，它们具有循环引用，即使它们没有附加到 DOM 树。在大多数情况下，每个节点都有一个 ownerDocument which 引用document。document至少通过document.body并再次document.body.ownerDocument引用回引用了 DOM 树document，这只是DOM 树中多个循环引用之一 。


更多建议

根据Mozilla 的 JSON 文档，JSON.stringify有第二个参数replacer可用于在解析树时过滤/忽略子项。但是，也许您可​​以避免循环引用。

在 Node.js 中我们不能。所以我们可以这样做：

function censor(censor) {
  var i = 0;

  return function(key, value) {
    if(i !== 0 && typeof(censor) === 'object' && typeof(value) == 'object' && censor == value) 
      return '[Circular]';

    if(i >= 29) // seems to be a harded maximum of 30 serialized objects?
      return '[Unknown]';

    ++i; // so we know we aren't using the original object anymore

    return value;  
  }
}

var b = {foo: {bar: null}};

b.foo.bar = b;

console.log("Censoring: ", b);

console.log("Result: ", JSON.stringify(b, censor(b)));

结果：

Censoring:  { foo: { bar: [Circular] } }
Result: {"foo":{"bar":"[Circular]"}}

不幸的是，在它自动假定它是循环的之前似乎最多有 30 次迭代。否则，这应该有效。我什至areEquivalent 从这里使用过，但JSON.stringify在 30 次迭代后仍然抛出异常。尽管如此，如果您真的需要它，它足以在顶层获得对象的体面表示。也许有人可以对此进行改进？在 HTTP 请求对象的 Node.js 中，我得到：

如果任何命令返回非零值，则中止 shell 脚本

Arya

2023-02-18 03:14

我有一个调用许多命令的 Bash shell 脚本。

如果任何命令返回非零值，我想让 shell 脚本自动退出并返回值 1。

如果不明确检查每个命令的结果，这可能吗？

例如，

dosomething1
if [[ $? -ne 0 ]]; then
    exit 1
fi

dosomething2
if [[ $? -ne 0 ]]; then
    exit 1
fi



解答

将此添加到脚本的开头：

set -e

如果一个简单的命令以非零退出值退出，这将导致 shell 立即退出。简单命令是不属于if, while, 或untiltest 或 or 列表的&&任何命令||。

有关更多详细信息，请参阅有关“set”内部命令的bash 手册。

当中间出现问题并破坏脚本其余部分的假设时，让脚本顽固地继续运行真的很烦人。我个人以set -e.

如果我专门使用 bash ，我将从

set -Eeuo pipefail

这涵盖了更多类似方式的错误处理。我认为这些是新 bash 程序的合理默认设置。有关这些选项的作用的更多信息，请参阅 bash 手册。


更多建议

添加到已接受的答案：

请记住，set -e有时是不够的，特别是如果您有管道。

例如，假设你有这个脚本

#!/bin/bash
set -e
./configure  > configure.log
make

configure...按预期工作：中止执行时出错。

明天你做一个看似微不足道的改变：

#!/bin/bash
set -e
./configure  | tee configure.log
make

...现在它不起作用。这在此处进行了解释，并提供了一种解决方法（仅限 Bash）：

#!/bin/庆典
设置-e
**设置-o管道失败**
./配置 | 三通配置.log
制作
