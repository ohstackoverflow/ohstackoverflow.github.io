对于 Python 中的错误/非法参数组合，我应该引发哪个异常？

Arya

2023-02-17 14:20

我想知道在 Python 中指示无效参数组合的最佳实践。我遇到过一些情况，你有这样的功能：

def import_to_orm(name, save=False, recurse=False):
    """
    :param name: Name of some external entity to import.
    :param save: Save the ORM object before returning.
    :param recurse: Attempt to import associated objects as well. Because you
        need the original object to have a key to relate to, save must be
        `True` for recurse to be `True`.
    :raise BadValueError: If `recurse and not save`.
    :return: The ORM object.
    """
    pass

唯一的烦恼是每个包都有自己的，通常略有不同BadValueError。我知道在 Java 中存在java.lang.IllegalArgumentException- 每个人都将在 Python 中创建自己的 s 是否很好理解BadValueError，或者是否有另一种首选方法？



解答

我只会提出ValueError，除非你需要一个更具体的异常..

def import_to_orm(name, save=False, recurse=False):
    if recurse and not save:
        raise ValueError("save must be True if recurse is True")

这样做真的没有意义- 您的自定义类在使用上与ValueErrorclass BadValueError(ValueError):pass相同，那么为什么不使用它呢？


更多建议

我会继承自ValueError

class IllegalArgumentError(ValueError):
    pass

有时创建您自己的异常更好，但继承自内置异常，它尽可能接近您想要的。

如果您需要捕获该特定错误，拥有一个名称会很有帮助。

是什么杀死了我的进程，为什么？

Arya

2023-02-17 14:19

我的应用程序在 Linux 上作为后台进程运行。它目前在终端窗口的命令行中启动。

最近有用户在执行应用一段时间后就莫名其妙死了。文本：

被杀

在终端上。这发生了两次。我问是否有人在不同的终端使用 kill 命令来终止进程？不。

在什么情况下 Linux 会决定终止我的进程？我相信 shell 显示“已终止”是因为进程在收到 kill(9) 信号后终止了。如果 Linux 发送了 kill 信号，系统日志中的某个地方是否应该有一条消息来解释它被杀死的原因？



解答

如果用户或系统管理员没有杀死内核可能具有的程序。内核只会在异常情况下终止进程，例如极度资源匮乏（想想 mem+swap 耗尽）。


更多建议

尝试：

dmesg -T| grep -E -i -B100 'killed process'

其中-B100表示杀死发生之前的行数。

在 Mac OS 上省略 -T 。

\n 和 \r 的区别？

Arya

2023-02-17 14:19

\n（换行）和\r（回车）有什么区别？

特别是，和之间有什么 实际 区别吗？有没有应该使用一个而不是另一个的地方？\n``\r



解答

就 ascii 码而言，它是 3——因为它们分别是 10 和 13；-)。

但说真的，有很多：

  • 在 Unix 和所有类 Unix 系统中，\n是行尾代码，\r没有什么特别的意思
  • 因此，在 C 和大多数以某种方式复制它（甚至远程）的语言中，\n是行尾的标准转义序列（根据需要转换为操作系统特定的序列或从操作系统特定的序列转换）
  • 在旧的 Mac 系统（OS X 之前的系统）中，\r是行尾代码
  • 在 Windows（和许多旧操作系统）中，行尾代码是 2 个字符，\r\n, 按此顺序
  • 作为（令人惊讶的；-）结果（回到比 Windows 早得多的操作系统），\r\n是 Internet 上文本格式的标准行终止
  • 对于类似机电电传打字机的“终端”，\r命令托架向左返回直到它到达最左边的挡块（缓慢的操作），\n命令滚轮卷起一条线（更快的操作）——这就是你总是有的原因\r 之前 \n，这样滚轮就可以在托架仍然向左移动的同时移动！-) 维基百科有更详细的解释。
  • 对于字符模式终端（通常模拟如上所述的更旧的打印终端），在原始模式下，\r并且\n行为相似（除了光标方面，因为没有 托架 或滚轮;-）

实际上，在写入文本文件的现代上下文中，您应该始终使用\n（如果您使用的是奇怪的操作系统，例如 Windows，底层运行时将转换它；-）。使用的唯一原因\r是如果你正在写一个字符终端（或者更可能是一个模拟它的“控制台窗口”）并且想要你写的下一行覆盖你刚刚写的最后一行（有时用于愚蠢的“ascii动画” “例如进度条的效果）——不过，这在 GUI 世界中已经相当过时了；-)。


更多建议

历史上 a\n用于将托架向下移动，而 the\r用于将托架移回页面左侧。
