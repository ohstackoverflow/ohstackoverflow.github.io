如何将 git 分支上的标签移动到不同的提交？

Arya

2023-02-17 05:41

我在 master 分支上创建了一个标签，v0.1如下所示：

git tag -a v0.1

但后来我意识到我仍然需要将一些更改合并到 0.1 版本的 master 中，所以我这样做了。但是现在我的v0.1标签卡在了错误的提交上（调用便利贴类比）。我希望它停留在 master 的最近一次提交上，而是停留在 master 上的第二个最近提交上。

我怎样才能将它移动到 master 上的最新提交？



解答

使用该-f选项git tag：

-f
--force

    Replace an existing tag with the given name (instead of failing)

-f您可能想结合使用-a以强制创建带注释的标签而不是非带注释的标签。

例子

  1. 在推送之前删除任何遥控器上的标签

    git push origin :refs/tags/<tagname>

  2. 替换标签以引用最近的提交

    git tag -fa <tagname>

  3. 将标签推送到远程源

    git push origin master --tags


更多建议

总结一下，如果您的遥控器被调用origin并且您正在master分支机构工作：

git tag -d <tagname>                  # delete the old tag locally
git push origin :refs/tags/<tagname>  # delete the old tag remotely
git tag <tagname> <commitId>          # make a new tag locally
git push origin <tagname>             # push the new local tag to the remote

描述：

  • 第 1 行删除本地环境中的标签。
  • 第 2 行删除远程环境中的标签。
  • 第 3 行将标记添加到不同的提交
  • 第 4 行将更改推送到远程

您还可以将第 4 行更改为git push origin --tags将所有本地标记更改/更新推送到远程仓库。

以上答案基于@eedeep 问题中的内容，以及Stuart Golodetz、Greg Hewgill和@ben-hocking 的答案，以及他们答案下方的评论，以及我的答案下方@NateS 的原始评论。

Python 中的旧样式类和新样式类有什么区别？

Arya

2023-02-17 05:40

Python 中的旧样式类和新样式类有什么区别？我什么时候应该使用一个或另一个？



解答

从 新式和经典类 ：

在 Python 2.1 之前，旧式类是用户唯一可用的风格。

（旧式）类的概念与类型的概念无关：如果x是旧式类的实例，则x.__class__指定 的类x，但type(x)始终是<type 'instance'>。

这反映了一个事实，即所有旧式实例，独立于它们的类，都是用一个称为实例的内置类型实现的。

Python 2.2 中引入了新型类，以统一类和类型的概念 。新式类只是用户定义的类型，不多也不少。

如果 x 是新式类的实例，则通常与（尽管不能保证这一点——允许新式类实例覆盖为 返回的值）type(x)相同。x.__class__``x.__class__

引入新型类的主要动机是提供具有完整元模型的统一对象模型 。

它还具有许多直接的好处，例如能够对大多数内置类型进行子类化，或者引入“描述符”，从而启用计算属性。

出于兼容性原因，默认情况下类仍然是旧式的 。

新式类是通过将另一个新式类（即类型）指定为父类来创建的，如果不需要其他父类，则指定“顶级类型”对象。

除了返回什么类型之外，新式类的行为在许多重要细节上都不同于旧式类。

其中一些更改是新对象模型的基础，例如调用特殊方法的方式。其他的是之前出于兼容性问题无法实现的“修复”，例如多重继承情况下的方法解析顺序。

Python 3 只有新式类 。

无论您是否继承自object，类在 Python 3 中都是新式的。


更多建议

声明方面：

新式类继承自 object 或另一个新式类。

class NewStyleClass(object):
    pass

class AnotherNewStyleClass(NewStyleClass):
    pass

老式课程没有。

class OldStyleClass():
    pass

Python 3 注意：

Python 3 不支持旧式类，因此上述任何一种形式都会产生新式类。

什么是幂等运算？

Arya

2023-02-17 05:40

什么是幂等运算？



解答

在计算中，幂等操作是指如果使用相同的输入参数多次调用它不会产生额外效果的操作。例如，从集合中删除一个项目可以被认为是对该集合的幂等操作。

在数学中，幂等运算是 f(f(x)) = f(x) 的 运算。例如，该abs()函数是幂等的，因为abs(abs(x)) = abs(x)对于所有x.

这些略有不同的定义可以通过考虑数学定义中的x 表示对象的状态而 f 是可能改变该对象的操作来协调。例如，考虑Pythonset及其discard方法。该discard方法从集合中删除一个元素，如果该元素不存在则不执行任何操作。所以：

my_set.discard(x)

与执行两次相同的操作具有完全相同的效果：

my_set.discard(x)
my_set.discard(x)

幂等操作通常用于网络协议的设计，其中执行操作的请求保证至少发生一次，但也可能发生多次。如果操作是幂等的，那么执行两次或更多次操作没有坏处。

有关更多信息，请参阅有关幂等性的维基百科文章。

上面的答案以前有一些不正确和误导性的例子。以下 2014 年 4 月之前撰写的评论指的是较旧的修订版。


更多建议

幂等操作可以重复任意次数，结果将与仅执行一次相同。在算术中，给数字加零是幂等的。

幂等性在“RESTful”Web 服务的上下文中被广泛讨论。REST 试图最大限度地利用 HTTP 来让程序访问 Web 内容，并且通常与基于 SOAP 的 Web 服务形成对比，后者只是在 HTTP 请求和响应中隧道远程过程调用样式服务。

REST 将 Web 应用程序组织成“资源”（如 Twitter 用户或 Flickr 图像），然后使用 HTTP 动词 POST、PUT、GET 和 DELETE 来创建、更新、读取和删除这些资源。

幂等性在 REST 中起着重要作用。如果您获取 REST 资源的表示（例如，从 Flickr 获取 jpeg 图像），并且操作失败，您可以一次又一次地重复获取，直到操作成功。对于网络服务来说，图像被获取多少次并不重要。同样，如果您使用 RESTful Web 服务更新您的 Twitter 帐户信息，您可以多次 PUT 新信息，以便从 Web 服务获得确认。PUT-ing 1000 次与 PUT-ing 一次相同。类似地，删除 REST 资源一千次与删除一次相同。因此，幂等性使得构建对通信错误具有弹性的 Web 服务变得容易得多。

进一步阅读： Richardson 和 Ruby 撰写的RESTful Web Services（幂等性在第 103-104 页讨论），以及 Roy Fielding关于 REST 的博士论文。Fielding 是 HTTP 1.1、RFC-2616 的作者之一，在9.1.2 节中讨论了幂等性。
