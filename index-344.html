如何修剪远程不再存在的本地跟踪分支？

Arya

2023-02-17 04:33

使用git remote prune originI 可以删除不再位于远程的本地分支。

但我也想删除从那些远程分支创建的本地分支（检查它们是否未合并会很好）。

我怎样才能做到这一点？



解答

剪枝后，可以通过 获取远程分支列表git branch -r。可以使用 检索带有远程跟踪分支的分支列表git branch -vv。因此，使用这两个列表，您可以找到不在远程列表中的远程跟踪分支。

这一行应该可以解决问题（需要bashor zsh，不适用于标准 Bourne shell）：

git fetch -p ; git branch -r | awk '{print $1}' | egrep -v -f /dev/fd/0 <(git branch -vv | grep origin) | awk '{print $1}' | xargs git branch -d

该字符串获取远程分支列表并将其egrep通过标准输入传递。git branch -vv并过滤具有远程跟踪分支的分支（使用和过滤那些具有）的分支origin，然后获取该输出的第一列，这将是分支名称。最后将所有分支名称传递到删除分支命令中。

由于它使用了该-d选项，因此它不会删除尚未合并到您运行此命令时所在分支的分支。


更多建议

如果要删除所有已经合并到master的本地分支，可以使用以下命令：

git branch --merged master | grep -v '^[ *]*master$' | xargs git branch -d

如果您使用mainmaster 分支，则应相应地修改命令：

git branch --merged main | grep -v '^[ *]*main$' | xargs git branch -d

如何在 Ruby 中将字符串转换为小写或大写

Arya

2023-02-17 04:32

在 Ruby 中如何将字符串转换为小写或大写？



解答

Ruby 有一些改变字符串大小写的方法。要转换为小写，请使用downcase：

"hello James!".downcase    #=> "hello james!"

类似地，upcase将每个字母大写并将capitalize字符串的第一个字母大写但将其余字母小写：

"hello James!".upcase      #=> "HELLO JAMES!"
"hello James!".capitalize  #=> "Hello james!"
"hello James!".titleize    #=> "Hello James!" (Rails/ActiveSupport only)

如果你想就地修改一个字符串，你可以给这些方法中的任何一个添加感叹号：

string = "hello James!"
string.downcase!
string   #=> "hello james!"

有关更多信息，请参阅String 的文档。


更多建议

您可以通过打开 irb 并运行以下命令来找出 String 上可用的所有方法：

"MyString".methods.sort

特别是对于字符串可用的方法列表：

"MyString".own_methods.sort

我用它来发现关于对象的新的和有趣的事情，否则我可能不知道这些事情的存在。

返回 IEnumerable<T> 与 IQueryable<T>

Arya

2023-02-17 04:32

IQueryable<T>return和 之间有什么区别IEnumerable<T>，什么时候应该优先选择另一个？

IQueryable<Customer> custs = from c in db.Customers
where c.City == "<City>"
select c;

IEnumerable<Customer> custs = from c in db.Customers
where c.City == "<City>"
select c;

两者都将被推迟执行吗？什么时候应该优先考虑一个？



解答

是的，两者都会给你延期执行。

不同之处在于它IQueryable<T>是允许 LINQ-to-SQL（实际上是 LINQ.-to- anything）工作的接口。因此，如果您进一步细化对 的查询IQueryable<T>，该查询将尽可能在数据库中执行。

对于这种IEnumerable<T>情况，它将是 LINQ-to-object，这意味着所有匹配原始查询的对象都必须从数据库加载到内存中。

在代码中：

IQueryable<Customer> custs = ...;
// Later on...
var goldCustomers = custs.Where(c => c.IsGold);

该代码将执行 SQL 以仅选择黄金客户。另一方面，下面的代码将执行数据库中的原始查询，然后过滤掉内存中的非金牌客户：

IEnumerable<Customer> custs = ...;
// Later on...
var goldCustomers = custs.Where(c => c.IsGold);

这是一个非常重要的区别，并且IQueryable<T>在许多情况下可以避免从数据库返回太多行。另一个典型的例子是进行分页：如果你使用Takeand Skipon IQueryable，你只会得到请求的行数；这样做IEnumerable<T>会导致所有行都加载到内存中。


更多建议

最佳答案很好，但它没有提到解释这两个接口“如何”不同的表达式树。基本上，有两组相同的 LINQ 扩展。Where(), Sum(), Count(),FirstOrDefault()等都有两个版本：一个接受函数，一个接受表达式。

  • 版本IEnumerable签名是：Where(Func<Customer, bool> predicate)

  • 版本IQueryable签名是：Where(Expression<Func<Customer, bool>> predicate)

您可能在没有意识到的情况下使用了这两个，因为它们都使用相同的语法调用：

例如Where(x => x.City == "<City>")适用于IEnumerable和IQueryable

  • Where()在集合上使用时IEnumerable，编译器将编译后的函数传递给Where()

  • Where()在集合上使用时IQueryable，编译器将表达式树传递给Where(). 表达式树就像反射系统，但用于代码。编译器将您的代码转换为一种数据结构，该数据结构以易于理解的格式描述您的代码的作用。

为什么要为这个表达式树而烦恼呢？ 我只想Where()过滤我的数据。主要原因是 EF 和 Linq2SQL ORM 都可以将表达式树直接转换为 SQL，这样您的代码执行速度就会快得多。

哦，这听起来像是免费的性能提升，AsQueryable()在那种情况下我应该到处使用吗？不，IQueryable只有当底层数据提供者可以用它做一些事情时才有用。将像普通的东西转换成List不会IQueryable给你带来任何好处。
