如何在 Python 中访问环境变量？

Arya

2023-02-16 20:17

如何在 Python 中获取环境变量的值？



解答

通过以下方式访问环境变量os.environ：

import os
print(os.environ['HOME'])

要查看所有环境变量的列表：

print(os.environ)

如果密钥不存在，尝试访问它会引发KeyError. 为避免这种情况：

# Returns `None` if the key doesn't exist
print(os.environ.get('KEY_THAT_MIGHT_EXIST'))

# Returns `default_value` if the key doesn't exist
print(os.environ.get('KEY_THAT_MIGHT_EXIST', default_value))

# Returns `default_value` if the key doesn't exist
print(os.getenv('KEY_THAT_MIGHT_EXIST', default_value))

获取远程分支到本地 GIT

Arya

2023-02-16 20:16

远程存储库包含各种分支，例如origin/daves_branch：

$ git branch -r
origin/HEAD -> origin/master
origin/daves_branch
origin/master

我如何获取daves_branch到本地存储库以便它跟踪origin/daves_branch？

我试过：

$ git fetch origin discover
$ git checkout discover



解答

更新：使用 Git 开关

下面写的所有信息都是准确的，但是git switch添加了一个新命令来简化工作。

如果daves_branch存在于远程存储库中，但不存在于您的本地分支中，您只需键入：

git switch daves_branch

由于您在本地没有分支，这将自动查看switch远程仓库。然后它还会自动设置远程分支跟踪。

请注意，如果daves_branch本地不存在，您需要git fetch先在本地使用switch.

原帖

您需要创建一个跟踪远程分支的本地分支。以下命令将创建一个名为 daves_branch 的本地分支，跟踪远程分支 origin/daves_branch 。当您推送更改时，远程分支将被更新。

对于最新版本的 Git：

git checkout --track origin/daves_branch

--track是git checkout -b [branch] [remotename]/[branch][remotename] 在这种情况下的 起源 和 [branch] 两次相同的简写，在这种情况下是 daves_branch 。

对于 Git 1.5.6.5 你需要这个：

git checkout --track -b daves_branch origin/daves_branch

对于 Git 1.7.2.3 及更高版本，这就足够了（它可能更早开始，但这是我能很快找到的最早确认）：

git checkout daves_branch

请注意，在最近的 Git 版本中，此命令不会创建本地分支，并且会将您置于“分离的 HEAD”状态。如果您想要本地分支机构，请使用该--track选项。

完整的细节在这里： 3.5 Git 分支 - 远程分支，跟踪分支

什么时候应该使用 static_cast、dynamic_cast、const_cast 和 reinterpret_cast？

Arya

2023-02-16 20:16

什么是正确的用法：

  • static_cast
  • dynamic_cast
  • const_cast
  • reinterpret_cast
  • (type)value（C型演员）
  • type(value)（函数式转换）

如何决定在哪些特定情况下使用哪一个？



解答

static_cast 是您应该尝试使用的第一个演员表。它执行类型之间的隐式转换（例如inttofloat或 pointer to void*），它还可以调用显式转换函数（或隐式转换函数）。在许多情况下，static_cast没有必要明确声明，但重要的是要注意T(something)语法等同于(T)something并且应该避免（稍后会详细介绍）。但是， AT(something, something_else)是安全的，并且保证调用构造函数。

static_cast也可以通过继承层次结构进行转换。向上铸造时（朝向基类）是不必要的，但向下铸造时只要不通过virtual继承进行铸造就可以使用。但是，它不进行检查，并且将static_cast层次结构下移到实际上不是对象类型的类型是未定义的行为。

const_cast 可用于删除或添加const到变量；没有其他 C++ 转换能够删除它（甚至不是reinterpret_cast）。重要的是要注意，const只有当原始变量是 undefined 时，修改以前的值才是重要的const；如果您使用它来取消const对未声明的内容的引用const，它是安全的。const例如，这在重载基于 的成员函数时很有用。它还可以用于const向对象添加内容，例如调用成员函数重载。

const_cast在 上也有类似的作用volatile，尽管这种情况不太常见。

dynamic_cast 专门用于处理多态性。您可以将指向任何多态类型的指针或引用强制转换为任何其他类类型（多态类型至少有一个声明或继承的虚函数）。您不仅可以将它用于向下投掷 - 您还可以向侧面投掷，甚至可以向上投掷另一条链条。将dynamic_cast寻找所需的对象并在可能的情况下将其返回。如果不能，它会nullptr在指针的情况下返回，或者std::bad_cast在引用的情况下抛出。

dynamic_cast但是有一些限制。virtual如果继承层次结构中有多个相同类型的对象（所谓的“可怕的钻石”）并且您没有使用继承，则它不起作用。它也只能通过公共继承——它总是无法通过protected或private继承。然而，这很少成为问题，因为这种形式的继承很少见。

reinterpret_cast 是最危险的转换，应该非常谨慎地使用。它将一种类型直接转换为另一种类型——例如将值从一个指针转换为另一个指针，或将指针存储在 中int，或其他各种讨厌的事情。在很大程度上，您获得的唯一保证reinterpret_cast是，通常如果您将结果转换回原始类型，您将获得完全相同的值（但如果中间类型小于原始类型，则 不会）。 也有许多转换 reinterpret_cast 不能做。它主要用于特别奇怪的转换和位操作，例如将原始数据流转换为实际数据，或将数据存储在指向对齐数据的指针的低位中。

C 风格的转换 和 函数风格的转换 分别是使用(type)objector 的转换type(object)，并且在功能上是等价的。它们被定义为以下第一个成功的：

  • const_cast
  • static_cast（尽管忽略访问限制）
  • static_cast（见上文），然后const_cast
  • reinterpret_cast
  • reinterpret_cast， 然后const_cast

因此，在某些情况下，它可以用作其他类型转换的替代品，但可能会非常危险，因为它能够转化为 a reinterpret_cast，并且在需要显式转换时应首选后者，除非您确定static_cast会成功或reinterpret_cast会失败. 即使那样，也要考虑更长、更明确的选择。

C 风格的转换在执行 a 时也会忽略访问控制static_cast，这意味着它们有能力执行其他转换无法执行的操作。不过，这主要是一种拼凑，在我看来，这只是避免 C 风格转换的另一个原因。


更多建议

  • 用于dynamic_cast在继承层次结构中转换指针/引用。

  • 用于static_cast普通类型转换。

  • 用于reinterpret_cast位模式的低级重新解释。使用时要格外小心。

  • 用于const_cast投掷const/volatile。避免这种情况，除非你被困在使用 const-incorrect API 中。
