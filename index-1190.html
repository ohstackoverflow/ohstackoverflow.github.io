如何找到 Git 分支的最近父级

Arya

2023-02-18 03:37

假设我有以下本地存储库，其中包含这样的提交树：

master --> a
            \
             \
      develop c --> d
               \
                \
         feature f --> g --> h

master这是我的 最新稳定版本代码 吗，这develop是我的 “下一个”版本代码 吗，feature是 正在准备的新功能吗？develop。

使用挂钩，我希望能够拒绝推送到feature我的远程存储库，除非提交f是 HEAD 的直接后代develop。即，提交树看起来像这样，因为功能已经git rebase打开d。

master --> a
            \
             \
      develop c --> d
                     \
                      \
               feature f --> g --> h

那么是否可以：

  • 确定父分支feature？
  • 确定父分支中的提交f是其后代？

从那里我会检查父分支的 HEAD 是什么，并查看f前任是否与父分支 HEAD 匹配，以确定该功能是否需要重新定位。



解答

假设远程存储库有一个 develop 分支的副本（你的初始描述描述它在本地存储库中，但听起来它也存在于远程），你应该能够实现我认为你想要的，但是方法和你想象的有点不一样。

Git 的历史基于提交的DAG。分支（和一般的“refs”）只是临时标签，指向不断增长的提交 DAG 中的特定提交。因此，分支之间的关系会随时间变化，但提交之间的关系不会。

    ---o---1                foo
            \
             2---3---o      bar
                  \
                   4
                    \
                     5---6  baz

它看起来像是baz基于（旧版本）bar？但是如果我们删除呢bar？

    ---o---1                foo
            \
             2---3
                  \
                   4
                    \
                     5---6  baz

现在看起来baz是基于foo. 但祖籍并baz没有改变。我们刚刚删除了一个标签（以及由此产生的悬空提交）。如果我们在 处添加一个新标签4呢？

    ---o---1                foo
            \
             2---3
                  \
                   4        quux
                    \
                     5---6  baz

现在看起来baz是基于quux. 尽管如此，血统并没有改变，只是标签改变了。

但是，如果我们问“commit 是6commit 的后代吗3？” （假设3和是完整的 SHA-1 提交名称），那么无论和标签是否存在6，答案都是“是” 。bar``quux

因此，您可以问诸如“推送的提交是开发 分支的当前提示的后代吗？”之类的问题，但您不能可靠地问“推送的提交的父分支是什么？”。

一个似乎接近你想要的最可靠的问题是：

对于所有推送的提交的祖先（不包括develop 的当前提示及其祖先），将当前的 develop 提示作为父级：

  • 是否至少存在一个这样的提交？
  • 所有这些提交都是单亲提交吗？

可以实现为：

pushedrev=...
basename=develop
if ! baserev="$(git rev-parse --verify refs/heads/"$basename" 2>/dev/null)"; then
    echo "'$basename' is missing, call for help!"
    exit 1
fi
parents_of_children_of_base="$(
  git rev-list --pretty=tformat:%P "$pushedrev" --not "$baserev" |
  grep -F "$baserev"
)"
case ",$parents_of_children_of_base" in
    ,)     echo "must descend from tip of '$basename'"
           exit 1 ;;
    ,*\ *) echo "must not merge tip of '$basename' (rebase instead)"
           exit 1 ;;
    ,*)    exit 0 ;;
esac

这将涵盖您想要限制的一些内容，但可能不会涵盖所有内容。

作为参考，这是一个扩展的示例历史记录：

    A                                   master
     \
      \                    o-----J
       \                  /       \
        \                | o---K---L
         \               |/
          C--------------D              develop
           \             |\
            F---G---H    | F'--G'--H'
                    |    |\
                    |    | o---o---o---N
                     \   \      \       \
                      \   \      o---o---P
                       \   \
                        R---S

上面的代码可以用来拒绝HandS而接受H', J, K, or N，但它也会接受Land P（他们涉及合并，但他们不合并 develop 的提示）。

也拒绝Land P，你可以换个问题问

对于所有推送的提交的祖先（不包括开发 的当前提示及其祖先）：

  • 有两个父母的承诺吗？
  • 如果没有，是否至少有一个这样的提交有开发 其（唯一）父级的当前提示？ 
pushedrev=...
basename=develop
if ! baserev="$(git rev-parse --verify refs/heads/"$basename" 2>/dev/null)"; then
    echo "'$basename' is missing, call for help!"
    exit 1
fi
parents_of_commits_beyond_base="$(
  git rev-list --pretty=tformat:%P "$pushedrev" --not "$baserev" |
  grep -v '^commit '
)"
case "$parents_of_commits_beyond_base" in
    *\ *)          echo "must not push merge commits (rebase instead)"
                   exit 1 ;;
    *"$baserev"*)  exit 0 ;;
    *)             echo "must descend from tip of '$basename'"
                   exit 1 ;;
esac

改写

表达该问题的另一种方式是“驻留在当前分支以外的分支上的最近提交是什么，那是哪个分支？”

一个办法

您可以通过一点命令​​行魔法找到它

git show-branch \
| sed "s/].*//" \
| grep "\*" \
| grep -v "$(git rev-parse --abbrev-ref HEAD)" \
| head -n1 \
| sed "s/^.*\[//"

使用AWK：

git show-branch -a \
| grep '\*' \
| grep -v `git rev-parse --abbrev-ref HEAD` \
| head -n1 \
| sed 's/[^\[]*//' \
| awk 'match($0, /\[[a-zA-Z0-9\/.-]+\]/) { print substr( $0, RSTART+1, RLENGTH-2 )}'

它是这样工作的：

  1. 显示所有提交的文本历史记录，包括远程分支。
  2. 当前提交的祖先用星号表示。过滤掉其他所有内容。
  3. 忽略当前分支中的所有提交。
  4. 第一个结果将是最近的祖先分支。忽略其他结果。
  5. 分支名称显示在 [括号中]。忽略括号外的所有内容，以及括号内的内容。
  6. 有时分支名称将包含一个~#或^#来指示引用的提交和分支提示之间有多少提交。我们不在乎。别理他们。

Git 中的“瓷器”一词是什么意思？

Arya

2023-02-18 03:37

“瓷器”一词偶尔会出现在 Git 文档中。这是什么意思？



解答

“瓷器”是通常用来制造马桶（有时还有洗脸盆等其他固定装置）的材料。这与“管道”（实际的管道和排水管）不同，瓷器为管道提供了更加用户友好的界面。

Git 类比地使用这个术语，将用户通常不需要直接使用的低级命令（“管道”）与更用户友好的高级命令（“瓷器”）分开。


更多建议

更重要的是，术语“瓷器”适用于高级命令，输出：

  • 意味着人类可读
  • 不打算被解析
  • 容易受到变化/演变的影响

这是关键：如果您编写脚本，则应尽可能使用具有稳定输出的 管道 命令。不是陶瓷命令。

但是， 您可以使用 porcelain 命令的输出，该命令--porcelain在脚本中有一个选项（见下文），例如：

git status --porcelain
git push --porcelain
git blame --porcelain

虽然 git 包含了它自己的 porcelain 层 ，但是它的底层命令足以支持替代 porcelain 的开发。 这些低级命令的
接口（输入、输出、选项集和语义）比 Porcelain 级命令要稳定得多，因为 这些命令主要用于脚本使用 。另一方面，为了改善最终用户体验，Porcelain 命令的界面可能会发生变化。 * *

请参阅“如何以编程方式确定是否有未提交的更改？ ”作为使用管道命令而不是陶瓷命令的示例。

注意：porcelain 命令可以有一个--porcelain选项。
例如：git status --porcelain，它指定 要解析的输出 。
