ImageMagick 安全策略“PDF”阻止转换

Arya

2023-02-17 18:35

Imagemagick 安全策略似乎不允许我执行从 pdf 到 png 的转换。转换其他扩展名似乎有效，只是不适用于 pdf。自从安装 imagemagick 后，我没有更改任何设置...如果操作系统很重要，我正在使用 Arch Linux。

user@machine $ convert -density 300 -depth 8 -quality 90 input.pdf output.png
convert: attempt to perform an operation not allowed by the security policy `PDF' @ error/constitute.c/IsCoderAuthorized/408.
convert: no images defined `output.png' @ error/convert.c/ConvertImageCommand/3288.



解答

在修复 Ghostscript 后，ImageMagick 的更改得以保留，因为应用程序（尤其是 Web 应用程序）经常将任意用户提供的文件提供给 ImageMagick，并不总是正确地执行格式限制，而且由于 Postscript（PDF 使用）是一个图灵完备的编程在沙箱中运行的语言，沙箱中总是存在另一个漏洞的可能性。

最好保留配置，这样 ImageMagick 拒绝处理需要运行程序的文件，而是在您 有意 允许 Postscript 渲染时直接调用 Ghostscript。

这将通过这样的 Ghostscript 命令来完成：

gs -dSAFER -r600 -sDEVICE=pngalpha -o foo.png myfile.pdf

是的，这是 GhostScript 命令 ImageMagic 调用的变体。（参见 ImageMagick 的delegates.xml.-o是 的简写-dBATCH -dNOPAUSE -sOutputFile=）

重要的是 ImageMagick 保持锁定状态，您不会不必要地调用中间程序，并且您可以更好地控制渲染参数。（例如，-r600渲染的 DPI 是否-sDEVICE=pngalpha允许您直接渲染成您想要的格式）


更多建议

此问题是安全漏洞的解决方法。该漏洞已在 Ghostscript 9.24 中得到解决，因此如果您拥有该版本或更新版本，则不再需要解决方法。

在 Ubuntu 19.04 到 22.04 以及可能带有 ImageMagick 6 的任何更高版本上，以下是通过删除该解决方法来解决问题的方法：

  1. 确保你的 Ghostscript ≥9.24：

    gs --version

  2. 如果是，只需从以下部分中删除整个以下部分/etc/ImageMagick-6/policy.xml：

    <!-- disable ghostscript format types -->

仅删除带有 inside 的行pattern="PDF"就足以重新启用 PDF 转换。不过，我看不出为其他基于 PostScript 的文件类型保留解决方法的充分理由。

悬停一个元素时如何影响其他元素

Arya

2023-02-17 18:35

我想做的是当某个东西div悬停时，它会影响另一个的属性div。

例如，在这个 JSFiddle 演示中，当您将鼠标悬停在#cube它上面时，它会发生变化background- color，但我想要的是当我将鼠标悬停在上面时#container，#cube它会受到影响。

div {
  outline: 1px solid red;
}

#container {
  width: 200px;
  height: 30px;
}

#cube {
  width: 30px;
  height: 100%;
  background-color: red;
}

#cube:hover {
  width: 30px;
  height: 100%;
  background-color: blue;
}


<div id="container">
  <div id="cube">
  </div>
</div>



解答

如果立方体直接在容器内：

#container:hover > #cube { background-color: yellow; }

如果立方体在容器旁边（在容器结束标签之后）：

#container:hover + #cube { background-color: yellow; }

如果立方体在容器内的某处：

#container:hover #cube { background-color: yellow; }

如果多维数据集是容器的兄弟：

#container:hover ~ #cube { background-color: yellow; }


更多建议

在此特定示例中，您可以使用：

#container:hover #cube {
    background-color: yellow;   
}

此示例仅适用于cube是 的子项container。对于更复杂的场景，您需要使用不同的 CSS，或者使用 JavaScript。

C# 接口。隐式实现与显式实现

Arya

2023-02-17 18:34

在 C# 中隐式 和 显式 实现接口有什么区别？

什么时候应该使用隐式，什么时候应该使用显式？

一个或另一个有任何优点和/或缺点吗？

Microsoft 的官方指南（来自第一版Framework Design Guidelines）声明 不推荐使用显式实现 ，因为它会给代码带来意想不到的行为。

我认为这个指南在 IoC 之前的时代 非常有效，那时你不把东西作为接口传递。

任何人也可以谈谈这方面吗？



解答

隐式 是指您通过类中的成员定义接口。 明确的 是当您在接口上的类中定义方法时。我知道这听起来令人困惑，但我的意思是：IList.CopyTo将隐式实现为：

public void CopyTo(Array array, int index)
{
    throw new NotImplementedException();
}

并明确为：

void ICollection.CopyTo(Array array, int index)
{
    throw new NotImplementedException();
}

不同之处在于，隐式实现允许您通过将接口强制转换为该类和接口本身而创建的类来访问接口。显式实现允许您仅通过将接口转换为接口本身来访问接口。

MyClass myClass = new MyClass(); // Declared as concrete class
myclass.CopyTo //invalid with explicit
((IList)myClass).CopyTo //valid with explicit.

我使用 explicit 主要是为了保持实现的简洁，或者当我需要两个实现时。无论如何，我很少使用它。

我确信有更多理由使用/不使用其他人会发布的显式。

请参阅此线程中的 下一篇文章 ，了解每篇文章背后的精彩推理。


更多建议

隐式定义只是将接口所需的方法/属性等作为公共方法直接添加到类中。

显式定义强制成员仅在您直接使用接口而不是底层实现时公开。在大多数情况下这是首选。

  1. 通过直接使用接口，您没有确认并将您的代码耦合到底层实现。
  2. 如果您的代码中已经有一个公共属性 Name，并且您想要实现一个也具有 Name 属性的接口，那么显式地执行此操作将使两者分开。即使他们在做同样的事情，我仍然会将显式调用委托给 Name 属性。您永远不会知道，您可能想要更改 Name 对普通类的工作方式以及 Name（接口属性）稍后的工作方式。
  3. 如果您隐式实现一个接口，那么您的类现在会公开可能只与接口的客户端相关的新行为，这意味着您没有让您的类足够简洁（我的意见）。
