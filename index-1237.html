为什么在 Java 导入语句中使用通配符不好？

Arya

2023-02-18 04:47

使用像这样的单个语句更加方便和干净

import java.awt.*;

而不是导入一堆单独的类

import java.awt.Panel;
import java.awt.Graphics;
import java.awt.Canvas;
...

在语句中使用通配符有什么问题import？



解答

唯一的问题是它会扰乱您的本地命名空间。例如，假设您正在编写一个 Swing 应用程序，因此需要java.awt.Event，并且还与公司的日历系统接口，该系统具有com.mycompany.calendar.Event. 如果您使用通配符方法导入两者，则会发生以下三种情况之一：

  1. java.awt.Event和之间存在彻底的命名冲突com.mycompany.calendar.Event，因此您甚至无法编译。
  2. 你实际上只导入了一个（你的两个导入中只有一个导入了.*），但它是错误的，你很难弄清楚为什么你的代码声称类型是错误的。
  3. 当你编译你的代码时，没有com.mycompany.calendar.Event，但是当他们后来添加一个时，你以前有效的代码突然停止编译。

显式列出所有导入的好处是，我一眼就能看出你打算使用哪个类，这只会让阅读代码变得更加容易。如果您只是做一件快速的一次性事情，则没有明显的 错误 ，但未来的维护者会感谢您的澄清。


更多建议

这里给 明星进口投票。import 语句旨在导入 包 ，而不是类。导入整个包要干净得多；此处确定的问题（例如java.sql.Datevs java.util.Date）很容易通过其他方式解决，并没有 真正 通过特定的进口解决，当然也不能证明对所有类别的疯狂迂腐的进口都是合理的。没有什么比打开源文件不得不翻阅 100 条导入语句更令人不安的了。

进行特定的导入会使重构更加困难；如果你删除/重命名一个类，你需要删除它的 所有 特定导入。如果将实现切换到同一包中的不同类，则必须修复导入。虽然这些额外的步骤可以自动化，但它们确实会影响生产力，并没有真正的收益。

如果默认情况下 Eclipse 不执行特定的类导入，那么每个人仍然会执行 star 导入。抱歉，进行特定进口确实没有合理的理由。

以下是处理类冲突的方法：

import java.sql.*;
import java.util.*;
import java.sql.Date;

如何强制 Git 在 Windows 下使用 LF 而不是 CR+LF？

Arya

LF我想强制 Git 在 Windows 下使用not检出文件CR+LF。
我检查了两个配置选项，但无法找到正确的设置组合。

我想将所有文件转换为具有LF换行符并将其保留LF在文件中。

备注：我使用过autocrlf = input，但这只会在您提交文件时修复文件。
我想强迫它让他们使用LF.

可能我不是很清楚：存储库已经在使用，但是使用Git for WindowsLF签出的文件正在使用，我想强制 Git 使用: forcing Unix line endings 来获取它们。CR+LF``LF ****

$ git config --list | grep crlf
core.autocrlf=input



解答

在 Windows 中获得 LF 结尾的正确方法是首先设置core.autocrlf为false：

git config --global core.autocrlf false

如果您使用的是 msysgit，则需要这样做，因为它true在其系统设置中将其设置为。

现在 git 不会做任何行结束规范化。如果您希望您签入的文件被规范化，请执行以下操作：text=auto在您的.gitattributes所有文件中设置：

* text=auto

并设置core.eol为lf：

git config --global core.eol lf

现在您还可以通过运行将单个存储库切换到 crlf（在工作目录中！）

git config core.eol crlf

完成配置后，您可能希望 git 规范化 repo 中的所有 文件 。为此，请转到存储库的根目录并运行以下命令：

git rm --cached -rf .
git diff --cached --name-only -z | xargs -n 50 -0 git add -f

如果您现在希望 git 也规范化您工作目录 中的文件，请运行以下命令：

git ls-files -z | xargs -0 rm
git checkout .


更多建议

我经常回到这个问题，尽管它的其他答案都不适合我。
也就是说，对我来说正确的答案是其他答案的混合。

我发现有效的是：

git config --global core.eol lf
git config --global core.autocrlf input

对于在设置这些全局设置后检出的回购（Git 存储库），所有内容都将按回购中的任何内容检出——希望是LF( \n)。
任何CRLF将转换为仅LF在签入（提交）时。

使用您已经签出的现有存储库 - 在存储库中具有正确的行结尾但不是您的工作副本 - 您可以运行以下命令来修复它：

git rm -rf --cached .
git reset --hard HEAD

rm这将在没有提示的情况下递归地（）删除-r（） ，从当前目录（）-f删除除您已编辑的文件（）之外的所有文件。然后将所有这些文件返回到它们具有真实行尾的状态（与回购中的内容匹配）。--cached``.``reset

如果您需要修复存储库中文件的行尾，我建议您使用一个编辑器，它可以让您像 IntelliJ 或 Sublime Text 一样批量执行此操作，但我相信任何好的编辑器都可能支持这一点。

在android中以编程方式获取屏幕密度？

如何在android中以编程方式获取屏幕密度？

我的意思是：如何找到当前设备的屏幕 dpi？



解答

您可以从DisplayMetrics结构中获取有关显示的信息：

DisplayMetrics metrics = getResources().getDisplayMetrics();

虽然 Android 不使用直接像素映射，但它使用少量量化的密度独立像素值，然后缩放到实际屏幕尺寸。因此该metrics.densityDpi属性将是常量之一DENSITY_xxx（120、160、213、240、320或dpi）480。640

如果您需要 实际的 lcd 像素密度（可能是 OpenGL 应用程序），您可以分别从水平和垂直密度的metrics.xdpi和metrics.ydpi属性中获取它。

如果您的目标 API 级别早于 4。该metrics.density属性是参考密度 (160dpi) 的浮点缩放因子。metrics.densityDpi可以计算现在提供的相同值

int densityDpi = (int)(metrics.density * 160f);


更多建议

这也有效：

 getResources().getDisplayMetrics().density;
