Arya

2023-02-17 04:38

在查看 tslint 规则的源代码时，我遇到了以下语句：

if (node.parent!.kind === ts.SyntaxKind.ObjectLiteralExpression) {
    return;
}

注意!之后的运算符node.parent。有趣的！

我首先尝试使用我当前安装的 TS (1.5.3) 版本在本地编译文件。由此产生的错误指向爆炸的确切位置：

$ tsc --noImplicitAny memberAccessRule.ts 
noPublicModifierRule.ts(57,24): error TS1005: ')' expected.

接下来我升级到最新的 TS (2.1.6)，编译它没有问题。所以它似乎是 TS 2.x 的特性。 但是 转译完全忽略了爆炸，导致以下 JS：

if (node.parent.kind === ts.SyntaxKind.ObjectLiteralExpression) {
    return;
}

到目前为止，我的 Google fu 让我失望了。

什么是 TS 的感叹号操作符，它是如何工作的？



解答

那是非空断言运算符。这是一种告诉编译器“这个表达式不能是nullor的方法，所以不要抱怨它是orundefined的可能性。” 有时类型检查器无法自己做出决定。null``undefined

TypeScript 发行说明中对此进行了解释：

在类型检查器无法断定该事实的上下文中，可以使用新的!后缀表达式运算符断言其操作数是非空且非未定义的。具体来说，该操作会产生一个with和excludedx!类型的值。与表单的类型断言and类似，非空断言运算符在发出的 JavaScript 代码中被简单地删除。x``null``undefined``<T>x``x as T``!

我发现在该解释中使用术语“断言”有点误导。它是“断言”的意思是 开发人员断言它 ，而不是要执行测试的意思。最后一行确实表明它不会导致发出任何 JavaScript 代码。

  • empty?对象是特定于类的，并且定义因类而异。如果字符串没有字符，则它是空的；如果数组不包含任何项，则它是空的。

Git：从 master 上未暂存/未提交的更改创建一个分支

Arya

2023-02-17 04:37

上下文：我正在研究 master 添加一个简单的功能。几分钟后，我意识到事情并没有那么简单，到一个新的分支机构工作应该会更好。

这总是发生在我身上，我不知道如何切换到另一个分支并在我离开主分支时进行所有这些未提交的更改。我本以为git stash && git stash branch new_branch会简单地做到这一点，但这就是我得到的：

你知道有没有办法做到这一点？



解答

无需藏匿。

2020 年更新/Git 2.23

Git 2.23 添加了新的switch子命令，试图清除一些由于超载使用checkout（切换分支、恢复文件、分离 HEAD 等）而引起的混乱。

从这个版本的 Git 开始，将git checkout下面的命令替换为：

git switch -c <new-branch>

行为保持不变。

更新 2020 / Git 2.23 之前

git checkout -b new_branch_name

不会触及您的本地更改。它只是从当前的 HEAD 创建分支并将 HEAD 设置在那里。所以我想这就是你想要的。

--- 编辑以解释结帐大师的结果 ---

您是否因为checkout master没有丢弃您的更改而感到困惑？

由于更改只是本地的，因此 git 不希望您轻易丢失它们。更改分支后，git 不会覆盖您的本地更改。你的结果checkout master是：

M   testing

，这意味着您的工作文件不干净。git 确实更改了 HEAD，但没有覆盖您的本地文件。这就是为什么你的最后状态仍然显示你的本地更改，尽管你在master.

如果您真的想放弃本地更改，则必须使用-f.

git checkout master -f

由于您的更改从未提交，因此您会丢失它们。

试着回到你的分支，提交你的更改，然后再次检查 master。

git checkout new_branch
git commit -a -m"edited"
git checkout master
git status

您应该M在第一次结帐后收到一条消息，但在 之后不会再收到消息checkout master，并且git status应该不会显示任何修改过的文件。

--- 编辑以消除关于工作目录（本地文件）的混淆---

在回答您的第一条评论时，本地更改只是......好吧，本地的。Git 不会自动保存它们，您必须告诉它保存它们以备后用。如果您进行更改但没有明确提交或存储它们，git 将不会对它们进行版本控制。如果您更改 HEAD ( checkout master)，则本地更改不会因为未保存而被覆盖。
