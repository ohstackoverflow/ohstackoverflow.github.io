如何在迭代时从列表中删除项目？

Arya

2023-02-17 09:15

这个问题的答案是 社区的努力。编辑现有答案以改进这篇文章。它目前不接受新的答案或互动。

我正在 Python 中迭代一个元组列表，如果它们满足特定条件，我将尝试删除它们。

for tup in somelist:
    if determine(tup):
         code_to_remove_tup

我应该用什么代替code_to_remove_tup？我不知道如何以这种方式删除该项目。



解答

您可以使用列表理解来创建一个仅包含您不想删除的元素的新列表：

somelist = [x for x in somelist if not determine(x)]

或者，通过分配给 slice somelist[:]，您可以改变现有列表以仅包含您想要的项目：

somelist[:] = [x for x in somelist if not determine(x)]

如果有其他参考somelist需要反映更改，则此方法可能会有用。

除了理解，您还可以使用itertools. 在 Python 2 中：

from itertools import ifilterfalse
somelist[:] = ifilterfalse(determine, somelist)

或者在 Python 3 中：

from itertools import filterfalse
somelist[:] = filterfalse(determine, somelist)


更多建议

建议列表理解的答案 几乎 是正确的——除了他们构建了一个全新的列表，然后给它与旧列表相同的名称，他们没有 修改 旧列表。这与您通过选择性删除所做的不同，如Lennart 的建议——它更快，但如果您的列表是通过多个引用访问的，那么您只是重新设置其中一个引用而不 更改 列表对象本身的事实可以导致微妙的、灾难性的错误。

幸运的是，获得列表理解的速度和就地更改所需的语义非常容易——只需编写代码：

somelist[:] = [tup for tup in somelist if determine(tup)]

请注意与其他答案的细微差别：这个答案没有 分配 给裸名。它分配给恰好是整个列表的列表切片，从而替换 同一个 Python 列表对象中的列表 内容 ，而不是像其他答案那样只重新设置一个引用（从前一个列表对象到新列表对象）。 ****

“插入忽略”与“插入...重复密钥更新”

Arya

2023-02-17 09:15

在执行INSERT包含多行的语句时，我想跳过可能会导致失败的重复条目。经过一些研究，我的选择似乎是使用：

  • ON DUPLICATE KEY UPDATE这意味着以某种代价进行不必要的更新，或者
  • INSERT IGNORE意味着邀请其他类型的失败在未经通知的情况下溜进来。

我的这些假设是否正确？简单地跳过可能导致重复的行并继续处理其他行的最佳方法是什么？



解答

我会推荐使用INSERT...ON DUPLICATE KEY UPDATE.

如果您使用INSERT IGNORE，那么如果该行导致重复键，则不会实际插入该行。但是该语句不会产生错误。它会生成警告。这些案例包括：

  • PRIMARY KEY在具有或约束的列中插入重复键UNIQUE。
  • 将 NULL 插入到具有NOT NULL约束的列中。
  • 向分区表中插入一行，但您插入的值未映射到分区。

如果使用REPLACE，MySQL 实际上会在内部执行 aDELETE后跟 an INSERT，这会产生一些意想不到的副作用：

  • 分配了一个新的自增 ID。
  • 具有外键的相关行可能会被删除（如果您使用级联外键）或者阻止REPLACE.
  • DELETE不必要地执行触发的触发器。
  • 副作用也会传播到副本。

更正： 和REPLACE都是INSERT...ON DUPLICATE KEY UPDATEMySQL 特有的非标准专有发明。ANSI SQL 2003 定义了一个MERGE语句可以解决同样的需求（以及更多），但是MySQL 不支持该MERGE语句。

用户试图编辑此帖子（编辑被版主拒绝）。该编辑试图添加一个INSERT...ON DUPLICATE KEY UPDATE导致分配新的自动增量 ID 的声明。确实是 生成了 新的id ，但是在变化的行中并没有使用。

请参阅下面的演示，使用 Percona Server 5.5.28 进行测试。配置变量innodb_autoinc_lock_mode=1（默认）：

mysql> create table foo (id serial primary key, u int, unique key (u));
mysql> insert into foo (u) values (10);
mysql> select * from foo;
+----+------+
| id | u    |
+----+------+
|  1 |   10 |
+----+------+

mysql> show create table foo\G
CREATE TABLE `foo` (
  `id` bigint(20) unsigned NOT NULL AUTO_INCREMENT,
  `u` int(11) DEFAULT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `u` (`u`)
) ENGINE=InnoDB AUTO_INCREMENT=2 DEFAULT CHARSET=latin1

mysql> insert into foo (u) values (10) on duplicate key update u = 20;
mysql> select * from foo;
+----+------+
| id | u    |
+----+------+
|  1 |   20 |
+----+------+

mysql> show create table foo\G
CREATE TABLE `foo` (
  `id` bigint(20) unsigned NOT NULL AUTO_INCREMENT,
  `u` int(11) DEFAULT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `u` (`u`)
) ENGINE=InnoDB AUTO_INCREMENT=3 DEFAULT CHARSET=latin1

以上演示了 IODKU 语句检测到重复项，并调用更新来更改u. 注意AUTO_INCREMENT=3指示生成了一个 id，但未在该行中使用。

而REPLACEdoes 删除原始行并插入新行，生成 并 存储一个新的自增 id：

mysql> select * from foo;
+----+------+
| id | u    |
+----+------+
|  1 |   20 |
+----+------+
mysql> replace into foo (u) values (20);
mysql> select * from foo;
+----+------+
| id | u    |
+----+------+
|  3 |   20 |
+----+------+


更多建议

如果您想了解这一切意味着什么，这里是所有内容的逐项说明：

CREATE TABLE `users_partners` (
  `uid` int(11) NOT NULL DEFAULT '0',
  `pid` int(11) NOT NULL DEFAULT '0',
  PRIMARY KEY (`uid`,`pid`),
  KEY `partner_user` (`pid`,`uid`)
) ENGINE=MyISAM DEFAULT CHARSET=utf8

主键基于此快速参考表的两列。主键需要唯一值。

让我们开始：

INSERT INTO users_partners (uid,pid) VALUES (1,1);
...1 row(s) affected

INSERT INTO users_partners (uid,pid) VALUES (1,1);
...Error Code : 1062
...Duplicate entry '1-1' for key 'PRIMARY'

INSERT IGNORE INTO users_partners (uid,pid) VALUES (1,1);
...0 row(s) affected

INSERT INTO users_partners (uid,pid) VALUES (1,1) ON DUPLICATE KEY UPDATE uid=uid
...0 row(s) affected

注意，上面通过将列设置为等于自身节省了太多额外的工作，实际上不需要更新

REPLACE INTO users_partners (uid,pid) VALUES (1,1)
...2 row(s) affected

现在是一些多行测试：

INSERT INTO users_partners (uid,pid) VALUES (1,1),(1,2),(1,3),(1,4)
...Error Code : 1062
...Duplicate entry '1-1' for key 'PRIMARY'

INSERT IGNORE INTO users_partners (uid,pid) VALUES (1,1),(1,2),(1,3),(1,4)
...3 row(s) affected

控制台中没有生成其他消息，现在表数据中有这 4 个值。我删除了除 (1,1) 之外的所有内容，这样我就可以在同一个运动场上进行测试

INSERT INTO users_partners (uid,pid) VALUES (1,1),(1,2),(1,3),(1,4) ON DUPLICATE KEY UPDATE uid=uid
...3 row(s) affected

REPLACE INTO users_partners (uid,pid) VALUES (1,1),(1,2),(1,3),(1,4)
...5 row(s) affected

所以你有它。由于这一切都是在几乎没有数据且不在生产中的新表上执行的，因此执行时间是微观的且无关紧要的。任何拥有真实世界数据的人都非常欢迎贡献它。
