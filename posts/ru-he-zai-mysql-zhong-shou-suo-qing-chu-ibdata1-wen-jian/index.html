如何在 MySQL 中收缩/清除 ibdata1 文件

Arya

2023-02-18 02:53

我在localhost中使用MySQL作为在R中进行统计的“查询工具”，也就是说，每次运行R脚本时，我都会创建一个新数据库（A），创建一个新表（B），将数据导入B ，提交一个查询得到我需要的，然后我丢B丢A。

它对我来说工作正常，但我意识到 ibdata 文件的大小正在迅速增加，我没有在 MySQL 中存储任何内容，但 ibdata1 文件已经超过 100 MB。

我正在使用或多或少的默认 MySQL 设置进行设置，有没有一种方法可以在一段固定的时间后自动收缩/清除 ibdata1 文件？



解答

不ibdata1收缩是 MySQL 的一个特别烦人的特性。除非删除所有数据库、删除文件并重新加载转储，否则文件ibdata1实际上无法缩小。

但是您可以配置 MySQL，以便将每个表（包括其索引）存储为一个单独的文件。那样ibdata1就不会长那么大了。根据Bill Karwin 的评论，从 MySQL 5.6.6 版本开始默认启用。

不久前我做了这个。但是，要将您的服务器设置为对每个表使用单独的文件，您需要更改my.cnf以启用此功能：

[mysqld]
innodb_file_per_table=1

https://dev.mysql.com/doc/refman/5.6/en/innodb-file-per-table-tablespaces.html

因为你想回收ibdata1你实际上必须删除文件的空间：

  1. 对除 和 数据库mysqldump之外的所有数据库、过程、触发器等执行操作 mysql``performance_schema
  2. 删除除上述 2 个数据库之外的 所有数据库 ****
  3. 停止mysql
  4. 删除ibdata1和ib_log文件
  5. 启动mysql
  6. 从转储中恢复

当您在第 5 步中启动 MySQL 时，将重新创建ibdata1和文件。ib_log

现在你可以走了。当您创建一个新的数据库进行分析时，表将位于单独的ibd*文件中，而不是在ibdata1. 由于您通常很快就会删除数据库，因此ibd*文件将被删除。

http://dev.mysql.com/doc/refman/5.1/en/drop-database.html

你可能已经看到了这个：
http ://bugs.mysql.com/bug.php?id=1341

使用命令ALTER TABLE <tablename> ENGINE=innodborOPTIMIZE TABLE <tablename>可以从 ibdata1 中提取数据和索引页以分离文件。但是，除非您执行上述步骤，否则 ibdata1 不会缩小。

关于information_schema，没必要也不可能掉。它实际上只是一堆只读视图，而不是表。并且没有与它们关联的文件，甚至没有数据库目录。正在informations_schema使用内存数据库引擎，并在停止/重新启动 mysqld 时被删除并重新生成。请参阅https://dev.mysql.com/doc/refman/5.7/en/information- schema.html。

对于 linux 系统，可以使用以下命令完成步骤 1-6：

  1. mysqldump -u [username] -p[root_password] [database_name] > dumpfilename.sql
  2. mysqladmin -u [username] -p[root_password] drop [database_name]
  3. sudo /etc/init.d/mysqld stop

  4. sudo rm /var/lib/mysql/ibdata1
    sudo rm /var/lib/mysql/ib_logfile*

  5. sudo /etc/init.d/mysqld start

  6. mysqladmin -u [username] -p[root_password] create [database_name]
  7. mysql -u [username] -p[root_password] [database_name] < dumpfilename.sql

警告：如果您在此 mysql 实例上有其他数据库，这些说明将导致您丢失其他数据库。 确保修改步骤 1、2 和 6、7 以涵盖您希望保留的所有数据库。
