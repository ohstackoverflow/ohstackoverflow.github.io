原始答案（2010 年 1 月）

A会更好，如Charles Bailey 在此处所述rebase --onto，您可以在集成分支之上重播给定范围的提交。（另外，在git rebase 手册页中查找“这里是如何将基于一个分支的主题分支移植到另一个分支” ，以查看实际示例）
git rebase --onto

如果您当前的分支是集成：

# Checkout a new temporary branch at the current location
git checkout -b tmp

# Move the integration branch to the head of the new patchset
git branch -f integration last_SHA-1_of_working_branch_range

# Rebase the patchset onto tmp, the old location of integration
git rebase --onto tmp first_SHA-1_of_working_branch_range~1 integration

这将重播之间的所有内容：

  • first_SHA-1_of_working_branch_range在（因此是）的父级之后~1：您要重播的第一个提交
  • 直到“ integration”（指向您要从working分支重播的最后一次提交）

到“ tmp”（指向integration之前指向的位置）

如果重放其中一个提交时有任何冲突：

  • 解决它并运行“ git rebase --continue”。
  • 或者跳过这个补丁，而是运行“ git rebase --skip”
  • 或者用“”取消所有东西git rebase --abort（并将分支放回分支integration上tmp）

之后rebase --onto，integration将回到集成分支的最后一次提交（即“ tmp”分支+所有重播提交）

使用 cherry-picking 或rebase --onto，不要忘记它会对后续合并产生影响，如此处所述。

这里讨论了一个纯粹的“ cherry-pick”解决方案，并且涉及如下内容：

如果你想使用补丁方法，那么“git format-patch|git am”和“git cherry”是你的选择。
目前，git cherry-pick只接受一次提交，但如果你想选择范围，B那D将是B^..Dgit 术语，所以

git rev-list --reverse --topo-order B^..D | while read rev 
do 
  git cherry-pick $rev || break 
done

但无论如何，当您需要“重播”一系列提交时，“重播”一词应该促使您使用rebaseGit 的“”功能。


更多建议

从 git v1.7.2 开始，cherry pick 可以接受一系列提交：

git cherry-pick学会了选择一系列提交（例如cherry-pick A..B和cherry-pick --stdin），也是如此git revert；rebase [-i]但是，这些不支持更好的排序控制。

正如Gabe Moothart指出的那样，cherry-pick A..B不会得到提交A（你需A~1..B要这样做），如果有任何冲突，git 不会像 rebase 那样自动继续（至少从 1.7.3.1 开始）。
