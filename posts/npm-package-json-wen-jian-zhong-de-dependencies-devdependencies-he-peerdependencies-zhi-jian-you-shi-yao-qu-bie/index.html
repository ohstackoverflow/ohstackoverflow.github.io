npm package.json 文件中的 dependencies、devDependencies 和 peerDependencies 之间有什么区别？

EDIT 还添加了peerDependencies，这是密切相关的，可能会引起混淆。



解答

重要行为差异总结：

  • dependencies都安装在：

    • npm install从包含的目录package.json
    • npm install $package在任何其他目录上

    • devDependencies是：

    • 也安装npm install在包含的目录中package.json，除非您传递--production标志（去投票Gayan Charith 的回答），或者如果NODE_ENV=production设置了环境变量

    • 没有安装在npm install "$package"任何其他目录上，除非你给它选项--dev。
    • 没有被传递安装。

    • peerDependencies:

    • 3.0 之前：如果缺少则始终安装，如果不同的依赖项使用多个不兼容版本的依赖项，则会引发错误。

    • expected to start on 3.0 (untested): 如果缺少 on 给出警告npm install，你必须自己手动解决依赖。运行时，如果缺少依赖项，则会出现错误（@nextgentech提到）这很好地解释了它：https ://flaviocopes.com/npm-peer-dependencies/
    • 在版本 7 中， peerDependencies 会自动安装，除非存在无法自动解决的上游依赖冲突

    • 传递性（由Ben Hutchison提到）：

    • dependencies可传递安装：如果 A 需要 B，B 需要 C，则安装 C，否则，B 无法工作，A 也无法工作。

    • devDependencies没有被传递安装。例如，我们不需要测试 B 来测试 A，因此可以忽略 B 的测试依赖项。

  • optionalDependencies（艾丹费尔德曼提到）

开发依赖

dependencies需要运行，devDependencies只需要开发，例如：单元测试，CoffeeScript 到 JavaScript 的转换，缩小，......

如果您要开发一个包，您可以下载它（例如通过git clone），转到它包含 的根目录package.json，然后运行：

npm install

因为你有实际的源代码，很明显你想开发它，所以默认情况下，dependencies（因为你必须，当然，运行开发）和devDependency依赖项也被安装。

但是，如果您只是一个最终用户，只想安装一个包来使用它，您可以从任何目录执行：

npm install "$package"

在这种情况下，您通常不需要开发依赖项，因此您只需获取使用包所需的内容：dependencies。

如果你真的想在这种情况下安装开发包，你可以将dev配置选项设置为true，可以从命令行设置为：

npm install "$package" --dev

该选项是false默认的，因为这是一种不太常见的情况。

对等依赖

(3.0之前测试过)

资料来源： https: //nodejs.org/en/blog/npm/peer-dependencies/

对于常规依赖项，您可以拥有多个版本的依赖项：它只是安装在node_modules依赖项的内部。

例如，如果dependency1和dependency2都依赖于dependency3不同的版本，项目树将如下所示：

root/node_modules/
                 |
                 +- dependency1/node_modules/
                 |                          |
                 |                          +- dependency3 v1.0/
                 |
                 |
                 +- dependency2/node_modules/
                                            |
                                            +- dependency3 v2.0/

然而，插件是通常不需要其他包的包，在此上下文中称为 主机。 反而：

  • 主机 需要插件 __
  • 插件提供主机期望找到的标准接口
  • 只有宿主会被用户直接调用，所以必须有一个单一的版本。

例如，如果dependency1and dependency2peer depend on dependency3，项目树将如下所示：

root/node_modules/
                 |
                 +- dependency1/
                 |
                 +- dependency2/
                 |
                 +- dependency3 v1.0/

即使您从未在文件dependency3中提及，也会发生这种情况package.json。

我认为这是控制反转设计模式的一个实例。

对等依赖的典型示例是 Grunt、主机及其插件。

例如，在像https://github.com/gruntjs/grunt-contrib-uglify这样的 Grunt 插件上，你会看到：

  • grunt是一个peer-dependency
  • 唯一require('grunt')的是tests/：它实际上没有被程序使用。

Gruntfile然后，当用户要使用插件时，他会通过添加一行隐式地要求插件grunt.loadNpmTasks('grunt-contrib- uglify')，但grunt用户将直接调用。

如果每个插件都需要不同的 Grunt 版本，那么这将不起作用。
