Git 处理 CRLF（回车、换行）的策略是什么？

Arya

2023-02-17 19:15

我尝试提交带有 CRLF 结尾行的文件，但失败了。

我在 Windows 计算机上花了一整天的时间尝试不同的策略，几乎被吸引而停止尝试使用 Git，而是尝试Mercurial。

如何正确处理 CRLF 行结尾？



解答

问了这个问题快四年了，我终于找到了 一个让我完全满意的答案 ！

请参阅github:help 的处理行尾指南中的详细信息。

Git 允许您直接使用文件中的文本属性为存储库设置行结束属性 .gitattributes 。此文件已提交到存储库并覆盖该core.autocrlf设置，使您能够确保所有用户的行为一致，无论他们的 git 设置如何。

因此

这样做的好处是您的行尾配置现在与您的存储库一起移动，您无需担心协作者是否具有正确的全局设置。

.gitattributes 这是一个文件的例子

# Auto detect text files and perform LF normalization
*        text=auto

*.cs     text diff=csharp
*.java   text diff=java
*.html   text diff=html
*.css    text
*.js     text
*.sql    text

*.csproj text merge=union
*.sln    text merge=union eol=crlf

*.docx   diff=astextplain
*.DOCX   diff=astextplain

# absolute paths are ok, as are globs
/**/postinst* text eol=lf

# paths that don't start with / are treated relative to the .gitattributes folder
relative/path/*.txt text eol=lf

对于最流行的编程语言，有一个方便的即用型 .gitattributes 文件集合。帮助您入门很有用。

创建或调整您的 后 .gitattributes ，您应该执行一劳永逸的行尾重新规范化。

请注意，当您在应用程序中打开项目的 Git 存储库后，GitHub Desktop应用程序可以建议并创建文件。 .gitattributes 要尝试这样做，请单击齿轮图标（在右上角）> 存储库设置 ... > 行尾和属性。系统将要求您添加推荐 .gitattributes ，如果您同意，该应用程序还将对您存储库中的所有文件执行规范化。

最后，Mind the End of Your Line文章提供了更多背景知识，并解释了 Git 如何处理手头的事情。我认为这 是必读的 。

您的团队中可能有一些用户使用 EGit 或 JGit（Eclipse 和 TeamCity 等工具使用它们）来提交他们的更改。那么你就不走运了，正如@gatinueta 在这个答案的评论中解释的那样：

如果您的团队中有人使用 Egit 或 JGit，此设置将不会完全满足您的要求，因为这些工具只会忽略 .gitattributes 并愉快地签入 CRLF 文件https://bugs.eclipse.org/bugs/show_bug.cgi? id=342372

一个技巧可能是让他们在另一个客户端中提交更改，比如SourceTree。我们的团队当时在许多用例中更喜欢该工具而不是 Eclipse 的 EGit。

谁说软件简单？:-/


更多建议

不要转换行尾。解释数据不是 VCS 的工作——只是对其进行存储和版本控制。无论如何，每个现代文本编辑器都可以阅读这两种行尾。
