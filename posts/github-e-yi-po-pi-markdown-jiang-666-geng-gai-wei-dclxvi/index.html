GitHub 恶意破坏 Markdown - 将 666 更改为 DCLXVI

Arya

2023-02-17 14:04

我的 GitHub 存储库中只有一个自述文件。在这个自述文件中，我在本地写了这个：

Factoids:
 - There are about six different ways to do everything in Forked.
 - There are actually six different ways to enter loops.
 - There are six directionals and six I/O commands.
 - 666. ha.

强调最后一行。GitHub 决定展示的 不是 666.

DCLXVI是666的罗马数字。

这真让我毛骨悚然。我的本地文件和原始文件都显示666.

GitHub 在做什么，为什么未编号列表上的缩进乱七八糟？这是复活节彩蛋，还是某种邪恶的错误？



解答

这似乎是github/markup issue 991之后的内容，在有序的子列表中，十进制数字自动变成罗马数字。

我找到了问题的原因。这是CSS

这是嵌套有序列表在 HTML 中呈现的预期方式。

这在 HTML 中不是预期的。https://jsfiddle.net/tf5jtv8s

我们不会对默认的 HTML 行为进行任何修改。

ol ol,ul ol{list-style-type:lower-roman}

我不知道 CSS，但我的理解是这是问题的原因。我可以通过禁用 CSS 获得预期的结果。（我来自我的手机，所以我不能使用浏览器检查器）

正如“ GitHub Flavored Markdown 的正式规范”中所述，GitHub Markdown 规范GFM：GitHub Flavored Markdown 规范建立在CommonMark 规范之上。

正如Tommi Kaikkonen在他的回答中提到的那样，有序列表是因为 666 之后的点。请参阅GFM 规范第 5.2 节。

如第 6.1 节所述，任何 ASCII 标点符号都可以反斜杠转义，以避免出现此问题。
这意味着：

- 666\. ha.

（如ForNeVeR的回答中明确所示）

这就是为什么666在 GitHub markdown 中将该数字更改为罗马数字的原因README。

迈克·利珀特评论道：

该列表中的第一个元素，因此它应该显示为inot dclxvi。
Markdown 有序列表忽略了实际使用的数字和顺序数字，我还没有找到改变它的方法。

然而，没有：它显示dclxvi，因为生成的 html 代码是，这与GFM 规范<ol start="666">一致：

如果列表项是有序的，那么它也会根据有序列表标记分配一个起始编号”

（此处，' 666' 是有序列表标记）

迈克补充说：

@VonC 对于其他人，这里是 VonC 文档链接的另一个有用摘录：

“有序列表的起始编号由其初始列表项的列表编号确定。后续列表项的编号将被忽略。”

另外，为什么间距搞砸了？我在你的回答中没听清楚

    <ol start="666">
      <li>ha.</li>
    </ol>

GitHub CSS 规则包括：

.markdown-body ol {
    padding-left: 2em;
}

如果你放3em，你会得到而不是


更多建议

在其后添加句点666使其成为有序列表标记。

GitHub 声明了使用罗马数字呈现有序列表标记的 CSS：

ol ol,ul ol {
    list-style-type: lower-roman
}

使用反斜杠转义句点，您应该会看到正确的输出。
