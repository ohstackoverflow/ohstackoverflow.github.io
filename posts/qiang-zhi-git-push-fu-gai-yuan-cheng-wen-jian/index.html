强制“git push”覆盖远程文件

Arya

2023-02-17 06:43

我想推送我的本地文件，并将它们放在远程仓库中，而不必处理合并冲突。我只希望我的本地版本优先于远程版本。

我怎样才能用 Git 做到这一点？



解答

您应该能够通过使用强制对远程仓库进行本地修订

git push -f <remote> <branch>

（例如git push -f origin master）。离开<remote>并将<branch>强制推送所有已设置的本地分支--set- upstream。

请注意，如果其他人正在共享此存储库，他们的修订历史将与新的冲突。如果他们在更改点之后有任何本地提交，他们将变得无效。

更新 ：我想我会添加一个旁注。如果您正在创建其他人将审查的更改，那么创建一个包含这些更改的分支并定期变基以使它们与主要开发分支保持同步的情况并不少见。让其他开发人员知道这会定期发生，这样他们就会知道会发生什么。

更新 2 ：由于越来越多的观众，我想添加一些额外的信息，说明当您upstream确实遇到强制推送时该怎么做。

假设我已经克隆了你的 repo 并添加了一些像这样的提交：

            D----E话题
           /
A----B----C开发

但是后来development分支被击中了rebase，这将导致我在运行时收到这样的错误git pull：

拆包对象：100% (3/3)，完成。
来自 <repo-location>
 * 分支开发 -> FETCH_HEAD
自动合并 <files>
CONFLICT（内容）：合并 <locations> 中的冲突
自动合并失败；修复冲突然后提交结果。

在这里我可以修复冲突和commit，但是那会让我留下一个非常丑陋的提交历史：

       C----D----E----F话题
      / /
A----B----------------C'发展

它可能看起来很诱人，git pull --force但要小心，因为这会让你陷入困境：

            D----E话题

A----B----C'发展

所以最好的选择可能是做一个git pull --rebase. 这将要求我像以前一样解决任何冲突，但对于每个步骤，我将使用git rebase --continue. 最后提交历史看起来会好得多：

            D'---E'话题
           /
A----B----C'发展

更新 3： 您也可以将该--force-with-lease选项用作“更安全”的强制推送，正如 Cupcake 在他的回答中提到的那样：

如果远程上有您不期望的新提交（从技术上讲，如果您还没有将它们提取到您的远程跟踪分支），使用“租约”强制推送允许强制推送失败，如果你不想不小心覆盖别人的你甚至不知道的提交，你只想覆盖你自己的：

git push <remote> <branch> --force-with-lease

更多建议

你想强推

您基本上想要做的是强制推送您的本地分支，以覆盖远程分支。

如果您想对以下每个命令进行更详细的解释，请参阅下面我的详细信息部分。使用 Git 强制推送基本上有 4 种不同的选择：

git push <remote> <branch> -f
git push origin master -f # Example

git push <remote> -f
git push origin -f # Example

git push -f

git push <remote> <branch> --force-with-lease

如果您想对每个命令进行更详细的解释，请参阅下面我的长答案部分。

警告： 强制推送将用您正在推送的分支的状态覆盖远程分支。在使用它之前确保这是您真正想要做的，否则您可能会覆盖您实际想要保留的提交。

强制推送细节

指定远程和分支

您可以完全指定特定的分支和远程。旗帜-f是短版的--force

git push <remote> <branch> --force
git push <remote> <branch> -f

省略分支

当 push branch 的分支被省略时，Git 会根据你的配置设置来计算它。在 2.0 之后的 Git 版本中，新的 repo 将具有默认设置来推送当前签出的分支：

git push <remote> --force

而在 2.0 之前，新的 repos 将具有默认设置以推送多个本地分支。有问题的设置是remote.<remote>.push和push.default设置（见下文）。

省略远程和分支

当 remote 和 branch 都被省略时，just 的行为由您的Git 配置设置git push --force决定：push.default

git push --force

  • 从 Git 2.0 开始，默认设置simple基本上只是将当前分支推送到其上游远程对应部分。远程由分支的branch.<remote>.remote设置决定，否则默认为原始仓库。

  • 在 Git 2.0 版之前，默认设置matching基本上只是将所有本地分支推送到远程上具有相同名称的分支（默认为 origin）。

push.default您可以通过阅读git help config或在线版本的 git-config(1) Manual Page来了解更多设置。

强制推动更安全--force-with-lease

如果远程上有您不期望的新提交（从技术上讲，如果您还没有将它们提取到您的远程跟踪分支），使用“租约”强制推送允许强制推送失败，如果你不想不小心覆盖别人的你甚至不知道的提交，你只想覆盖你自己的：

git push <remote> <branch> --force-with-lease
