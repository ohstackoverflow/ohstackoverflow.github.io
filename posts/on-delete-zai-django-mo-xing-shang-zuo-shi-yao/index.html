在 Django 1.9 中更改：

on_delete现在可以用作第二个位置参数（以前它通常只作为关键字参数传递）。在 Django 2.0 中它将是一个必需的参数。

使用示例是：

from django.db import models

class Car(models.Model):
    manufacturer = models.ForeignKey(

解答

这是删除 引用 对象时要采用的行为。它不是 Django 特有的；这是一个 SQL 标准。尽管 Django 在 SQL 之上有自己的实现。 (1)

发生此类事件时有七种可能的操作：

  • CASCADE：删除引用的对象时，也会删除引用它的对象（例如，当您删除博客文章时，您可能还想删除评论）。SQL 等价物：CASCADE.
  • PROTECT：禁止删除引用对象。要删除它，您必须手动删除所有引用它的对象。SQL 等价物：RESTRICT.
  • RESTRICT: （在 Django 3.1 中引入）PROTECT与SQL 更准确匹配的类似行为RESTRICT。（参见django 文档示例）
  • SET_NULL：将引用设置为 NULL（要求字段可为空）。例如，当您删除一个用户时，您可能希望保留他在博客文章中发表的评论，但说它是由匿名（或已删除）用户发表的。SQL 等价物：SET NULL.
  • SET_DEFAULT：设置默认值。SQL 等价物：SET DEFAULT.
  • SET(...)：设置给定值。这不是 SQL 标准的一部分，完全由 Django 处理。
  • DO_NOTHING：可能是一个非常糟糕的主意，因为这会在您的数据库中产生完整性问题（引用一个实际上不存在的对象）。SQL 等价物：NO ACTION. (2)

资料来源：Django 文档

例如，另请参阅PostgreSQL 的文档。

在大多数情况下，CASCADE这是预期的行为，但对于每个 ForeignKey，您应该始终问自己在这种情况下的预期行为是什么。PROTECT并且SET_NULL经常有用。设置CASCADE它不应该的地方，可能会通过简单地删除单个用户来级联删除所有数据库。

澄清级联方向的附加说明

有趣的是，CASCADE许多人并不清楚行动的方向。实际上，有趣的是注意到 只有 动作CASCADE不清楚。我知道级联行为可能令人困惑，但是您必须认为 它与任何其他操作的方向相同 。因此，如果你觉得CASCADE方向不明确，其实就是on_delete行为不明确。

在您的数据库中，外键基本上由一个整数字段表示，该字段的值是外部对象的主键。假设您有一个条目 comment_A ，它有一个条目 article_B 的外键。如果删除条目 comment_A ，一切都很好。 article_B曾经没有 comment_A 生活，如果它被删除也不要打扰。但是，如果您删除 article_B ，那么 comment_A 就会崩溃！它从来没有没有 article_B 而需要它，它是它的属性的一部分（article=article_B，但是 article_B 是什么？？？）。这是on_delete确定如何解决此问题的步骤 完整性错误 ，或者说：

  • “不！拜托！不要！我不能没有你！” （在 Django/SQL 中说PROTECTor ）RESTRICT
  • “好吧，如果我不是你的，那我就不是任何人的” （据说SET_NULL）
  • “再见世界，我不能没有article_B” 然后自杀（这是CASCADE行为）。
  • “没关系，我有多余的情人，我会从现在开始参考 article_C” （SET_DEFAULT，甚至SET(...)）。
  • “我无法面对现实，我会一直呼唤你的名字，即使那是我唯一能做的！” ( DO_NOTHING)

我希望它能使级联方向更清晰。:)

脚注

(1) Django在SQL之上有自己的实现。而且，正如@JoeMjr2 在下面的评论中提到的，Django 不会创建 SQL 约束。如果您希望约束由您的数据库来确保（例如，如果您的数据库被另一个应用程序使用，或者如果您不时地在数据库控制台中挂起），您可能希望自己手动设置相关约束。在 Django 中添加对数据库级删除约束的支持是公开的。

(2) 实际上，有一种情况是DO_NOTHING有用的：如果你想跳过 Django 的实现，自己在数据库级别实现约束。
