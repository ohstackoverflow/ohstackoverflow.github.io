设置超时还是设置间隔？

Arya

2023-02-17 13:04

据我所知，这两个 javascript 的行为方式相同：

选项A：

function myTimeoutFunction()
{
    doStuff();
    setTimeout(myTimeoutFunction, 1000);
}

myTimeoutFunction();

选项B：

function myTimeoutFunction()
{
    doStuff();
}

myTimeoutFunction();
setInterval(myTimeoutFunction, 1000);

使用setTimeout和setInterval有什么区别吗？



解答

他们本质上尝试做同样的事情，但该setInterval方法将比该setTimeout方法更准确，因为setTimeout等待 1000 毫秒，运行函数然后设置另一个超时。所以等待时间实际上超过 1000 毫秒（如果您的函数需要很长时间才能执行，则等待时间会更长）。

尽管有人可能认为它会 恰好setInterval每 1000 毫秒执行一次，但重要的是要注意它也会延迟，因为 JavaScript 不是一种多线程语言，这意味着 - 如果脚本的其他部分正在运行 - 间隔将有等待它完成。 ****setInterval

在这个 Fiddle中，您可以清楚地看到超时会落后，而间隔几乎一直在几乎 1 次调用/秒（脚本正在尝试这样做）。如果将顶部的速度变量更改为 20 之类的小值（意味着它将尝试每秒运行 50 次），则间隔永远不会达到平均每秒 50 次迭代。

延迟几乎总是可以忽略不计，但如果你正在编程一些非常精确的东西，你应该选择一个 自调整定时器 （它本质上是一个基于超时的定时器，它会根据它创建的延迟不断调整自己）

有什么不同吗？

是的。Timeout 在调用 setTimeout() 后执行一定时间；Interval 在前一个间隔触发后执行一定时间。

如果您的 doStuff() 函数需要一段时间才能执行，您会注意到差异。例如，如果我们用 表示对 setTimeout/setInterval 的调用.，用 触发超时/间隔，用*执行 JavaScript 代码[-----]，时间线如下所示：

下一个复杂情况是，如果在 JavaScript 已经忙于做某事（例如处理前一个间隔）时触发了一个间隔。在这种情况下，间隔会被记住，并在前一个处理程序完成并将控制权返回给浏览器时立即发生。例如，对于有时短 ([-]) 有时长 ([-----]) 的 doStuff() 过程：

.    *    *    •    *    •    *    *
     [-]  [-----][-][-----][-][-]  [-]

• 表示无法立即执行其代码的间隔触发，而是被挂起。

因此，间隔尝试“赶上”以回到计划。但是，它们不会一个接一个地排队：每个时间间隔只能有一个等待执行。（如果他们都排队，浏览器将留下一个不断扩大的未完成执行列表！）

.    *    •    •    x    •    •    x
     [------][------][------][------]

x 表示无法执行或被挂起的间隔触发，因此被丢弃。

如果您的 doStuff() 函数习惯性地花费比为其设置的时间间隔更长的时间来执行，浏览器将占用 100% 的 CPU 来尝试为其提供服务，并且响应速度可能会变慢。

你用哪个，为什么？

Chained-Timeout 为浏览器提供了有保证的空闲时间；Interval 尝试确保它正在运行的功能尽可能接近其预定时间执行，但以牺牲浏览器 UI 可用性为代价。

我会考虑一次性动画的间隔，我希望它尽可能流畅，而链式超时对于页面加载时一直发生的持续动画更为礼貌。对于要求不高的用途（例如每 30 秒触发一次的普通更新程序或其他），您可以安全地使用其中任何一个。

在浏览器兼容性方面，setTimeout 早于 setInterval，但您今天遇到的所有浏览器都支持这两者。多年来最后一个掉队的是 WinMo <6.5 中的 IE Mobile，但希望它现在也已经过去了。
