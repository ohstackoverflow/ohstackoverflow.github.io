MongoDB 关系：嵌入还是引用？

Arya

2023-02-17 22:34

我想设计一个带有一些评论的问题结构。我应该使用哪种关系进行评论：embed或reference？

带有一些评论的问题，如stackoverflow，将具有如下结构：

Question
    title = 'aaa'
    content = 'bbb'
    comments = ???

起初，我想到了使用嵌入式注释（我认为embed在 MongoDB 中是推荐的），像这样：

Question
    title = 'aaa'
    content = 'bbb'
    comments = [ { content = 'xxx', createdAt = 'yyy'}, 
                 { content = 'xxx', createdAt = 'yyy'}, 
                 { content = 'xxx', createdAt = 'yyy'} ]

很明显，但我担心这种情况： 如果我想编辑指定的评论，我如何获得它的内容和它的问题？ 没有_id让我找到一个，也没有question_ref让我找到它的问题。（有没有办法在没有_idand 的情况下做到这一点question_ref？）

我必须使用ref而不是吗embed？然后我必须创建一个新的评论集合吗？



解答

这与其说是一门科学，不如说是一门艺术。Mongo Documentation on Schemas是一个很好的参考，但这里有一些需要考虑的事项：

  • 尽可能多地投入

文档数据库的乐趣在于它消除了很多连接。您的第一直觉应该是尽可能多地放置在一个文档中。因为 MongoDB 文档具有结构，并且因为您可以在该结构内进行高效查询（这意味着您可以获取所需的文档部分，所以文档大小不应该让您太担心），因此不需要立即规范化数据，例如你会在 SQL 中。特别是，除了其父文档之外没有用处的任何数据都应该是同一文档的一部分。

  • 将可以从多个地方引用的数据单独放入自己的集合中。

这与其说是“存储空间”问题，不如说是“数据一致性”问题。如果许多记录将引用相同的数据，那么更新单个记录并在其他地方保留对它的引用会更有效且更不容易出错。

  • 文档大小注意事项

MongoDB 对单个文档施加了 4MB（16MB 和 1.8）的大小限制。在 GB 的数据世界中，这听起来很小，但它也是 3 万条推文或 250 个典型的 Stack Overflow 答案或 20 张闪烁的照片。另一方面，这比人们希望一次在典型网页上显示的信息要多得多。首先考虑什么会使您的查询更容易。在许多情况下，对文档大小的关注将是过早的优化。

  • 复杂的数据结构：

MongoDB 可以存储任意深度嵌套的数据结构，但不能有效地搜索它们。如果您的数据形成一棵树、森林或图形，您实际上需要将每个节点及其边存储在单独的文档中。（请注意，还应考虑专门为此类数据设计的数据存储）

还指出不可能返回文档中元素的子集。如果您需要从每个文档中挑选一些内容，将它们分开会更容易。

  • 数据一致性

MongoDB 在效率和一致性之间做出权衡。规则是对单个文档的更改 始终是 原子的，而对多个文档的更新永远不应假定为原子的。也没有办法“锁定”服务器上的记录（您可以使用例如“锁定”字段将其构建到客户端的逻辑中）。在设计架构时，请考虑如何保持数据的一致性。通常，您在文档中保存的内容越多越好。

对于您所描述的内容，我会嵌入评论，并为每个评论提供一个带有 ObjectID 的 id 字段。ObjectID 中嵌入了一个时间戳，因此您可以根据需要使用它而不是创建时间。


更多建议

一般来说，如果实体之间存在一对一或一对多关系，则嵌入比较好，如果实体之间存在多对多关系，则引用比较好。
