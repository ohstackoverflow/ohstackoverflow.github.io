*.h 或 *.hpp 用于您的类定义

Arya

2023-02-17 16:08

我一直使用一个*.h文件作为我的类定义，但在阅读了一些 boost 库代码后，我意识到它们都使用*.hpp. 我一直对那个文件扩展名很反感，我想主要是因为我不习惯。

*.hpp使用over的优点和缺点是什么*.h？



解答

以下是 C 与 C++ 标头具有不同命名的几个原因：

  • 自动代码格式化，您可能对格式化 C 和 C++ 代码有不同的准则。如果标题由扩展名分隔，您可以将编辑器设置为自动应用适当的格式
  • 命名，我参与过一些项目，其中有用 C 编写的库，然后用 C++ 实现了包装器。由于标头通常具有相似的名称，即 Feature.h 与 Feature.hpp，因此很容易区分它们。
  • 包容，也许你的项目有更合适的可用 C++ 编写的版本，但你使用的是 C 版本（见上一点）。如果标头以实现它们的语言命名，您可以轻松发现所有 C 标头并检查 C++ 版本。

请记住，C 不是 C ++，除非您知道自己在做什么，否则混合搭配是非常危险的。适当地命名您的来源可以帮助您区分这些语言。


更多建议

我使用 .hpp 是因为我希望用户区分哪些标头是 C++ 标头，哪些标头是 C 标头。

当您的项目同时使用 C 和 C++ 模块时，这可能很重要：就像其他人在我之前解释的那样，您应该非常小心地做，并且它从您通过扩展提供的“合同”开始

.hpp ：C++ 头文件

（或 .hxx，或 .hh，或其他）

此标头仅适用于 C++。

如果您在 C 模块中，甚至不要尝试包含它。你不会喜欢它，因为没有努力使它对 C 语言友好（太多会丢失，如函数重载、名称空间等）。

.h : C/C++ 兼容或纯 C 头文件

C 源代码和 C++ 源代码可以直接或间接包含此标头。

它可以直接包含，受__cplusplus宏保护：

  • 这意味着，从 C++ 的角度来看，C 兼容代码将被定义为extern "C".
  • 从 C 的角度来看，所有 C 代码都是清晰可见的，但 C++ 代码将被隐藏（因为它不会在 C 编译器中编译）。

例如：

#ifndef MY_HEADER_H
#define MY_HEADER_H

   #ifdef __cplusplus
      extern "C"
      {
   #endif

   void myCFunction() ;

   #ifdef __cplusplus
      } // extern "C"
   #endif

#endif // MY_HEADER_H

或者它可以通过相应的 .hpp 标头间接包含在extern "C"声明中。

例如：

#ifndef MY_HEADER_HPP
#define MY_HEADER_HPP

extern "C"
{
#include "my_header.h"
}

#endif // MY_HEADER_HPP

和：

#ifndef MY_HEADER_H
#define MY_HEADER_H

void myCFunction() ;

#endif // MY_HEADER_H
