功能分支变基后 Git 推送被拒绝

Arya

2023-02-17 03:53

好的，我以为这是一个简单的 git 场景，我错过了什么？

我有一个master分支和一个feature分支。我做了一些工作master，一些feature，然后更多master。我最终得到这样的结果（字典顺序意味着提交的顺序）：

A--B--C------F--G  (master)
       \    
        D--E  (feature)

我可以毫无问题地git push origin master保持远程master更新，也可以git push origin feature（打开时feature）为我的工作维护远程备份feature。到现在为止，我们都很好。

但是现在我想在 master 上的提交feature之上进行 rebase F--G，所以我git checkout feature和git rebase master. 还好。现在我们有：

A--B--C------F--G  (master)
                 \
                  D'--E'  (feature)

问题： 当我想备份新的变基feature分支时git push origin feature， 推送被拒绝， 因为树由于变基而改变了。这只能通过git push --force origin feature.

--force我讨厌在不确定自己是否需要它的情况下使用它。那么，我需要它吗？变基 是否必然 意味着下一个push应该是--force完整的？

这个功能分支不与任何其他开发人员共享，所以我对强制 推送没有任何问题，我不会丢失任何数据，这个问题更概念化。



解答

问题是git push假设远程分支可以快速转发到您的本地分支，也就是说本地和远程分支之间的所有区别都在于本地在末尾有一些新的提交，如下所示：

Z--X--R         <- origin/some-branch (can be fast-forwarded to Y commit)
       \        
        T--Y    <- some-branch

当您执行git rebase提交时，D 和 E 将应用于新的基础并创建新的提交。这意味着在变基之后你有类似的东西：

A--B--C------F--G--D'--E'   <- feature-branch
       \  
        D--E                <- origin/feature-branch

在那种情况下，远程分支无法快速转发到本地。虽然，理论上本地分支可以合并到远程（显然在这种情况下你不需要它），但是因为git push只执行快进合并它会抛出错误。

而--force选项所做的只是忽略远程分支的状态并将其设置为您正在推送的提交。所以简单地用 localgit push --force origin feature-branch覆盖。origin/feature-branch``feature-branch

master在我看来，只要你是唯一一个在该分支上工作的人，将功能分支变基并强制将它们推回到远程存储库就可以了。


更多建议

而不是使用-f或--force开发人员应该使用

--force-with-lease

为什么？因为它会检查远程分支的变化，这绝对是个好主意。假设 James 和 Lisa 在同一个功能分支上工作，而 Lisa 已经推送了一个提交。James 现在重新定位他的本地分支，并在尝试推送时被拒绝。当然 James 认为这是由于 rebase 和 uses 导致的--force，并且会重写 Lisa 的所有更改。如果 James 使用过，--force-with-lease他会收到警告，提示有人提交了其他人。我不明白为什么有人会在 rebase 之后使用--force而不是push 。--force-with-lease
