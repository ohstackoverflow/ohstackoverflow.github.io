__dirname 和 ./ 在 node.js 中有什么区别？

Arya

2023-02-18 02:18

当在 Node.js 中编程并引用位于与当前目录相关的某个位置的文件时，是否有任何理由使用变量__dirname而不是常规变量./？到目前为止，我一直在我的代码中使用 ./，只是发现了 的存在__dirname，并且基本上想知道将我的 ./ 转换为它是否明智，如果是这样，为什么这是一个聪明的主意.



解答

要旨

在 Node.js 中，__dirname始终是当前正在执行的脚本所在的目录（参见 this）。因此，如果您键入__dirnameinto /d1/d2/myscript.js，则值为/d1/d2。

相比之下，当您使用和等库时，它会为您提供在终端窗口中.运行命令的目录（即您的工作目录） 。从技术上讲，它最初是您的工作目录，但可以使用.node``path``fs``process.chdir()

例外情况是您使用.with时require()。里面的路径require总是相对于包含对require.

例如...

假设您的目录结构是

/dir1
  /dir2
    pathtest.js

并pathtest.js包含

var path = require("path");
console.log(". = %s", path.resolve("."));
console.log("__dirname = %s", path.resolve(__dirname));

你也是

cd /dir1/dir2
node pathtest.js

你得到

. = /dir1/dir2
__dirname = /dir1/dir2

您的工作目录就是/dir1/dir2这样.解决的。由于pathtest.js位于/dir1/dir2那也是__dirname解决的问题。

但是，如果您从/dir1

cd /dir1
node dir2/pathtest.js

你得到

. = /dir1
__dirname = /dir1/dir2

在那种情况下，您的工作目录就是/dir1这样，这就是.解决的问题，但__dirname仍然解决了/dir1/dir2.

使用.内部require...

如果dir2/pathtest.js你在里面require调用 include 一个文件，dir1你 总是 会这样做

require('../thefile')

因为里面的路径require总是相对于你调用它的文件。它与您的工作目录无关。


更多建议

./指的是当前工作目录，require()函数中除外。使用时require()，它转换./为当前调用的文件的目录。__dirname始终是当前文件的目录。

例如，具有以下文件结构

/home/user/dir/files/config.json

{
  "hello": "world"
}

/home/user/dir/files/somefile.txt

text file

/home/user/dir/dir.js

var fs = require('fs');

console.log(require('./files/config.json'));
console.log(fs.readFileSync('./files/somefile.txt', 'utf8'));

如果我cd进入/home/user/dir并运行node dir.js我会得到

{ hello: 'world' }
text file

但是当我运行相同的脚本时，/home/user/我得到

{ hello: 'world' }

Error: ENOENT, no such file or directory './files/somefile.txt'
    at Object.openSync (fs.js:228:18)
    at Object.readFileSync (fs.js:119:15)
    at Object.<anonymous> (/home/user/dir/dir.js:4:16)
    at Module._compile (module.js:432:26)
    at Object..js (module.js:450:10)
    at Module.load (module.js:351:31)
    at Function._load (module.js:310:12)
    at Array.0 (module.js:470:10)
    at EventEmitter._tickCallback (node.js:192:40)

使用./worked with requirebut not for fs.readFileSync. 那是因为 forfs.readFileSync转换./为 cwd（在本例中为/home/user/）。并且/home/user/files/somefile.txt不存在。
