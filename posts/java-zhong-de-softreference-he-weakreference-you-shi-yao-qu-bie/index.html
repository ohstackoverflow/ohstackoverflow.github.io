Java 中的 SoftReference 和 WeakReference 有什么区别？

Arya

2023-02-17 10:10

java.lang.ref.WeakReference和 和有什么不一样java.lang.ref.SoftReference？



解答

来自Ethan Nicholas 的Understanding Weak References ：

弱引用

弱 引用 ，简单地说，就是一种引用的强度不足以强制对象保留在内存中。弱引用允许您利用垃圾收集器的能力来确定您的可达性，因此您不必自己做。您创建这样的弱引用：

WeakReference weakWidget = new WeakReference(widget);

然后在代码的其他地方，您可以使用它weakWidget.get()来获取实际Widget对象。当然，弱引用的强度不足以阻止垃圾回收，因此您可能会发现（如果没有对小部件的强引用）weakWidget.get()突然开始返回null。

...

软引用

软 引用 与弱引用完全一样，只是它不那么急于丢弃它所引用的对象。仅弱可达的对象（对它的最强引用是WeakReferences）将在下一个垃圾收集周期中被丢弃，但软可达的对象通常会停留一段时间。

SoftReferences不需要 与 有任何不同的行为WeakReferences，但实际上只要内存供应充足，软可达对象通常会被保留。这使它们成为缓存的极好基础，例如上面描述的图像缓存，因为您可以让垃圾收集器担心对象的可达性（强可达对象永远不会从缓存中删除）以及它的 严重 程度需要他们正在消耗的内存。

Peter Kessler 在评论中补充道：

Sun JRE 对待 SoftReferences 的方式与 WeakReferences 不同。如果可用内存没有压力，我们会尝试保留 SoftReference 引用的对象。一个细节：“-client”和“-server”JRE 的策略不同：-client JRE 试图通过更喜欢清除 SoftReferences 而不是扩展堆来保持占用空间小，而 -server JRE 试图保持你的通过更喜欢扩展堆（如果可能）而不是清除 SoftReferences 来提高性能。一种尺寸并不适合所有尺寸。


更多建议

急切地收集弱引用。如果 GC 发现一个对象是弱可达的（只能通过弱引用可达），它会立即清除对该对象的弱引用。因此，它们很适合保留对某个对象的引用，您的程序也为其保留（强引用）“关联信息”，例如有关类的缓存反射信息或对象的包装器等。任何使在与它关联的对象被 GC-ed 之后保留它是没有意义的。当弱引用被清除时，它会排入您的代码在某处轮询的引用队列中，并且它也会丢弃关联的对象。也就是说，您保留了有关对象的额外信息，但一旦它所引用的对象消失，就不需要该信息。实际上，在某些情况下，您甚至可以将 WeakReference 子类化，并在 WeakReference 子类的字段中保留有关该对象的相关额外信息。WeakReference 的另一个典型用途是与 Maps 结合使用以保持规范实例。

另一方面，SoftReferences 适用于缓存外部的、可重新创建的资源，因为 GC 通常会延迟清除它们。虽然可以保证在抛出 OutOfMemoryError 之前所有 SoftReferences 都会被清除，因此理论上它们不会导致 OOME[*]。

典型的用例示例是保留文件中内容的解析形式。您将实现一个系统，在该系统中您将加载一个文件、对其进行解析，并将 SoftReference 保存到已解析表示的根对象。下次您需要该文件时，您将尝试通过 SoftReference 检索它。如果你能检索到它，你就省去了另一次加载/解析，如果 GC 同时清除了它，你就重新加载它。这样，您就可以利用空闲内存来优化性能，但不会冒 OOME 的风险。

现在为 [*]。保留 SoftReference 本身不会导致 OOME。另一方面，如果你错误地将 SoftReference 用于一个任务，一个 WeakReference 是要使用的（即，你保持与对象相关联的信息以某种方式被强烈引用，并在 Reference 对象被清除时丢弃它），你可以运行 OOME 作为您轮询 ReferenceQueue 并丢弃关联对象的代码可能碰巧没有及时运行。

因此，决定取决于使用情况——如果你正在缓存构建起来很昂贵但仍然可以从其他数据重建的信息，请使用软引用——如果你要保留对某些数据的规范实例的引用，或者你想引用一个对象而不“拥有”它（从而防止它被 GC'd），使用弱引用。
