我如何正确地强制 Git 推送？

Arya

2023-02-17 00:32

我已经设置了一个远程非裸“主”回购并将其克隆到我的计算机上。我做了一些本地更改，更新了我的本地存储库，并将更改推送回我的远程存储库。到那时为止一切都很好。

现在，我不得不更改远程仓库中的某些内容。然后我在本地仓库中更改了一些内容。我意识到不需要更改远程仓库。所以我尝试git push从我的本地回购到我的远程回购，但我收到如下错误：

为了防止您丢失历史记录，拒绝了非快进更新在再次推送之前合并远程更改。git push --help有关详细信息，请参阅“关于快进的注意事项”部分。

我想那可能是一个

git push --force

会强制我的本地副本将更改推送到远程副本并使其相同。 它确实强制更新 ，但是当我返回远程仓库并进行提交时，我注意到文件包含过时的更改（主要远程仓库以前拥有的更改）。

正如我在其中一个答案的评论中提到的：

[我] 尝试过强制，但是当返回到主服务器以保存更改时，我得到了过时的暂存。因此，当我提交时，存储库是不一样的。当我再次尝试使用 git push 时，我得到了同样的错误。

我该如何解决这个问题？



解答

做就是了：

git push origin <your_branch_name> --force

或者如果你有一个特定的回购协议：

git push https://git.... --force

这将删除您之前的提交并推送您当前的提交。

它可能不合适，但如果有人偶然发现此页面，认为他们可能需要一个简单的解决方案......

短旗

另请注意，它-f是 的缩写--force，所以

git push origin <your_branch_name> -f

也会工作。


更多建议

如果push --force不起作用，你可以做 push --delete 。查看此实例的第 2 行：

git reset --hard HEAD~3  # reset current branch to 3 commits ago
git push origin master --delete  # do a very very bad bad thing
git push origin master  # regular push

但要小心...

永远不要回到公开的 git 历史！

换句话说：

  • 永远不要force推送公共存储库。
  • 不要这样做或任何可能破坏某人pull.
  • 永远不要在某人可能已经提取的 回购协议reset中或rewrite历史记录中。 __

当然，即使是这条规则也有非常罕见的例外，但在大多数情况下不需要这样做，而且会给其他人带来麻烦。

相反，做一个还原。

并 始终小心你推送到公共回购的内容 。恢复：

git revert -n HEAD~3..HEAD  # prepare a new commit reverting last 3 commits
git commit -m "sorry - revert last 3 commits because I was not careful"
git push origin master  # regular push

实际上， 两个 原始 HEAD（来自 revert 和来自 邪恶的 reset ）将包含相同的文件。

编辑以添加更新的信息和更多参数push --force

考虑使用 lease 而不是 push 来推动力，但仍然更喜欢 revert

另一个问题push --force可能是有人在你之前推送任何东西，但在你已经获取之后。如果你现在强制你的 rebased 版本，你将 替换其他人的工作 。

git push --force-with-lease在git 1.8.5中引入（感谢@VonC对问题的评论）试图解决这个特定问题。基本上，如果自从您最近一次获取后修改了遥控器，它将带来错误并且不会推送。

如果您确实确定push --force需要 a，但仍想防止出现更多问题，这很好。我会说这应该是默认push --force行为。但这还远远不能成为强制push. 在你的rebase 之前 获取 的人仍然会有很多麻烦，如果你已经 恢复 ，这些问题可以很容易地避免。

既然我们在谈论git --push实例......

为什么会有人想要强制推送？

@linquiz在评论中带来了一个很好的推力示例： 敏感数据 。您错误地泄露了不应推送的数据。如果你足够快，你可以通过在顶部强行推动来 “修复”它。*

*除非您还执行垃圾收集或以某种方式清理数据，否则数据仍将位于远程服务器上。它也有可能被已经 获取 它的其他人传播，但你明白了。 __
