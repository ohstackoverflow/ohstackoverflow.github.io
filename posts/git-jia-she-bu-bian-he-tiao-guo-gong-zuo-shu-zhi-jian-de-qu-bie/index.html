Git - “假设不变”和“跳过工作树”之间的区别

Arya

2023-02-17 21:30

我对不想提交到我的存储库的文件进行了本地更改。它是用于在服务器上构建应用程序的配置文件，但我想使用不同的设置在本地构建。自然地，当我将“git status”作为要暂存的东西时，文件总是会出现。我想隐藏此特定更改而不提交它。我不会对文件进行任何其他更改。

澄清一下，使用 .gitignore 不是我想要的，因为这只会阻止添加新文件。我想忽略对存储库中已有文件的更改。

经过一番挖掘，我看到了 2 个选项：assume-unchanged和skip-worktree. 上一个问题在这里谈到了它们，但并没有真正解释它们的区别。

这两个命令有何不同？为什么有人会使用一个或另一个？



解答

你想要skip-worktree。

assume- unchanged专为检查一组文件是否已被修改的情况而设计；当您设置该位时，git（当然）假定与索引的该部分对应的文件在工作副本中未被修改。所以它避免了混乱的stat调用。每当文件在索引中的条目更改时（因此，当文件在上游更改时），该位就会丢失。

skip-worktree不仅如此：即使 wheregit 知道 文件已被修改（或需要由 areset --hard等修改），它也会假装它没有被修改，而是使用索引中的版本。这一直持续到索引被丢弃。

此处对这种差异的后果和典型用例进行了很好的总结：http://fallengamer.livejournal.com/93321.html。

从那篇文章：

  • --assume-unchanged假定开发人员 不应 更改文件。此标志旨在提高SDK 等不变文件夹的 性能。
  • --skip-worktree当您指示 git 不要触摸特定文件时很有用，因为开发人员 应该 更改它。例如，如果主存储库上游托管了一些生产就绪的 配置文件 ，并且您不想意外地提交对这些文件的更改，--skip-worktree这正是您想要的。

操作

  • 文件在本地存储库和上游都被更改
    git pull ：
    Git 无论如何都会保留本地更改。
    因此，您不会意外丢失用任何标志标记的任何数据。

    • 带有标志的文件 assume-unchanged ：Git 不会覆盖本地文件。相反，它会输出冲突并建议如何解决它们
    • 带有标志的文件 skip-worktree ：Git 不会覆盖本地文件。相反，它会输出冲突并建议如何解决它们

  • 文件在本地存储库和上游都被更改，无论如何尝试拉动使用结果需要一些额外的手动工作，但至少如果您有任何本地更改，您不会丢失任何数据。
    git stash
    git pull
    skip-worktree

    • 带有标志的文件 assume-unchanged ：丢弃所有本地更改，并且没有任何恢复它们的可能性。效果就像' git reset --hard'。' git pull'调用会成功
    • 带有标志的文件 skip-worktree ：Stash 不适用于skip-worktree文件。' git pull' 将失败并出现与上述相同的错误。开发人员被迫手动重置skip-worktree标志，以便能够隐藏并完成失败pull。

  • 没有本地更改，上游文件已更改这两个标志都不会阻止您获取上游更改。Git 检测到您违背了承诺，并选择通过重置标志来反映现实。
    git pull
    assume-unchanged

    • 带有标志的文件 assume-unchanged ：内容已更新，标志丢失。
      ' git ls-files -v' 将显示标志已修改为H(from h)。

    • 带标志的文件 skip-worktree ：内容更新，标志保留。
      ' git ls-files -v' 将显示S与之前相同的标志pull。

  • 更改本地文件后，Git 不会触及文件并反映文件的真实情况（承诺不变的文件实际上已更改）。
    git reset --hard
    skip-worktree``assume-unchanged

    • 带标志的文件 assume-unchanged ：还原文件内容。标志重置为H(from h)。
    • 带标志的文件 skip-worktree ：文件内容完好无损。国旗保持不变。

他补充了以下分析：

  • 看起来 skip-worktree 正在 非常努力地保存您的本地数据 。但它不会阻止您在安全的情况下进行上游更改。另外 git 不会重置pull.
    但是忽略 'reset --hard ' 命令可能会让开发人员 大吃一惊。

  • Assume-unchanged标志可能会在操作中丢失pull，并且此类文件中的本地更改对 git 似乎并不重要。

Junio 特别指出，assume-unchanged文件更改可能会意外提交：“如果 Git 可以确定标记为assume- unchanged已更改的路径而不会产生额外的 lstat(2) 成本，它保留报告路径 已被 修改的权利（因此，git commit -a可以自由地进行更改）。”

他总结道：

实际上，这两个标志都不够直观 。

  • assume-unchanged 假定开发人员不应更改文件。如果一个文件被改变了——那么那个改变并不重要。此标志旨在提高 SDK 等不变文件夹的性能。
    但是如果诺言被打破并且文件实际上被更改了，git 会恢复标志以反映现实。在通常不打算更改的文件夹中有一些不一致的标志可能是可以的。

  • 另一方面， skip-worktree 当您指示 git 不要触摸特定文件时，这很有用。这对于已经跟踪的配置文件很有用。
    上游主存储库托管一些生产就绪的配置，但您希望更改配置中的一些设置以便能够进行一些本地测试。而且您不想不小心检查此类文件中的更改以影响生产配置。在那种情况下skip- worktree是完美的场景。

doc：劝阻用户不要试图忽略跟踪的文件

签字人：Jeff King
签字人：brian m. 卡尔森

用户想要忽略 Git 跟踪的文件更改是很常见的。

这种情况的常见场景是 IDE 设置和配置文件，它们通常不应被跟踪，并且可能使用模板机制从跟踪文件生成。

但是，用户了解了 assume-unchanged 和 skip-worktree 位，并尝试使用它们来执行此操作。

git checkout这是有问题的，因为当设置这些位时，许多操作的行为都符合用户的预期，但在需要替换文件时它们通常无济于事。

在这种情况下没有明智的行为，因为有时数据是宝贵的，例如某些配置文件，有时是不相关的数据，用户会很乐意丢弃。

由于这不是受支持的配置，并且用户很容易将现有功能误用于非预期目的，从而导致普遍的悲伤和困惑 ，让我们在文档中记录现有行为和陷阱，以便git update-index用户知道他们应该探索替代解决方案。

此外，让我们提供一个推荐的解决方案来处理配置文件的常见情况，因为在许多环境中都成功使用了众所周知的方法。

手册git update-index页现在包括：

用户经常尝试使用assume-unchanged和skip-worktree位来告诉 Git 忽略对被跟踪文件的更改。这并不像预期的那样工作，因为 Git 在执行某些操作时可能仍会根据索引检查工作树文件。通常，Git 不提供忽略跟踪文件更改的方法，因此建议使用替代解决方案。

例如，如果您要更改的文件是某种配置文件，则存储库可以包含一个示例配置文件，然后可以将其复制到忽略的名称中并进行修改。存储库甚至可以包含一个脚本，将示例文件视为模板，自动修改和复制它。
