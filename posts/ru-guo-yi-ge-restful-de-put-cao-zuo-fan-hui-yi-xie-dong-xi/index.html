如果一个 RESTful 的“PUT”操作返回一些东西......

Arya

2023-02-18 05:06

PUT我想知道人们对在响应正文中不返回任何内容（null）的 RESTful 操作有何看法。



解答

HTTP 规范 ( RFC 2616 ) 有许多适用的建议。这是我的解释：

  • 200 OK对现有资源的更新成功 PUT 的HTTP 状态代码。不需要响应主体。（根据第 9.6 节，204 No Content更合适。）
  • 新资源成功 PUT 的HTTP 状态代码201 Created，在 Location 标头字段中返回新资源的最具体 URI，响应正文中回显该资源的任何其他相关 URI 和元数据。（RFC 2616 第 10.2.2 节）
  • 409 Conflict由于第 3方修改而失败的 PUT 的HTTP 状态代码，在响应正文中包含尝试的更新与当前资源之间的差异列表。（RFC 2616 第 10.4.10 节）
  • PUT 失败的HTTP 状态代码400 Bad Request，在响应正文中使用自然语言文本（例如英语）解释 PUT 失败的原因。（RFC 2616 第 10.4 节）


更多建议

与这里的大多数答案相反，我实际上认为 PUT 应该返回更新后的资源（当然除了 HTTP 代码之外）。

您之所以希望返回资源作为 PUT 操作的响应，是因为当您向服务器发送资源表示时，服务器也可以对该资源进行一些处理，因此客户端想知道该资源是如何处理的请求成功完成后的样子。（否则它将不得不发出另一个 GET 请求）。
