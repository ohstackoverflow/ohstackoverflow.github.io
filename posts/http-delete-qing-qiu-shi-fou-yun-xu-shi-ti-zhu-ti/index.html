HTTP DELETE 请求是否允许实体主体？

Arya

2023-02-17 07:59

发出 HTTP DELETE 请求时，请求 URI 应完全标识要删除的资源。但是，是否允许添加额外的元数据作为请求实体主体的一部分？



解答

该规范没有明确禁止或阻止它，所以我倾向于说它是允许的。

Microsoft 也有同样的看法（我能听到听众的窃窃私语），他们在 MSDN 文章中提到了ADO.NET 数据服务框架的 DELETE 方法：

如果 DELETE 请求包含实体主体，则实体将被忽略 [...]

此外，这是RFC2616 (HTTP 1.1) 关于请求的内容：

  • 实体 主体 仅在 消息主体 存在时才存在（第 7.2 节）
  • 消息体 的存在通过包含Content-Length或Transfer-Encoding标头来表示（第 4.3 节）
  • 当请求方法的规范不允许发送实体主体时，不得包含消息 主体 （ 第 4.3 节）
  • 仅在 TRACE 请求中明确禁止实体 主体 ，所有其他请求类型不受限制（第 9 节，特别是 9.8）

对于响应，这已被定义：

  • 是否包含 消息体 取决于请求方法 和 响应状态（第 4.3 节）
  • 明确禁止响应 HEAD 请求的消息正文（第 9 节，特别是9.4 ）
  • 消息 正文 在 1xx（信息）、204（无内容）和 304（未修改）响应中被明确禁止（第 4.3 节）
  • 所有其他响应都包含消息正文，尽管它的长度可能为零（第 4.3 节）

更新

在 RFC 9110（2022 年 6 月）中，GET、HEAD 和 DELETE 上的请求主体不可互操作这一事实已得到澄清。

9.3.5 节删除

虽然请求消息框架独立于所使用的方法，但在 DELETE 请求中接收的内容没有一般定义的语义，不能改变请求的含义或目标，并且可能导致某些实现拒绝请求并关闭连接，因为它的潜在作为请求走私攻击（[HTTP/1.1] 的第 11.2 节）。客户端不应该在 DELETE 请求中生成内容，除非直接向源服务器发出，该源服务器先前已在带内或带外指示此类请求具有目的并将得到充分支持。源服务器不应该依赖私有协议来接收内容，因为 HTTP 通信的参与者通常不知道请求链上的中介。


更多建议

HTTP 1.1 规范 ( RFC 7231 ) 的 2014 年更新明确允许 DELETE 请求中的实体主体：

DELETE 请求消息中的有效载荷没有定义的语义；在 DELETE 请求上发送有效负载主体可能会导致某些现有实现拒绝该请求。
