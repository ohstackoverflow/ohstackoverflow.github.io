“=”和“<-”赋值运算符有什么区别？

Arya

2023-02-17 09:39

=赋值运算符和R中的区别是什么<-？

我知道运算符略有不同，如本例所示

x <- y <- 5
x = y = 5
x = y <- 5
x <- y = 5
# Error in (x <- y) = 5 : could not find function "<-<-"

但这是唯一的区别吗？



解答

=赋值运算符和R中的区别是什么<-？

如您的示例所示，=并且<-运算符优先级略有不同（当它们混合在同一表达式中时，这决定了评估的顺序）。实际上，?Syntax在 R 中给出了如下运算符优先级表，从高到低：

…
‘-> ->>’           rightwards assignment
‘<- <<-’           assignment (right to left)
‘=’                assignment (right to left)
…

但这是唯一的区别吗？

由于您询问的是 赋值运算符 ：是的，这是唯一的区别。但是，您不这样认为是可以原谅的。?assignOps甚至声称存在更多差异的R 文档：

运算符<-可以在任何地方使用，而运算符=只允许在顶层使用（例如，在命令提示符下键入的完整表达式中）或作为括号表达式列表中的子表达式之一。

让我们不要过分强调它： R 文档是错误的 。这很容易证明：我们只需要找到一个运算符的反例，=它不是 (a) 在顶层，也不是 (b) 花括号表达式列表中的子表达式（即{…; …}）。- 无需再费周折：

x
# Error: object 'x' not found
sum((x = 1), 2)
# [1] 3
x
# [1] 1

=显然，我们在上下文 (a) 和 (b) 之外使用 执行了一项任务。那么，为什么核心 R 语言功能的文档几十年来一直是错误的？

这是因为在 R 的语法中，符号=有两个不同的含义，经常被混淆（即使是专家，包括在上面引用的文档中）：

  1. 第一个意思是作为 赋值运算符 。到目前为止，这就是我们所讨论的全部内容。
  2. 第二个含义不是运算符，而是表示在函数调用中 传递命名参数的 语法标记。 与运算 符 不同，它在运行时不执行任何操作，它仅更改表达式的解析方式。 =

那么 R 如何决定给定的用法是=指运算符还是命名参数传递？让我们来看看。

在任何一般形式的代码中......

‹function_name›(‹argname› = ‹value›, …)
‹function_name›(‹args›, ‹argname› = ‹value›, …)

… the = 是定义命名参数传递的标记：它不是 赋值 运算符。此外，在某些句法上下文中=是完全 禁止的：

if (‹var› = ‹value›) …
while (‹var› = ‹value›) …
for (‹var› = ‹value› in ‹value2›) …
for (‹var1› in ‹var2› = ‹value›) …

这些中的任何一个都会引发错误“在‹bla›中出现意外的'='”。

在任何其他上下文中，=指的是赋值运算符调用。特别是，仅在子表达式周围加上括号就可以使上述 (a) 中的任何一个有效，并且 (b) 成为一个 赋值 。例如，以下执行分配：

median((x = 1 : 10))

但是也：

if (! (nf = length(from))) return()

现在您可能会反对这样的代码是残酷的（您可能是对的）。但我从函数中提取了这段代码base::file.copy（替换<-为=）——它是许多核心 R 代码库中的普遍模式。

R 文档可能基于John Chambers 的原始解释实际上正确地解释了这一点：

[=赋值]只允许在语法中的两个地方：在顶层（作为一个完整的程序或用户输入的表达式）；当与周围的逻辑结构隔离时，通过大括号或一对额外的括号。

总之，默认情况下运算符<-和=做同样的事情。但是它们中的任何一个都可以被单独 覆盖 以改变它的行为。相比之下，<-and ->（从左到右的赋值）虽然在语法上不同，但总是调用 相同的 函数。覆盖一个也会覆盖另一个。知道这一点很少有实际意义，但 可以 用于一些有趣的恶作剧。
