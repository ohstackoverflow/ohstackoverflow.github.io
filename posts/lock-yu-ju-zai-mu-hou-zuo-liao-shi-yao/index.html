lock 语句在幕后做了什么？

Arya

2023-02-17 22:02

我看到为了使用非线程安全的对象，我们用这样的锁包装代码：

private static readonly Object obj = new Object();

lock (obj)
{
    // thread unsafe code
}

那么，当多个线程访问同一代码时会发生什么（假设它在 ASP.NET Web 应用程序中运行）。他们在排队吗？如果是这样，他们会等多久？

使用锁对性能有何影响？



解答

该lock语句由 C# 3.0 翻译为以下内容：

var temp = obj;

Monitor.Enter(temp);

try
{
    // body
}
finally
{
    Monitor.Exit(temp);
}

在 C# 4.0 中，这发生了变化，现在生成如下：

bool lockWasTaken = false;
var temp = obj;
try
{
    Monitor.Enter(temp, ref lockWasTaken);
    // body
}
finally
{
    if (lockWasTaken)
    {
        Monitor.Exit(temp); 
    }
}

您可以在此处找到有关功能的更多Monitor.Enter信息。引用 MSDN：

用于Enter获取作为参数传递的对象上的 Monitor。如果另一个线程已经Enter在该对象上执行了一个但还没有执行相应的Exit，则当前线程将阻塞直到另一个线程释放该对象。Enter同一个线程在不阻塞的情况下多次调用是合法的；Exit但是，在等待该对象的其他线程解除阻塞之前，必须调用相同数量的调用。

该Monitor.Enter方法将无限等待；它不会 超时 。


更多建议

它比你想象的要简单——

根据Microsoft 的说法：lock关键字确保一个线程不会进入代码的临界区，而另一个线程则在临界区中。如果另一个线程试图输入锁定的代码，它将等待、阻塞，直到对象被释放。

关键字在块的开头和块的结尾调用lock。关键字实际上在后端处理类。Enter``Exit``lock``Monitor

例如：

private static readonly Object obj = new Object();

lock (obj)
{
    // critical section
}

上面代码中，首先线程进入临界区，然后加锁obj。当另一个线程试图进入时，它也会尝试锁定obj，它已经被第一个线程锁定。第二个线程将不得不等待第一个线程释放obj。当第一个线程离开时，另一个线程将锁定obj并进入临界区。
