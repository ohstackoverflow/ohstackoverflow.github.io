为什么 GCC 不将 a*a*a*a*a*a 优化为 (a*a*a)*(a*a*a)？

Arya

2023-02-16 21:58

我正在对科学应用程序进行一些数值优化。pow(a,2)我注意到的一件事是，GCC 会通过将调用编译成 来优化调用a*a，但调用pow(a,6)没有优化，实际上会调用库函数pow，这大大降低了性能。（相比之下，英特尔 C++ 编译器可执行文件icc将消除对库的调用pow(a,6)。）

我很好奇的是，当我替换pow(a,6)为a*a*a*a*a*a使用 GCC 4.5.1 和选项“ -O3 -lm -funroll-loops -msse4”时，它使用了 5mulsd条指令：

movapd  %xmm14, %xmm13
mulsd   %xmm14, %xmm13
mulsd   %xmm14, %xmm13
mulsd   %xmm14, %xmm13
mulsd   %xmm14, %xmm13
mulsd   %xmm14, %xmm13

而如果我写(a*a*a)*(a*a*a)，它会产生

movapd  %xmm14, %xmm13
mulsd   %xmm14, %xmm13
mulsd   %xmm14, %xmm13
mulsd   %xmm13, %xmm13

它将乘法指令的数量减少到 3。icc具有类似的行为。

为什么编译器无法识别这种优化技巧？



解答

因为Floating Point Math 不是 Associative。在浮点乘法中对操作数进行分组的方式会影响答案的数值准确性。

因此，大多数编译器对重新排序浮点计算非常保守，除非他们可以确定答案将保持不变，或者除非你告诉他们你不关心数值准确性。例如：允许 gcc 重新关联浮点操作的 gcc选项-fassociative-math，甚至-ffast-math允许更积极地权衡准确性与速度的选项。


更多建议

Lambdageek 正确地指出，由于结合律不适用于浮点数，因此a*a*a*a*a*ato(a*a*a)*(a*a*a)可能会更改值。这就是 C99 不允许它的原因（除非用户通过编译器标志或 pragma 特别允许）。通常，假设程序员写下她所做的是有原因的，编译器应该尊重这一点。如果你愿意(a*a*a)*(a*a*a)，写那个。

不过，写起来可能很痛苦；为什么编译器不能在你使用时做[你认为的]正确的事情pow(a,6)？因为这样做是 错误的。 在具有良好数学库的平台上，pow(a,6)比a*a*a*a*a*aor更准确(a*a*a)*(a*a*a)。只是为了提供一些数据，我在我的 Mac Pro 上进行了一个小实验，测量在 [1,2] 之间的所有单精度浮点数计算 a^6 时的最严重错误：

worst relative error using    powf(a, 6.f): 5.96e-08
worst relative error using (a*a*a)*(a*a*a): 2.94e-07
worst relative error using     a*a*a*a*a*a: 2.58e-07

使用pow而不是乘法树将误差范围减少了 4 倍 。编译器不应该（并且通常不会）进行增加错误的“优化”，除非用户许可这样做（例如通过-ffast- math）。

请注意，GCC 提供了__builtin_powi(x,n)的替代方法pow( )，它应该生成内联乘法树。如果您想以准确性换取性能，但又不想启用快速数学，请使用它。
