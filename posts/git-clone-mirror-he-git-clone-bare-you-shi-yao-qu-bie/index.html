git clone --mirror 和 git clone --bare 有什么区别

Arya

2023-02-17 21:50

git clone 帮助页面有这样的说法--mirror：

设置远程存储库的镜像。这意味着--bare。

但没有详细说明--mirror克隆人与克隆人有何不同--bare。



解答

不同之处在于，在使用时--mirror， 所有引用都 按原样 复制。这意味着一切：远程跟踪分支、注释、refs/originals/*（来自过滤器分支的备份）。克隆的仓库拥有一切。它还进行了设置，以便远程更新将从原点重新获取所有内容（覆盖复制的引用）。这个想法实际上是镜像存储库，拥有一个完整的副本，这样你就可以在多个地方托管你的中央存储库，或者备份它。想想直接复制 repo，除了以更优雅的 git 方式。

新文档几乎说明了所有这些：

--mirror

设置源存储库的镜像。这意味着--bare。与 相比--bare，--mirror不仅将源的本地分支映射到目标的本地分支，它还映射了所有 refs（包括远程分支、注释等）并设置了 refspec 配置，使得所有这些 refs 都被目标存储库中的 agit remote update覆盖.

我原来的回答还指出了裸克隆和普通（非裸）克隆之间的区别——非裸克隆设置了远程跟踪分支，只为创建本地分支，而裸克隆直接复制分支HEAD。

假设 origin 有几个分支（master (HEAD)、next、pu和maint）、一些标签（v1、v2、v3）、一些远程分支（devA/master、devB/master）和一些其他引用（refs/foo/bar、refs/foo/baz，可能是注释、存储、其他开发人员的名称空间，谁知道呢）。

  • git clone origin-url （非裸）：您将复制所有标签，一个本地分支master (HEAD)跟踪一个远程分支origin/master，以及远程分支origin/next、origin/pu和origin/maint。设置跟踪分支以便如果您执行类似 的操作git fetch origin，它们将按您期望的方式获取。任何远程分支（在克隆的远程中）和其他引用都被完全忽略。

  • git clone --bare origin-url ：您将复制所有标签，本地分支master (HEAD)，，，和，next没有远程跟踪分支。也就是说，所有分支都按原样复制，并且完全独立设置，不期望再次获取。任何远程分支（在克隆的远程中）和其他引用都被完全忽略。pu``maint

  • git clone --mirror origin-url ：这些参考文献中的每一个都将按原样复制。您将获得所有标签、本地分支master (HEAD)、、、和、远程分支和、其他参考和。一切都与克隆遥控器中的完全一样。设置了远程跟踪，以便如果您运行所有引用都将从原始位置覆盖，就好像您刚刚删除了镜像并重新克隆了它一样。正如文档最初所说，它是一面镜子。它应该是功能相同的副本，可以与原件互换。next``pu``maint``devA/master``devB/master``refs/foo/bar``refs/foo/baz``git remote update


更多建议

$ git clone --mirror $URL

是简写

$ git clone --bare $URL
$ (cd $(basename $URL) && git remote add --mirror=fetch origin $URL)

（直接从这里复制）

与 相比--bare，--mirror不仅将源的本地分支映射到目标的本地分支，它还映射了所有 refs（包括远程分支、注释等）并设置了 refspec 配置，使得所有这些 refs 都被目标存储库中的 agit remote update覆盖.
