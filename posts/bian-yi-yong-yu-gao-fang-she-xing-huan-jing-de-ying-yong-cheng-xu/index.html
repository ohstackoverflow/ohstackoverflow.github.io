编译用于高放射性环境的应用程序

Arya

2023-02-17 01:17

我们正在编译一个嵌入式 C++ 应用程序，该应用程序部署在受电离辐射轰击的环境中的屏蔽设备中。我们正在使用 GCC 和 ARM 的交叉编译。部署后，我们的应用程序会生成一些错误数据，并且比我们希望的更频繁地崩溃。硬件是为这个环境设计的，我们的应用程序已经在这个平台上运行了好几年。

我们是否可以对我们的代码进行更改，或者可以进行编译时改进以识别/纠正由单个事件干扰引起的软错误和内存损坏？有没有其他开发人员成功地减少了软错误对长时间运行的应用程序的有害影响？



解答

从事小型卫星*软件/固件开发和环境测试工作约4-5年，我想在这里分享我的经验。

*（ 小型卫星比大卫星更容易发生单事件干扰，因为它的电子元件相对较小，尺寸有限 ）

非常简洁和直接：没有任何机制可以从软件/固件本身 可检测到的错误情况中恢复， 至少 在某处 有一份软件/固件的 最低工作版本 副本 用于 恢复 目的 - 并且有 硬件支持恢复 （功能）。

现在，这种情况通常在硬件和软件层面都得到了处理。在这里，应您的要求，我将分享我们在软件层面可以做的事情。

  1. ...恢复目的... 提供在真实环境中更新/重新编译/刷新您的软件/固件的能力。这几乎是高度电离环境中任何软件/固件 必备的功能。 如果没有这个，你 可以 拥有任意多的冗余软件/硬件，但在某一时刻，它们都会崩溃。所以，准备这个功能吧！

  2. ...最低工作版本... 在您的代码中拥有响应式、多副本、最低版本的软件/固件。这就像 Windows 中的安全模式。与其只有一个功能齐全的软件版本，不如拥有多个最低版本的软件/固件副本。最小副本通常比完整副本的大小小得多，并且几乎总是只有 以下 两个或三个特征：

    1. 能够听取来自外部系统的命令，
    2. 能够更新当前的软件/固件，
    3. 能够监控基本操作的内务数据。

    4. ...复制...某处... 某处有冗余软件/固件。

    5. 无论 有无冗余硬件，您都可以尝试在您的 ARM uC 中安装冗余软件/固件。这通常是通过 在不同的地址中 有两个或多个相同的软件/固件来相互发送心跳来完成的——但一次只有一个是活动的。如果已知一个或多个软件/固件无响应，请切换到其他软件/固件。使用这种方法的好处是我们可以在错误发生后立即进行功能替换——无需与负责检测和修复错误的任何外部系统/方进行任何联系（在卫星情况下，通常是任务控制中心（中冶））。

严格来说，在没有冗余硬件的情况下，这样做的缺点是你实际上 无法 消除 所有的 单点故障。至少，您仍然会有 一个 单点故障，即 开关本身 （或通常是代码的开头）。 尽管如此，对于在高度电离环境中受尺寸限制的设备（例如微微/毫微微卫星），在不 增加硬件的情况下将单点故障减少到一个点仍然值得考虑。此外，用于切换的代码片段肯定会比整个程序的代码少得多——显着降低了其中出现单一事件的风险。

2. 但如果你不这样做，你应该在你的外部系统中至少有一个副本，它可以与设备联系并更新软件/固件（在卫星情况下，它又是任务控制中心）。

3. 您还可以将副本保存在设备的永久内存中，可以触发该副本以恢复正在运行的系统的软件/固件

  1. ...可检测的错误情况。 错误必须是可 检测的 ，通常是通过硬件 纠错/检测电路 或一小段用于纠错/检测的代码。最好把这样的代码放在小的、多的、 独立 于主要软件/固件的地方。它的主要任务 只是 检查/纠正。硬件电路/固件是否 可靠 （例如它比其余的更抗辐射 - 或者具有多个电路/逻辑），那么您可以考虑用它进行错误校正。但如果不是，最好将其作为错误检测。可以通过外部系统/设备进行校正。对于纠错，您可以考虑使用像 Hamming/Golay23 这样的基本纠错算法，因为它们可以更容易地在电路/软件中实现。但这最终取决于您团队的能力。对于错误检测，通常使用 CRC。

  2. ...支持恢复的硬件 现在，来到这个问题上最困难的方面。最终，恢复需要负责恢复的硬件 至少 可以正常运行。如果硬件永久损坏（通常在其 总电离剂量 达到一定水平后发生），那么（遗憾的是）软件无法帮助恢复。因此，对于暴露于高辐射水平的设备（例如卫星），硬件无疑是最重要的考虑因素。

除了以上针对单事件扰乱导致预期固件错误的建议外，我还建议您：

  1. 子系统间通信协议中的错误检测和/或纠错算法。这是另一个几乎必须具备的，以避免从其他系统接收到不完整/错误的信号

  2. 过滤您的 ADC 读数。不要直接 使用 ADC 读数。通过中值过滤器、均值过滤器或任何其他过滤器对其进行过滤 - 永远不要 相信单一的读数值。采样更多，而不是更少——合理。


更多建议

NASA 有一篇关于抗辐射软件的论文。它描述了三个主要任务：

  1. 定期监控内存错误，然后清除这些错误，
  2. 强大的错误恢复机制，以及
  3. 如果某些东西不再有效，则能够重新配置。

请注意，内存扫描速率应足够频繁，以便很少发生多位错误，因为大多数ECC内存可以从单位错误中恢复，而不是多位错误。

健壮的错误恢复包括控制流传输（通常在错误之前的某个点重新启动进程）、资源释放和数据恢复。

他们对数据恢复的主要建议是避免对它的需要，通过将中间数据视为临时数据，以便在错误之前重新启动也将数据回滚到可靠状态。这听起来类似于数据库中“事务”的概念。

他们讨论了特别适合面向对象语言（如 C++）的技术。例如

  1. 用于连续内存对象的基于软件的 ECC
  2. 按合同编程：验证前置条件和后置条件，然后检查对象以验证它是否仍处于有效状态。

而且，碰巧的是，NASA 已经将 C++ 用于火星探测器等重大项目。

C++ 类抽象和封装实现了多个项目和开发人员之间的快速开发和测试。

他们避免了某些可能产生问题的 C++ 特性：

  1. 例外情况
  2. 模板
  3. Iostream（无控制台）
  4. 多重继承
  5. 运算符重载（除了newand delete）
  6. 动态分配（使用专用内存池和放置new以避免系统堆损坏的可能性）。
