为什么这些构造使用前后递增的未定义行为？

Arya

2023-02-17 10:12 

#include <stdio.h>

int main(void)
{
   int i = 0;
   i = i++ + ++i;
   printf("%d\n", i); // 3

   i = 1;
   i = (i++);
   printf("%d\n", i); // 2 Should be 1, no ?

   volatile int u = 0;
   u = u++ + ++u;
   printf("%d\n", u); // 1

   u = 1;
   u = (u++);
   printf("%d\n", u); // 2 Should also be one, no ?

   register int v = 0;
   v = v++ + ++v;
   printf("%d\n", v); // 3 (Should be the same as u ?)

   int w = 0;
   printf("%d %d\n", ++w, w); // shouldn't this print 1 1

   int x[2] = { 5, 8 }, y = 0;
   x[y] = y ++;
   printf("%d %d\n", x[0], x[1]); // shouldn't this print 0 8? or 5 0?
}



解答

C 有未定义行为的概念，即某些语言结构在语法上是有效的，但您无法预测代码运行时的行为。

据我所知，该标准没有明确说明 为什么 存在未定义行为的概念。在我看来，这仅仅是因为语言设计者希望在语义上有一些余地，而不是要求所有实现以完全相同的方式处理整数溢出，这很可能会带来严重的性能成本，他们只是留下了行为undefined 这样，如果您编写导致整数溢出的代码，则任何事情都可能发生。

那么，考虑到这一点，为什么会出现这些“问题”？该语言明确指出某些事情会导致未定义的行为。没有问题，不涉及“应该”。如果在声明涉及的变量之一时未定义的行为发生变化volatile，那不会证明或改变任何东西。它是 未定义的 ；你无法推理这种行为。

你最有趣的例子，那个

u = (u++);

是未定义行为的教科书示例（请参阅维基百科关于序列点的条目）。


更多建议

这里的大部分答案都引用自 C 标准，强调这些构造的行为是未定义的。要理解 为什么这些构造的行为是 undefined ，让我们首先根据 C11 标准来理解这些术语：

排序： （5.1.2.3）

给定任意两个求值A和B，如果A在 之前排序B，则 的执行A应先于 的执行B。

未排序：

如果A未在 之前或之后排序B，则A和B未排序。

评估可以是以下两种情况之一：

  • 值计算 ，计算表达式的结果；和
  • side effects ，即对象的修改。

序列点：

表达式求值之间序列点的存在A意味着与关联的B每个 值计算 和 副作用 在与关联的每个 值计算 和 副作用A之前排序。 B

现在来问这个问题，对于像这样的表达

int i = 1;
i = i++;

标准说：

6.5 表达式：

如果标量对象的副作用相对于同一标量对象的不同副作用或使用同一标量对象的值的值计算是未 排序 的 ，则 行为是未定义的 。[...]

因此，上述表达式调用 UB，因为同一对象上的两个副作用i相对于彼此是无序的。这意味着它不会按顺序分配给副作用是i在副作用之前还是之后完成++。 根据赋值发生在增量之前还是之后，将产生不同的结果，这就是undefined behavior
的情况之一。 ****

让我们重命名i赋值左边的 beil和赋值右边的（在表达式中i++） be ir，那么表达式就像

il = ir++     // Note that suffix l and r are used for the sake of clarity.
              // Both il and ir represents the same object.

关于 Postfix运算符的重要一点++是：

仅仅因为++变量出现在变量之后并不意味着增量发生得晚。 只要编译器确保使用原始值， 增量就可以在编译器喜欢的任何时候发生。

这意味着表达式il = ir++可以被评估为

temp = ir;      // i = 1
ir = ir + 1;    // i = 2   side effect by ++ before assignment
il = temp;      // i = 1   result is 1

或者

temp = ir;      // i = 1
il = temp;      // i = 1   side effect by assignment before ++
ir = ir + 1;    // i = 2   result is 2

导致两个不同的结果1，2这取决于分配的副作用顺序，++因此调用 UB。
