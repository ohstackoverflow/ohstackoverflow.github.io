Gemfile.lock 应该包含在 .gitignore 中吗？

Arya

2023-02-18 05:58

我对捆绑器及其生成的文件有点陌生。我有一份来自 GitHub 的 git repo 的副本，许多人都在贡献它，所以我很惊讶地发现 bundler 创建了一个在 repo 中不存在且不在列表中的文件.gitignore。

因为我已经 fork 了它，所以我知道将它添加到 repo 不会破坏主 repo 的任何内容，但是如果我执行 pull request，它会导致问题吗？

应该Gemfile.lock包含在存储库中吗？



解答

2021 年的简单答案： Gemfile.lock 也应该在 Rubygems 的版本控制中 。接受的答案现在是 11 岁。

这里有一些推理（从评论中挑选）：

@josevalim https://github.com/heartcombo/devise/pull/3147#issuecomment-52193788

Gemfile.lock 应该保留在存储库中，因为贡献者和开发人员应该能够分叉项目并使用保证工作的版本运行它。

@rafaelfranca https://github.com/rails/rails/pull/18951#issuecomment-74888396

我认为即使对于插件也忽略锁定文件不是一个好主意。

这意味着“git clone；bundle；rake test”序列不能保证通过，因为您的许多依赖项中的一个已升级并使您的代码中断。此外，正如@chancancode 所说，平分它变得更加困难。

Rails 在 git 中也有 Gemfile.lock：

  • https://github.com/rails/rails/commit/0ad6d27643057f2eccfe8351409a75a6d1bbb9d0


更多建议

真正的问题发生在您开发需要具有可配置数据库适配器的开源 Rails 应用程序时。我正在开发 Fat Free CRM 的 Rails 3 分支。我的偏好是 postgres，但我们希望默认数据库是 mysql2。

在这种情况下，Gemfile.lock仍然需要使用默认的 gem 集进行签入，但我需要忽略我在我的机器上对其所做的更改。为此，我运行：

git update-index --assume-unchanged Gemfile.lock

并反转：

git update-index --no-assume-unchanged Gemfile.lock

在您的Gemfile. 这将根据您的 database.yml 加载适当的数据库适配器 gem。

# Loads the database adapter gem based on config/database.yml (Default: mysql2)
# -----------------------------------------------------------------------------
db_gems = {"mysql2"     => ["mysql2", ">= 0.2.6"],
           "postgresql" => ["pg",     ">= 0.9.0"],
           "sqlite3"    => ["sqlite3"]}
adapter = if File.exists?(db_config = File.join(File.dirname(__FILE__),"config","database.yml"))
  db = YAML.load_file(db_config)
  # Fetch the first configured adapter from config/database.yml
  (db["production"] || db["development"] || db["test"])["adapter"]
else
  "mysql2"
end
gem *db_gems[adapter]
# -----------------------------------------------------------------------------

我不能说这是否是既定的最佳实践，但它对我来说效果很好。
