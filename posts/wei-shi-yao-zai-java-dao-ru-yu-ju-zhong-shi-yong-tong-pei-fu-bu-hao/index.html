为什么在 Java 导入语句中使用通配符不好？

Arya

2023-02-18 04:47

使用像这样的单个语句更加方便和干净

import java.awt.*;

而不是导入一堆单独的类

import java.awt.Panel;
import java.awt.Graphics;
import java.awt.Canvas;
...

在语句中使用通配符有什么问题import？



解答

唯一的问题是它会扰乱您的本地命名空间。例如，假设您正在编写一个 Swing 应用程序，因此需要java.awt.Event，并且还与公司的日历系统接口，该系统具有com.mycompany.calendar.Event. 如果您使用通配符方法导入两者，则会发生以下三种情况之一：

  1. java.awt.Event和之间存在彻底的命名冲突com.mycompany.calendar.Event，因此您甚至无法编译。
  2. 你实际上只导入了一个（你的两个导入中只有一个导入了.*），但它是错误的，你很难弄清楚为什么你的代码声称类型是错误的。
  3. 当你编译你的代码时，没有com.mycompany.calendar.Event，但是当他们后来添加一个时，你以前有效的代码突然停止编译。

显式列出所有导入的好处是，我一眼就能看出你打算使用哪个类，这只会让阅读代码变得更加容易。如果您只是做一件快速的一次性事情，则没有明显的 错误 ，但未来的维护者会感谢您的澄清。


更多建议

这里给 明星进口投票。import 语句旨在导入 包 ，而不是类。导入整个包要干净得多；此处确定的问题（例如java.sql.Datevs java.util.Date）很容易通过其他方式解决，并没有 真正 通过特定的进口解决，当然也不能证明对所有类别的疯狂迂腐的进口都是合理的。没有什么比打开源文件不得不翻阅 100 条导入语句更令人不安的了。

进行特定的导入会使重构更加困难；如果你删除/重命名一个类，你需要删除它的 所有 特定导入。如果将实现切换到同一包中的不同类，则必须修复导入。虽然这些额外的步骤可以自动化，但它们确实会影响生产力，并没有真正的收益。

如果默认情况下 Eclipse 不执行特定的类导入，那么每个人仍然会执行 star 导入。抱歉，进行特定进口确实没有合理的理由。

以下是处理类冲突的方法：

import java.sql.*;
import java.util.*;
import java.sql.Date;
