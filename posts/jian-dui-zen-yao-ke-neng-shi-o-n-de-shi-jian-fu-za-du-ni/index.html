哪个实现buildHeap更有效？

这两种解决方案都将生成有效的堆。不出所料，效率更高的是第二个使用siftDown.

令 h = log n 代表堆的高度。该siftDown方法所需的工作由总和给出

(0 * n/2) + (1 * n/4) + (2 * n/8) + ... + (h * 1).

总和中的每一项都是给定高度的节点必须移动的最大距离（底层为零，根为 h）乘以该高度的节点数。相反，调用siftUp每个节点的总和是

(h * n/2) + ((h-1) * n/4) + ((h-2)*n/8) + ... + (0 * 1).

应该清楚的是，第二个总和更大。仅第一项就是 hn/2 = 1/2 n log n ，因此这种方法最多具有复杂性 O(n log n) 。

我们如何证明该siftDown方法的总和确实是 O(n) ？

一种方法（还有其他分析也有效）是将有限和变成无限级数，然后使用泰勒级数。我们可以忽略第一项，它是零：

如果您不确定为什么这些步骤中的每一个都有效，这里是用文字说明该过程的理由：

  • 项都是正的，所以有限和一定小于无限和。
  • 该级数等于在x=1/2 处计算的幂级数。
  • 该幂级数等于（常数倍）泰勒级数的导数 f(x)=1/(1-x) 。
  • x=1/2 在那个泰勒级数的收敛区间内。
  • 因此，我们可以将泰勒级数替换为 1/(1-x) ，进行微分、求值，求出无穷级数的值。

由于无限总和恰好为 n ，我们得出结论，有限总和并不更大，因此为 O(n) 。

为什么堆排序需要 O(n log n) 时间？

如果可以buildHeap在线性时间内运行，为什么堆排序需要 O(n log n) 时间？好吧，堆排序包括两个阶段。首先，我们调用buildHeap数组，如果以最佳方式实现，它需要 O(n) 时间。下一阶段是重复删除堆中最大的项，并将其放在数组的末尾。因为我们从堆中删除了一个项目，所以在堆的末尾之后总是有一个空位可以存储该项目。所以堆排序通过依次移除下一个最大的项并将其放入数组中，从最后一个位置开始并向前面移动来实现排序顺序。正是这最后一部分的复杂性在堆排序中占主导地位。循环看起来像这样：

for (i = n - 1; i > 0; i--) {
    arr[i] = deleteMax();
}

显然，循环运行 O(n) 次（准确地说是 n - 1 ，最后一项已经到位）。deleteMax堆的复杂度是 O(log n) 。它通常通过删除根（堆中剩余的最大项）并用堆中的最后一项替换它来实现，这是一个叶，因此是最小的项之一。这个新根几乎肯定会违反堆属性，因此您必须调用siftDown直到将其移回可接受的位置。这也具有将下一个最大的项目向上移动到根的效果。请注意，与我们从树底部buildHeap调用的大多数节点相比，我们现在在每次迭代时都从树顶部调用！siftDown``siftDown 尽管树在收缩，但收缩得不够快 ：树的高度保持不变，直到您移除了前半部分节点（当您完全清除底层时）。然后对于下一个季度，高度为 h - 1 。所以第二阶段的总工作是

h*n/2 + (h-1)*n/4 + ... + 0 * 1.

注意开关：现在零工作案例对应于单个节点， h 工作案例对应于一半的节点。这个总和是 O(n log n)，buildHeap就像使用 siftUp 实现的低效版本一样。但在这种情况下，我们别无选择，因为我们正在尝试排序，并且我们需要接下来移除下一个最大的项目。

综上所述，堆排序的工作是两个阶段的总和： 构建堆的 O(n) 时间和 按顺序移除每个节点的时间 O(n log n) ，因此复杂度为 O(n log n)。您可以证明（使用信息论中的一些想法）对于基于比较的排序， O(n log n) 无论如何是您所希望的最好的，因此没有理由对此感到失望或期望堆排序实现O(n) 时间限制buildHeap。
