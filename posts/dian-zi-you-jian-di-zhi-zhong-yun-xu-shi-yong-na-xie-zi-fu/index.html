我不是在询问完整的电子邮件验证。

我只想知道电子​​邮件地址中允许使用的字符user- name和部分字符。server这可能过于简单化了，也许电子邮件地址可以采用其他形式，但我不在乎。我只询问这个简单的形式：（user- name@server例如 wild.wezyr@best-server-ever.com），并且在这两个部分中都允许使用字符。



解答

请参阅RFC 5322：Internet 消息格式，并在较小程度上参阅RFC 5321：简单邮件传输协议。

RFC 822也涵盖了电子邮件地址，但主要涉及其结构：

 addr-spec   =  local-part "@" domain        ; global address     
 local-part  =  word *("." word)             ; uninterpreted
                                             ; case-preserved

 domain      =  sub-domain *("." sub-domain)     
 sub-domain  =  domain-ref / domain-literal     
 domain-ref  =  atom                         ; symbolic reference

和往常一样，维基百科有一篇关于电子邮件地址的不错的文章：

电子邮件地址的本地部分可以使用以下任何 ASCII 字符：

  • 大写和小写拉丁字母AtoZ和ato z；
  • 数字0到9;
  • 特殊字符!#$%&'*+-/=?^_~； * 点.，前提是除非被引用，否则它不是第一个或最后一个字符，并且除非被引用，否则它不会连续出现（例如John..Doe@example.com不允许但"John..Doe"@example.com`被允许）；
  • 允许使用空格和"(),:;<>@[\]字符但有限制（它们只允许在带引号的字符串内，如下文所述，此外，反斜杠或双引号前面必须有反斜杠）； * 注释允许在局部部分的两端加上括号；例如john.smith(comment)@example.com和(comment)john.smith@example.com都等同于john.smith@example.com。

除了 ASCII 字符之外，自 2012 年起，您还可以使用上面的国际字符，编码为 UTF-8，如 RFC 6532 规范U+007F中所述并在维基百科上进行了解释。请注意，截至 2019 年，这些标准仍标记为已提议，但正在缓慢推出。此规范中的更改基本上将国际字符添加为有效的字母数字字符 (atext)，而不会影响允许和限制特殊字符的规则，如和。!#``@:

有关验证，请参阅使用正则表达式验证电子邮件地址。

该domain部分定义如下：

协议的 Internet 标准（征求意见稿）规定组件主机名标签只能包含 ASCII 字母athrough z（不区分大小写）、数字0through9和连字符 ( -)。RFC 952中主机名的原始规范规定标签不能以数字或连字符开头，也不能以连字符结尾。但是，后续规范 ( RFC 1123 ) 允许主机名标签以数字开头。不允许使用其他符号、标点字符或空格。


更多建议

小心！这个线程中有一堆知识腐烂（以前是真的，现在不是了）。

为了避免当前和未来世界以及来自世界任何地方的实际电子邮件地址的误报拒绝，您至少需要了解RFC 3490的高级概念，“应用程序中的国际化域名 (IDNA)” 。我知道美国和 A 国的人通常不了解这一点，但它已经在世界范围内 广泛使用并迅速增加（主要是非英语主导的部分）。

要点是您现在可以使用像 mason@日本.com 和 wildwezyr@fahrvergnügen.net 这样的地址。不，这还不能与那里的所有东西兼容（正如上面许多人所抱怨的，即使是简单的 qmail 样式 +ident 地址也经常被错误地拒绝）。但是有一个 RFC，有一个规范，它现在得到了 IETF 和 ICANN 的支持，而且 - 更重要的是 - 有越来越多的实施支持目前正在使用的改进。

在我搬回日本并开始看到像 hei@やる.ca 这样的电子邮件地址和像这样的亚马逊 URL 之前，我自己对这种发展并不太了解：

http://www.amazon.co.jp/エレクトロニクス-テジタルカマラ- ポータプルオードイ/b/ref=topnav_storetab_e?ie=UTF8&node=3210981

我知道您不想要指向规范的链接，但如果您完全依赖互联网论坛上黑客的过时知识，您的电子邮件验证器最终将拒绝非英语用户越来越希望使用的电子邮件地址。对于那些用户来说，这种验证将与我们都讨厌的常见的脑死亡形式一样烦人，这种形式无法处理 + 或三部分域名或其他任何内容。

所以我并不是说这不是一件麻烦事，而是“在某些/任何/无条件下允许”的完整字符列表是（几乎）所有语言的所有字符。如果你想“接受所有有效的电子邮件地址（也有许多无效的）”那么你必须考虑国际化域名，这基本上使基于字符的方法无用（抱歉），除非你首先转换国际化的电子邮件地址（死因为2015 年 9 月，过去是这样的——Punycode的工作替代方案在这里）。

这样做之后，您可以遵循上面的（大部分）建议。
