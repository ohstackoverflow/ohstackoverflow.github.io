解答

在 x86_64 Core2 Duo (2.5GHz) 机器上使用, ,为 Haskell 和GHC 7.0.3C编译使用。gcc 4.4.6``Linux 2.6.29``ghc -O2 -fllvm -fforce-recomp``gcc -O3 -lm

  • 你的 C 例程在 8.4 秒内运行（比你的运行速度快可能是因为-O3）
  • Haskell 解决方案在 36 秒内运行（由于标志-O2）
  • 您的factorCount'代码未明确键入并默认为Integer（感谢 Daniel 在这里纠正我的误诊！）。Int使用和时间更改为 11.1 秒 给出显式类型签名（无论如何这是标准做法） ****
  • 在factorCount'你不必要地打电话fromIntegral。修复不会导致任何更改（编译器很聪明，你很幸运）。
  • 您使用的modwhererem更快且足够。这会将时间更改为 8.5 秒 。

  • factorCount'不断应用两个永远不会改变的额外参数 ( number, sqrt)。工人/包装器转换为我们提供：

    $ time ./so 842161320

    real 0m7.954s
    user 0m7.944s
    sys 0m0.004s

没错， 7.95秒 。始终 比 C 解决方案快半秒 。没有-fllvm标志我仍然得到8.182 seconds，所以 NCG 后端在这种情况下也表现良好。

结论：Haskell 很棒。

编辑：既然我们已经探索过了，让我们来解决问题

问题 1：erlang、python 和 haskell 是否会因为使用任意长度的整数而降低速度，或者只要值小于 MAXINT 就不会降低速度？

在 Haskell 中，usingInteger比 using 慢Int，但慢多少取决于执行的计算。幸运的是（对于 64 位机器）Int就足够了。为了可移植性，您可能应该重写我的代码以使用Int64or Word64（C 不是唯一带有 a 的语言long）。

问题2：为什么haskell这么慢？是否有关闭刹车的编译器标志或者是我的实现？（后者很有可能，因为 haskell 对我来说是一本有七封印章的书。）

问题 3：您能给我一些提示，说明如何在不改变我确定因素的方式的情况下优化这些实施吗？任何方式的优化：更好、更快、更“原生”的语言。

我上面就是这么回答的。答案是

  • 0）通过使用优化-O2
  • 1) 尽可能使用快速（特别是：可拆箱）类型
  • 2）rem不是mod（经常被遗忘的优化）和
  • 3) worker/wrapper 转换（可能是最常见的优化）。

问题 4：我的功能实现是否允许 LCO，从而避免将不必要的帧添加到调用堆栈？

是的，那不是问题所在。干得好，很高兴你考虑到了这一点。


更多建议

Erlang 实现存在一些问题。作为以下基准，我测量的未修改 Erlang 程序的执行时间为 47.6 秒，而 C 代码为 12.7 秒。

（编辑：自Erlang/OTP version 24, 2021起，Erlang有一个自动的JIT编译器，+native不再支持或不需要旧的编译器选项。我把下面这段话留作历史文档没有改动。关于的言论还是export_all有些对 JIT 生成良好代码的能力有效。）

如果你想运行计算密集型的 Erlang 代码，你应该做的第一件事就是使用本地代码。编译erlc +native euler12时间减少到 41.3 秒。然而，这比对此类代码进行本机编译的预期速度要低得多（仅 15%），问题在于您对-compile(export_all). 这对实验很有用，但所有函数都可能从外部访问的事实导致本机编译器非常保守。（普通的 BEAM 仿真器并没有受到太大影响。）将此声明替换为-export([solve/0]).可以提供更好的加速：31.5 秒（比基线快 35%）。

但是代码本身有一个问题：对于factorCount 循环中的 每次迭代，您都执行此测试：

factorCount (_, Sqrt, Candidate, Count) when Candidate == Sqrt -> Count + 1;

C 代码不执行此操作。通常，在相同代码的不同实现之间进行公平比较可能会很棘手，特别是如果算法是数字的，因为您需要确保它们实际上在做同样的事情。由于某处的某些类型转换，一个实现中的轻微舍入错误可能导致它比另一个实现更多的迭代，即使两者最终达到相同的结果。

为了消除这种可能的错误来源（并摆脱每次迭代中的额外测试），我重写了 factorCount 函数，如下所示，并严格仿照 C 代码：

factorCount (N) ->
    Sqrt = math:sqrt (N),
    ISqrt = trunc(Sqrt),
    if ISqrt == Sqrt -> factorCount (N, ISqrt, 1, -1);
       true          -> factorCount (N, ISqrt, 1, 0)
    end.

factorCount (_N, ISqrt, Candidate, Count) when Candidate > ISqrt -> Count;
factorCount ( N, ISqrt, Candidate, Count) ->
    case N rem Candidate of
        0 -> factorCount (N, ISqrt, Candidate + 1, Count + 2);
        _ -> factorCount (N, ISqrt, Candidate + 1, Count)
    end.

这种重写、非export_all本地编译为我提供了以下运行时间：

$ erlc +native euler12.erl
$ time erl -noshell -s euler12 solve
842161320

real    0m19.468s
user    0m19.450s
sys 0m0.010s

与 C 代码相比，这还算不错：

$ time ./a.out 
842161320

real    0m12.755s
user    0m12.730s
sys 0m0.020s

考虑到 Erlang 根本不适合编写数字代码，在这样的程序上只比 C 慢 50% 就已经很不错了。

最后，关于你的问题：

问题 1：erlang、python 和 haskell 是否会因为使用任意长度的整数而降低速度，或者只要值小于 MAXINT 就不会降低速度？

是的，有点。在 Erlang 中，没有办法说“使用环绕式 32/64 位算术”，因此除非编译器可以证明整数的某些界限（通常不能），否则它必须检查所有计算以了解如果它们可以放入单个标记词中，或者是否必须将它们变成堆分配的 bignums。即使在运行时实践中从未使用过大数字，也必须执行这些检查。另一方面，这意味着您 知道 如果您突然给算法提供比以前更大的输入，算法将永远不会因为意外的整数回绕而失败。

问题 4：我的功能实现是否允许 LCO，从而避免将不必要的帧添加到调用堆栈？

是的，您的 Erlang 代码在上次调用优化方面是正确的。
