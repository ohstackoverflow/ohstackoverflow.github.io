在 Intel CPU 上用 _mm_popcnt_u64 将 32 位循环计数器替换为 64 位会引入疯狂的性能偏差

Arya

2023-02-17 01:15

我一直在寻找处理popcount大量数据的最快方法。我遇到了一个 非常奇怪的 效果：将循环变量从更改unsigned为uint64_t使我的 PC 上的性能下降了 50%。

基准

#include <iostream>
#include <chrono>
#include <x86intrin.h>

int main(int argc, char* argv[]) {

    using namespace std;
    if (argc != 2) {
       cerr << "usage: array_size in MB" << endl;
       return -1;
    }

    uint64_t size = atol(argv[1])<<20;
    uint64_t* buffer = new uint64_t[size/8];
    char* charbuffer = reinterpret_cast<char*>(buffer);
    for (unsigned i=0; i<size; ++i)
        charbuffer[i] = rand()%256;

    uint64_t count,duration;
    chrono::time_point<chrono::system_clock> startP,endP;
    {
        startP = chrono::system_clock::now();
        count = 0;
        for( unsigned k = 0; k < 10000; k++){
            // Tight unrolled loop with unsigned
            for (unsigned i=0; i<size/8; i+=4) {
                count += _mm_popcnt_u64(buffer[i]);
                count += _mm_popcnt_u64(buffer[i+1]);
                count += _mm_popcnt_u64(buffer[i+2]);
                count += _mm_popcnt_u64(buffer[i+3]);
            }
        }
        endP = chrono::system_clock::now();
        duration = chrono::duration_cast<std::chrono::nanoseconds>(endP-startP).count();
        cout << "unsigned\t" << count << '\t' << (duration/1.0E9) << " sec \t"
             << (10000.0*size)/(duration) << " GB/s" << endl;
    }
    {
        startP = chrono::system_clock::now();
        count=0;
        for( unsigned k = 0; k < 10000; k++){
            // Tight unrolled loop with uint64_t
            for (uint64_t i=0;i<size/8;i+=4) {
                count += _mm_popcnt_u64(buffer[i]);
                count += _mm_popcnt_u64(buffer[i+1]);
                count += _mm_popcnt_u64(buffer[i+2]);
                count += _mm_popcnt_u64(buffer[i+3]);
            }
        }
        endP = chrono::system_clock::now();
        duration = chrono::duration_cast<std::chrono::nanoseconds>(endP-startP).count();
        cout << "uint64_t\t"  << count << '\t' << (duration/1.0E9) << " sec \t"
             << (10000.0*size)/(duration) << " GB/s" << endl;
    }

    free(charbuffer);
}

解答

罪魁祸首：虚假数据依赖 （编译器甚至都不知道）

在 Sandy/Ivy Bridge 和 Haswell 处理器上，指令：

popcnt  src, dest

似乎对目标寄存器有错误的依赖性dest。即使指令只写入它，指令也会等到dest准备就绪后再执行。这种错误的依赖关系（现在）被英特尔记录为勘误表HSD146 (Haswell)和SKL029 (Skylake)

Skylake 为lzcnt和tzcnt修复了这个问题。
Cannon Lake（和 Ice Lake）为popcnt.
bsf/bsr具有真正的输出依赖性：输入 = 0 时输出未修改。（但是没有办法利用内在函数来利用它——只有 AMD 记录它并且编译器不公开它。）

（是的，这些指令都运行在同一个执行单元上）。

这种依赖性不仅会阻止popcnt单个循环迭代中的 4 s。它可以进行循环迭代，使处理器无法并行化不同的循环迭代。

对比和其他调整不会直接影响问题unsigned。uint64_t但是它们会影响将寄存器分配给变量的寄存器分配器。

在您的情况下，速度是粘附在（错误的）依赖链上的直接结果，具体取决于寄存器分配器决定做什么。

  • 13 GB/s 有一条链：popcnt- add- popcnt- popcnt→ 下一次迭代
  • 15 GB/s 有一条链：popcnt- add- popcnt- add→ 下一次迭代
  • 20 GB/s 有一条链：popcnt- popcnt→ 下一次迭代
  • 26 GB/s 有一条链：popcnt- popcnt→ 下一次迭代

20 GB/s 和 26 GB/s 之间的差异似乎是间接寻址的一个小瑕疵。无论哪种方式，一旦达到此速度，处理器就会开始遇到其他瓶颈。

那么编译器出了什么问题呢？

似乎 GCC 和 Visual Studio 都不知道popcnt有这样的错误依赖。然而，这些错误的依赖关系并不少见。这只是编译器是否意识到这一点的问题。

popcnt不完全是最常用的指令。因此，一个主要的编译器可能会遗漏这样的东西也就不足为奇了。似乎也没有任何地方提到这个问题的文档。如果英特尔不公开，那么在有人偶然碰到它之前，外面的人都不会知道。

（ 更新： 从 4.9.2 版开始，GCC 意识到这种错误依赖，并在启用优化时生成代码来补偿它。来自其他供应商的主要编译器，包括 Clang、MSVC，甚至英特尔自己的 ICC 还没有意识到这个微架构错误并且不会发出补偿它的代码。）

为什么 CPU 会有这样的错误依赖？

我们bsf可以推测：它运行在与/bsr具有输出依赖性的相同执行单元上。（POPCNT在硬件中是如何实现的？）。对于这些指令，Intel 将 input=0 的整数结果记录为“未定义”（ZF=1），但 Intel 硬件实际上提供了更强的保证以避免破坏旧软件：输出未修改。AMD 记录了这种行为。 __

据推测，根据输出为这个执行单元制作一些微指令是不方便的，而其他的则不是。

AMD 处理器似乎没有这种错误的依赖性。

我编写了一个等效的 C 程序来进行实验，我可以确认这种奇怪的行为。更重要的是，gcc相信 64 位整数（无论如何应该是size_t......）更好，因为 usinguint_fast32_t导致 gcc 使用 64 位 uint。

我对汇编做了一些改动：
只需采用 32 位版本，在程序的内部 popcount 循环中将所有 32 位指令/寄存器替换为 64 位版本。观察：代码 与 32 位版本一样快！

这显然是一个 hack，因为变量的大小并不是真正的 64 位，因为程序的其他部分仍然使用 32 位版本，但只要内部 popcount-loop 控制性能，这是一个好的开始.

然后，我从程序的 32 位版本复制了内循环代码，将其破解为 64 位，修改了寄存器以使其替代 64 位版本的内循环。 此代码的运行速度也与 32 位版本一样快。

我的结论是，这是编译器错误的指令调度，而不是 32 位指令的实际速度/延迟优势。

（警告：我破解了程序集，可能会在不注意的情况下破坏某些东西。我不这么认为。）
