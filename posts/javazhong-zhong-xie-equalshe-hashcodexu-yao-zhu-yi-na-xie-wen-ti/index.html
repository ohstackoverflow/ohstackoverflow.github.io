equals覆盖and时必须考虑哪些问题/陷阱hashCode？



解答

理论（对于语言律师和数学倾向者）：

equals()( javadoc ) 必须定义等价关系（它必须是 自反的 、 对称的 和 传递的 ）。此外，它必须是 一致的 （如果对象没有被修改，那么它必须保持返回相同的值）。此外，o.equals(null)必须始终返回 false。

hashCode()( javadoc ) 也必须 一致 （如果对象未根据 修改equals()，则它必须保持返回相同的值）。

两种方法之间的关系 是：

无论何时a.equals(b)，则a.hashCode()必须与 相同b.hashCode()。

在实践中：

如果你覆盖了一个，那么你应该覆盖另一个。

使用您用于计算的同一组字段equals()来计算hashCode()。

使用来自Apache Commons Lang库的优秀帮助程序类EqualsBuilder和HashCodeBuilder。一个例子：

public class Person {
    private String name;
    private int age;
    // ...

    @Override
    public int hashCode() {
        return new HashCodeBuilder(17, 31). // two randomly chosen prime numbers
            // if deriving: appendSuper(super.hashCode()).
            append(name).
            append(age).
            toHashCode();
    }

    @Override
    public boolean equals(Object obj) {
       if (!(obj instanceof Person))
            return false;
        if (obj == this)
            return true;

        Person rhs = (Person) obj;
        return new EqualsBuilder().
            // if deriving: appendSuper(super.equals(obj)).
            append(name, rhs.name).
            append(age, rhs.age).
            isEquals();
    }
}

还要记住：

使用基于散列的集合或映射（例如HashSet、LinkedHashSet、HashMap、Hashtable或WeakHashMap ）时，请确保您放入集合的关键对象的 hashCode() 在对象位于集合中时永远不会更改。确保这一点的可靠方法是使您的密钥不可变，这还有其他好处。


更多建议

如果您正在处理使用 Hibernate 之类的对象关系映射器 (ORM) 持久化的类，如果您认为这已经不合理地复杂了，那么有一些问题值得注意！

延迟加载的对象是子类

如果您的对象是使用 ORM 持久化的，那么在许多情况下您将处理动态代理以避免过早地从数据存储中加载对象。这些代理作为您自己类的子类实现。这意味着this.getClass() == o.getClass()将会回归false。例如：

Person saved = new Person("John Doe");
Long key = dao.save(saved);
dao.flush();
Person retrieved = dao.retrieve(key);
saved.getClass().equals(retrieved.getClass()); // Will return false if Person is loaded lazy

如果您正在处理 ORM，那么 usingo instanceof Person是唯一能够正确运行的东西。

延迟加载的对象具有空字段

ORM 通常使用 getter 来强制加载延迟加载的对象。这意味着person.name如果null是person延迟加载，即使person.getName()强制加载并返回“John Doe”也是如此。根据我的经验，这种情况更常出现在hashCode()和中equals()。

如果您正在处理 ORM，请确保始终使用 getter，并且永远不要在hashCode()和中使用字段引用equals()。

保存一个对象会改变它的状态

持久对象通常使用一个id字段来保存对象的键。首次保存对象时，该字段将自动更新。不要在hashCode(). 但是你可以在equals().

我经常使用的模式是

if (this.getId() == null) {
    return this == other;
}
else {
    return this.getId().equals(other.getId());
}

但是：你不能包含getId()在hashCode(). 如果这样做，当对象被持久化时，它会hashCode发生变化。如果该对象在 a 中HashSet，您将“再也不会”找到它。

在我的Person示例中，我可能会使用getName()forhashCode和getId()plus getName()（只是为了偏执狂）for equals()。如果 有一些“碰撞”的风险，那没关系hashCode()，但永远不会equals()。

hashCode() 应该使用不变的属性子集equals()
