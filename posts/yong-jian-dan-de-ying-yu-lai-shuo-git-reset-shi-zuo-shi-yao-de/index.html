不幸的是，我读得越多，就越显得我不完全理解它。我来自 SVN 背景，而 Git 是一个全新的范式。我很容易就掌握了 mercurial，但 Git 更具技术性。

我认为git reset接近hg revert，但似乎有差异。

那么具体是git reset做什么的呢？请包括以下方面的详细解释：

  • 选项--hard,--soft和--merge;
  • 您使用的奇怪符号，HEAD例如HEAD^and HEAD~1；

  • 对工作副本的影响，HEAD以及您的全局压力水平。



解答

一般来说，git reset的功能是获取当前分支并将其重置为指向其他地方，并且可能会带来索引和工作树。更具体地说，如果您的主分支（当前签出）是这样的：

- A - B - C (HEAD, master)

并且您意识到您希望 master 指向 B，而不是 C，您将使用git reset B将其移动到那里：

- A - B (HEAD, master)      # - C is still here, but there's no branch pointing to it anymore

题外话：这跟结帐不一样。如果你运行git checkout B，你会得到这个：

- A - B (HEAD) - C (master)

您最终处于分离的 HEAD 状态。HEAD，工作树，索引全部匹配B，但主分支留在后面C。D如果你在此时进行新的提交，你会得到这个，这可能不是你想要的：

- A - B - C (master)
       \
        D (HEAD)

请记住，reset 不会进行提交，它只是更新一个分支（它是指向提交的指针）以指向不同的提交。其余的只是索引和工作树发生的情况的详细信息。

用例

git reset我将在下一节中对各种选项的描述中涵盖许多主要用例。它真的可以用于各种各样的事情；共同点是它们都涉及重置分支、索引和/或工作树以指向/匹配给定的提交。

注意事项

  • --hard会导致你真的失去工作。它会修改您的工作树。

  • git reset [options] commit可能会导致您（某种程度上）丢失提交。在上面的玩具示例中，我们丢失了 commit C。它仍在回购协议中，您可以通过查看git reflog show HEAD或找到它git reflog show master，但实际上不再可以从任何分支访问它。

  • Git 会在 30 天后永久删除此类提交，但在此之前您可以通过再次将分支指向它来恢复 C ( git checkout C; git branch <new branch name>)。

参数

解释手册页，最常见的用法是 form git reset [<commit>] [paths...]，它将给定路径重置为给定提交的状态。如果未提供路径，则重置整个树，如果未提供提交，则将其视为 HEAD（当前提交）。这是跨 git 命令的常见模式（例如 checkout、diff、log，尽管确切的语义有所不同），所以这应该不足为奇。

例如，git reset other-branch path/to/foo将 path/to/foo 中的所有内容重置为其在其他分支中的状态，git reset -- .将当前目录重置为其在 HEAD 中的状态，以及一个简单的git reset将所有内容重置为其在 HEAD 中的状态。

主要工作树和索引选项

有四个主要选项可以控制重置期间工作树和索引发生的情况。

记住，索引是 git 的“暂存区”——当你准备提交时，它就是git add事情发生的地方。

  • --hard使一切都与您重置​​为的提交相匹配。这可能是最容易理解的。您所有的本地更改都会受到破坏。一个主要用途是吹走你的工作而不是切换提交：git reset --hard意味着git reset --hard HEAD，即不更改分支但摆脱所有本地更改。另一种是简单地将一个分支从一个地方移动到另一个地方，并保持索引/工作树同步。 这是真正可以让你失去工作的那个，因为它修改了你的工作树。 在运行任何reset --hard.

  • --mixed是默认值，即git reset意味着git reset --mixed. 它重置索引，但不重置工作树。这意味着您的所有文件都完好无损，但是原始提交与您重置为的提交之间的任何差异都将显示为具有 git 状态的本地修改（或未跟踪的文件）。当你意识到你做了一些错误的提交时使用它，但是你想保留你所做的所有工作以便你可以修复它并重新提交。为了提交，您必须再次将文件添加到索引 ( git add ...)。

  • --soft不触及索引 或 工作树。您的所有文件都与 一样完好无损--mixed，但所有更改都显示为changes to be committedgit 状态（即已签入以准备提交）。当您意识到自己做了一些错误的提交时使用它，但工作一切都很好 - 您需要做的就是以不同的方式重新提交。索引未受影响，因此您可以根据需要立即提交 - 生成的提交内容将与重置前的内容完全相同。

  • --merge最近添加，旨在帮助您中止失败的合并。这是必要的，因为git merge实际上会让您尝试与脏工作树（具有本地修改的工作树）合并，只要这些修改位于不受合并影响的文件中。git reset --merge重置索引（比如--mixed- 所有更改都显示为本地修改），并重置受合并影响的文件，但不影响其他文件。这有望将一切恢复到错误合并之前的状态。您通常将其用作git reset --merge（意思是git reset --merge HEAD），因为您只想重置合并，而不是实际移动分支。（HEAD尚未更新，因为合并失败）

更具体地说，假设您已经修改了文件 A 和 B，并且您试图在修改了文件 C 和 D 的分支中合并。由于某种原因合并失败，您决定中止合并。你用git reset --merge。它使 C 和 D 恢复到它们在 中的状态HEAD，但只留下您对 A 和 B 的修改，因为它们不是尝试合并的一部分。

奇怪的符号

您提到的“奇怪符号”（HEAD^和HEAD~1）只是指定提交的简写，而不必使用像3ebe3f6. 它在git-rev-parse 的手册页的“指定修订”部分中有完整的记录，有很多示例和相关语法。插入符号和代字号实际上意味着不同的东西：

  • HEAD~``HEAD~1是commit's first parent 的缩写，表示提交的第一个父级。HEAD~2表示提交的第一个父级的第一个父级。将其视为HEAD~n“HEAD 之前的 n 次提交”或“HEAD 的第 n 代祖先”。
  • HEAD^(or HEAD^1) 也表示提交的第一个父级。HEAD^2表示提交的 第二个 父级。请记住，正常的合并提交有两个父项——第一个父项是合并到的提交，第二个父项是被合并的提交。一般来说，合并实际上可以有任意多个父项（章鱼合并）。

更多建议

请记住，git您有：

  • 指针 _HEAD ，_它告诉您正在处理的提交
  • 工作 树 ，代表系统上文件的状态
  • 暂存 区 （也称为 index ），它“暂存”更改，以便以后可以将它们一起提交

请包括以下方面的详细解释：

--hard,--soft和--merge;

按照危险性递增的顺序：

  • --soft移动HEAD但不接触暂存区或工作树。
  • --mixed移动HEAD和更新暂存区，但不移动和更新工作树。
  • --merge移动HEAD，重置暂存区，并尝试将工作树中的所有更改移动到新的工作树中。
  • --hard移动HEAD 并 调整你的暂存区和工作树到新的HEAD，扔掉所有东西。

具体用例和工作流程；

  • --soft当你想移动到另一个提交并在不“失去你的位置”的情况下修补东西时使用。你很少需要这个。

--

# git reset --soft example
touch foo                            // Add a file, make some changes.
git add foo                          // 
git commit -m "bad commit message"   // Commit... D'oh, that was a mistake!
git reset --soft HEAD^               // Go back one commit and fix things.
git commit -m "good commit"          // There, now it's right.

--

  • --mixed当您想查看另一次提交时的样子，但又不想丢失已有的任何更改时，请使用（这是默认设置）。

  • --merge当你想移动到一个新的地方但将你已经拥有的更改合并到工作树中时使用。

  • 用于--hard清除所有内容并在新提交时重新开始。
