“编程到接口”是什么意思？

Arya

2023-02-17 09:53

我看到这个被提到过几次，但我不清楚它是什么意思。你什么时候以及为什么要这样做？

我知道接口的作用，但我不清楚这一点让我觉得我错过了正确使用它们的机会。

如果您要这样做，是否就是这样：

IInterface classRef = new ObjectWhatever()

您可以使用任何实现IInterface? 你什么时候需要这样做？我唯一能想到的是，如果你有一个方法并且你不确定除了它实现之外将传递什么对象IInterface。我无法想象您需要多久这样做一次。

另外，你怎么能写一个方法来接受一个实现接口的对象呢？那可能吗？



解答

对于这个问题，这里有一些很好的答案，这些问题涉及关于接口和松散耦合代码、控制反转等各种非常详细的信息。有一些相当令人兴奋的讨论，所以我想借此机会稍微分解一下以了解为什么接口有用。

当我第一次接触界面时，我也对它们的相关性感到困惑。我不明白你为什么需要它们。如果我们使用像 Java 或 C# 这样的语言，我们就已经有了继承，我将接口视为一种 较弱的 继承形式，我想，“何必呢？” 在某种意义上我是对的，你可以将接口视为一种弱继承形式，但除此之外，我最终理解了它们作为一种语言结构的用途，将它们视为一种对共同特征或表现出的行为进行分类的方法。可能有许多不相关的对象类。

例如——假设您有一个 SIM 游戏并且有以下类：

class HouseFly inherits Insect {
    void FlyAroundYourHead(){}
    void LandOnThings(){}
}

class Telemarketer inherits Person {
    void CallDuringDinner(){}
    void ContinueTalkingWhenYouSayNo(){}
}

显然，这两个对象在直接继承方面没有任何共同点。但是，你可以说他们都很烦人。

假设我们的游戏需要一些随机的 东西 来惹恼玩家吃晚饭。这可能是一个HouseFly或一个Telemarketer或两者兼而有之——但是您如何允许两者都具有单一功能？你如何要求每种不同类型的对象以相同的方式“做他们讨厌的事情”？

要意识到的关键是 aTelemarketer和HouseFly共享一个共同的松散解释行为，即使它们在建模方面完全不同。因此，让我们创建一个两者都可以实现的接口：

interface IPest {
    void BeAnnoying();
}

class HouseFly inherits Insect implements IPest {
    void FlyAroundYourHead(){}
    void LandOnThings(){}

    void BeAnnoying() {
        FlyAroundYourHead();
        LandOnThings();
    }
}

class Telemarketer inherits Person implements IPest {
    void CallDuringDinner(){}
    void ContinueTalkingWhenYouSayNo(){}

    void BeAnnoying() {
        CallDuringDinner();
        ContinueTalkingWhenYouSayNo();
    }
}

我们现在有两个类，每个类都可能以自己的方式令人讨厌。而且它们不需要派生自相同的基类并共享共同的固有特征——它们只需要满足契约IPest——契约很简单。你只需要BeAnnoying。对此，我们可以建模如下：

class DiningRoom {

    DiningRoom(Person[] diningPeople, IPest[] pests) { ... }

    void ServeDinner() {
        when diningPeople are eating,

        foreach pest in pests
        pest.BeAnnoying();
    }
}

这里我们有一个餐厅，可以容纳一些食客和一些害虫——注意界面的使用。这意味着在我们的小世界里，pests数组的一个成员实际上可以是一个Telemarketer对象，也可以是一个HouseFly对象。

该ServeDinner方法在晚餐供应时调用，我们在餐厅的人应该吃饭。在我们的小游戏中，那是我们的害虫工作的时候——每只害虫都被指示通过界面来烦人IPest。通过这种方式，我们可以很容易地同时拥有这两者Telemarketers，HouseFlys并且以他们自己的方式惹恼我们——我们只关心我们在对象中有什么东西DiningRoom是害虫，我们并不真正关心它是什么，他们可以什么都没有与其他共同。

这个非常人为的伪代码示例（拖延的时间比我预期的要长得多）只是为了说明在我们何时可以使用接口方面最终为我打开灯的那种东西。我提前为这个例子的愚蠢道歉，但希望它能帮助你理解。而且，可以肯定的是，您在这里收到的其他已发布答案确实涵盖了当今在设计模式和开发方法中使用接口的范围。


更多建议

我以前给学生的具体例子是他们应该写

List myList = new ArrayList(); // programming to the List interface

代替

ArrayList myList = new ArrayList(); // this is bad

这些在一个简短的程序中看起来完全一样，但如果你在你的程序中继续使用myList100 次，你就会开始看到不同之处。第一个声明确保您只调用接口myList定义的方法List（因此没有ArrayList特定方法）。如果您以这种方式对接口进行编程，稍后您可以决定您是否真的需要

List myList = new TreeList();

你只需要在那个地方改变你的代码。您已经知道，您的其余代码不会执行任何会因更改实现而被破坏的事情， 因为 您是针对 接口编程 的。

当您谈论方法参数和返回值时，好处更加明显（我认为）。以此为例：

public ArrayList doSomething(HashMap map);

该方法声明将您绑定到两个具体实现（ArrayList和HashMap）。一旦从其他代码调用该方法，对这些类型的任何更改可能意味着您也将不得不更改调用代码。最好对接口进行编程。

public List doSomething(Map map);

现在你返回什么样的，或者作为参数传入List什么样的都无关紧要了。Map您在方法内部所做的更改doSomething不会强制您更改调用代码。
