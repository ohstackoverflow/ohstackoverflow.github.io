为什么不继承List<T>呢？

Arya

2023-02-17 00:52

在计划我的程序时，我经常从这样的思路开始：

足球队只是足球运动员的名单。因此，我应该用：

var football_team = new List<FootballPlayer>();

此列表的顺序代表球员在花名册中列出的顺序。

但后来我意识到，除了单纯的球员名单之外，球队还有其他必须记录的属性。比如本赛季的总成绩，目前的预算，球衣的颜色，string代表球队名字的a等等。

那么我认为：

好吧，一个足球队就像一个球员列表，但此外，它还有一个名字 (a string) 和总得分 (an int)。.NET 不提供用于存储足球队的类，因此我将创建自己的类。最相似和最相关的现有结构是List<FootballPlayer>，因此我将继承它：

class FootballTeam : List<FootballPlayer>
{
    public string TeamName;
    public int RunningTotal
}

但事实证明，一条准则说你不应该继承自List<T>. 我在两个方面对这个指南感到非常困惑。

为什么不？

显然List以某种方式针对性能进行了优化。为何如此？如果我扩展，我会导致什么性能问题List？究竟什么会坏？

我看到的另一个原因是它List由 Microsoft 提供，我无法控制它，所以在公开“公共 API”之后我无法更改它。但我很难理解这一点。什么是公共 API，我为什么要关心？如果我当前的项目没有也不太可能有这个公共 API，我可以放心地忽略这个指南吗？如果我确实继承自List 并且 结果证明我需要一个公共 API，我会有什么困难？

为什么这很重要？列表就是列表。什么可能会改变？我可能想要改变什么？

最后，如果 Microsoft 不想让我继承自List，他们为什么不创建这个类sealed？

我还应该使用什么？

显然，对于自定义集合，Microsoft 提供了一个Collection应该扩展的类，而不是List. 但是这个类很空，并没有很多有用的东西，比如AddRange，例如。jvitor83 的回答为该特定方法提供了性能原理，但是慢怎么AddRange不比没有好呢AddRange？

继承自Collection比继承自要多得多List，而且我看不出有什么好处。微软肯定不会无缘无故地让我做额外的工作，所以我不禁觉得我不知何故误解了什么，继承Collection实际上不是解决我问题的正确方法。

我看过诸如实施IList. 就是不行。这是几十行样板代码，我一无所获。

最后，一些人建议将其包装List在一些东西中：

class FootballTeam 
{ 
    public List<FootballPlayer> Players; 
}

这有两个问题：

  1. 它使我的代码不必要地冗长。我现在必须打电话，my_team.Players.Count而不仅仅是my_team.Count. 值得庆幸的是，使用 C# 我可以定义索引器以使索引变得透明，并转发内部的所有方法List……但那是很多代码！我做这些工作能得到什么？

  2. 这只是简单的没有任何意义。足球队没有球员名单。这 是 球员名单。您不会说“John McFootballer 加入了 SomeTeam 的球员”。你说“约翰加入了 SomeTeam”。您不向“字符串的字符”添加字母，而是向字符串添加字母。您不是将一本书添加到图书馆的图书中，而是将一本书添加到图书馆中。

我意识到“幕后”发生的事情可以说是“将 X 添加到 Y 的内部列表”，但这似乎是一种非常违反直觉的思考世界的方式。

我的问题（总结）

表示数据结构的正确 C# 方法是什么，“逻辑上”（也就是说，“对人的思想”）只是带有list一些things花里胡哨的数据结构？

继承自List<T>总是不可接受的吗？什么时候可以接受？为什么/为什么不？程序员在决定是否继承时必须考虑什么List<T>？



解答

这里有一些很好的答案。我想补充以下几点。

表示数据结构的正确 C# 方法是什么，“逻辑上”（也就是说，“对人的思维”）只是一些带有一些花里胡哨的事物的列表？

请任意十个熟悉足球存在的非计算机程序员填空：

足球队是一种特殊的_____

有 没有人 说“带有一些花里胡哨的足球运动员名单”，或者他们都说“运动队”或“俱乐部”或“组织”？你认为足球队是 一种特殊类型的球员名单 存在于你的人类思想中，而且只存在于你的人类思想中。

List<T>是一种 机制 。足球队是一个 业务对象 ——也就是说，一个代表程序 业务领域中的某些概念的对象。 不要混合那些！足球队 是一种 球队；它 有一个 花名册，一个花名册 是一个球员列表 。花名册不是 特定类型的球员名单 。花名册 是 玩家列表。所以创建一个名为Rostera 的属性List<Player>。并在你做的时候做ReadOnlyList<Player>，除非你相信每个了解足球队的人都可以从名单中删除球员。

继承自List<T>总是不可接受的吗？

谁不能接受？我？不。

更新

我想了想，还有另一个原因不将足球队建模为球员名单。事实上，将一支足球队建模为也 有 球员名单可能不是一个好主意。拥有球员名单的球队的问题在于，你得到的只是球队 在某一时刻的 快照 。我不知道你的这个课程的商业案例是什么，但如果我有一个代表足球队的课程，我想问它这样的问题：“2003 年到 2013 年间有多少海鹰队球员因伤缺席比赛？” 或者“之前为另一支球队效力的丹佛球员的跑码数同比增幅最大？” 或者“今年小猪队一路走好吗？ ” __

也就是说，在我看来，一支足球队可以很好地建模为 历史事实的集合 ，例如一名球员何时被招募、受伤、退役等。显然，目前的球员名单是一个重要的事实，可能应该是最重要的——中心，但您可能还想用这个对象做其他有趣的事情，需要更多的历史视角。


更多建议

哇，你的帖子有一大堆问题和要点。您从 Microsoft 得到的大部分推理都恰到好处。让我们从关于的一切开始List<T>

  • List<T> 高度 优化。它的主要用途是用作对象的私有成员。
  • Microsoft 没有将其密封，因为有时您可能希望创建一个名称更友好的类：class MyList<T, TX> : List<CustomObject<T, Something<TX>> { ... }. 现在就像做一样简单var list = new MyList<int, string>();。
  • CA1002：不要公开通用列表：基本上，即使您打算将此应用程序作为唯一开发人员使用，也值得使用良好的编码实践进行开发，因此它们会灌输到您和第二天性中。IList<T>如果您需要任何消费者拥有索引列表，您仍然可以将列表公开为 an 。这使您可以稍后更改类中的实现。
  • 微软之所以Collection<T>非常通用，是因为它是一个通用的概念……名字说明了一切；它只是一个集合。有更精确的版本，例如SortedCollection<T>、ObservableCollection<T>、ReadOnlyCollection<T>等。每个版本都实现IList<T>但 不 List<T>实现。
  • Collection<T>允许覆盖成员（即添加、删除等），因为它们是虚拟的。List<T>才不是。
  • 你问题的最后一部分是正确的。足球队不仅仅是一个球员列表，因此它应该是一个包含该球员列表的类。思考组合与继承。足球队 有 球员名单（花名册），但不是球员名单。

如果我正在编写这段代码，该类可能看起来像这样：

public class FootballTeam<T>//generic class
{
    // Football team rosters are generally 53 total players.
    private readonly List<T> _roster = new List<T>(53);

    public IList<T> Roster
    {
        get { return _roster; }
    }

    // Yes. I used LINQ here. This is so I don't have to worry about
    // _roster.Length vs _roster.Count vs anything else.
    public int PlayerCount
    {
        get { return _roster.Count(); }
    }

    // Any additional members you want to expose/wrap.
}
