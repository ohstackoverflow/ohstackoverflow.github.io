为什么打印“B”比打印“#”慢得多？

Arya

2023-02-16 20:19

我生成了1000x的两个矩阵1000：

第一个矩阵：O和#。
第二个矩阵：O和B。

使用以下代码，第一个矩阵需要 8.52 秒才能完成：

Random r = new Random();
for (int i = 0; i < 1000; i++) {
    for (int j = 0; j < 1000; j++) {
        if(r.nextInt(4) == 0) {
            System.out.print("O");
        } else {
            System.out.print("#");
        }
    }

   System.out.println("");
 }

使用此代码，第二个矩阵需要 259.152 秒才能完成：

Random r = new Random();
for (int i = 0; i < 1000; i++) {
    for (int j = 0; j < 1000; j++) {
        if(r.nextInt(4) == 0) {
            System.out.print("O");
        } else {
            System.out.print("B"); //only line changed
        }
    }

    System.out.println("");
}

截然不同的运行时间背后的原因是什么？

正如评论中所建议的那样，打印只System.out.print("#");需要7.8871几秒钟，而System.out.print("B");给出still printing....

正如其他人指出它对他们正常工作一样，我尝试了Ideone.com，例如，两段代码以相同的速度执行。

测试条件：

  • 我从 Netbeans 7.2 运行了这个测试，输出到它的控制台
  • 我用来System.nanoTime()测量



解答

纯粹的猜测 是您使用的终端尝试进行文字换行而不是字符换行，并将其B视为单词字符而不#是非单词字符。因此，当它到达一行的末尾并寻找一个可以断行的地方时，它#几乎会立即看到并愉快地在那里断行；而对于B，它必须继续搜索更长时间，并且可能有更多文本要换行（这在某些终端上可能很昂贵，例如，输出退格键，然后输出空格以覆盖被换行的字母）。

但这纯粹是猜测。


更多建议

我在 Eclipse 和 Netbeans 8.0.2 上进行了测试，两者都使用 Java 版本 1.8；我用来System.nanoTime()测量。

蚀：

我 在这两种情况下得到了相同的时间 ——大约 1.564 秒 。

网豆：

  • 使用“#”： 1.536 秒
  • 使用“B”： 44.164 秒

因此，看起来 Netbeans 在打印到控制台时性能不佳。

经过更多研究，我意识到问题是Netbeans 的最大缓冲区的换行（它不限于System.out.println命令），由这段代码演示：

for (int i = 0; i < 1000; i++) {
    long t1 = System.nanoTime();
    System.out.print("BBB......BBB"); \\<-contain 1000 "B"
    long t2 = System.nanoTime();
    System.out.println(t2-t1);
    System.out.println("");
}

每次迭代的时间结果小于 1 毫秒，但 每五次迭代 除外，当时时间结果约为 225 毫秒。类似（以纳秒为单位）：

BBB...31744
BBB...31744
BBB...31744
BBB...31744
BBB...226365807
BBB...31744
BBB...31744
BBB...31744
BBB...31744
BBB...226365807
.
.
.

等等..

概括：

  1. Eclipse 与“B”完美配合
  2. Netbeans有个换行问题可以解决（因为eclipse中不会出现这个问题）（不用在B("B")后面加空格）。
