merge --squash 和 rebase 有什么区别？

Arya

2023-02-18 03:30

我试图了解壁球和变基之间的区别。据我了解，一个人在做变基时会进行挤压。



解答

和git merge --squash都git rebase --interactive可以产生“压缩的”提交。但它们有不同的用途。

  • git merge --squash abranch

将在目标分支上生成压缩提交，而不标记任何合并关系。（注意：它不会立即产生提交：您需要一个额外的git commit -m "squash branch"）

如果您想完全丢弃源分支，这很有用（从SO 问题中获取的架构）：

git checkout stable

          X               stable
         /
a---b---c---d---e---f---g tmp

到：

git merge --squash tmp
git commit -m "squash tmp"


# In the following graph, G is c--d--e--f--g squashed together

          X-------------G stable
         /
a---b---c---d---e---f---g tmp

然后删除tmp分支。

注意：git merge有一个--commit选项，但不能与 一起使用--squash。 永远 不可能一起使用--commitand --squash。自 Git 2.22.1（2019 年第 3 季度）起，这种不兼容性变得明确：

请参阅Vishal Verma ( )的提交 1d14d0c（2019 年 5 月 24 日）。（由Junio C Hamano合并——提交33f2790，2019年 7 月 25 日）reloadbrain``gitster

merge:--commit拒绝--squash

以前，在--squash提供时，' option_commit' 被静默删除。对于试图显式使用 squash 的无提交行为的用户来说，这可能会让他们感到惊讶--commit。

git/git builtin/merge.c#cmd_merge()现在包括：

if (option_commit > 0)
    die(_("You cannot combine --squash with --commit."));
  • git rebase --interactive

在一个新的基础上重播你的部分或全部提交，允许你压缩（或最近“修复”，参见这个SO 问题），直接进入：

git checkout tmp
git rebase -i stable

   stable
      X----------------G tmp
     /
a---b

如果您选择压缩所有提交tmp（但是，相反merge --squash，您可以选择重播一些，并压缩其他）。

所以区别是：

  • squash不触及您的源分支（tmp此处）并在您想要的位置创建单个提交。
  • rebase允许您 继续使用相同的源分支 （仍然tmp）：
    • 新基地
    • 更干净的历史
