计数 (*) 与计数 (1) - SQL Server

Arya

2023-02-17 11:33

只是想知道你们中是否有人使用Count(1)过Count(*)，性能上是否存在明显差异，或者这是否只是过去的遗留习惯？

具体数据库为SQL Server 2005.



解答

没有区别。

原因：

在线书籍说“ COUNT ( { [ [ ALL | DISTINCT ] expression ] | * } )”

“1”是一个非空表达式：所以它与COUNT(*). 优化器认识到它的本质：微不足道。

EXISTS (SELECT * ...与或相同EXISTS (SELECT 1 ...

例子：

SELECT COUNT(1) FROM dbo.tab800krows
SELECT COUNT(1),FKID FROM dbo.tab800krows GROUP BY FKID

SELECT COUNT(*) FROM dbo.tab800krows
SELECT COUNT(*),FKID FROM dbo.tab800krows GROUP BY FKID

相同的 IO，相同的计划，有效

DBA.SE 上的类似问题。

编辑，2011 年 12 月

COUNT(*)在ANSI-92中特别提到（寻找“ Scalar expressions 125”）

案件：

a) 如果指定了 COUNT(*)，则结果是 T 的基数。

也就是说，ANSI 标准将其视为显而易见的意思。 由于COUNT(1)这种迷信，已被 RDBMS 供应商优化掉。否则它将根据 ANSI 进行评估 __

b) 否则，令 TX 为单列表，它是将 应用于 T 的每一行并消除空值的结果。如果消除了一个或多个空值，则会引发完成条件：warning-


更多建议

在 SQL Server 中，这些语句产生相同的计划。

与流行的观点相反，在 Oracle 中他们也是如此。

SYS_GUID()在 Oracle 中是计算密集型函数。

在我的测试数据库中，t_even是一个包含1,000,000行的表

这个查询：

SELECT  COUNT(SYS_GUID())
FROM    t_even

运行几48秒钟，因为该函数需要评估每个SYS_GUID()返回的值以确保它不是NULL.

但是，此查询：

SELECT  COUNT(*)
FROM    (
        SELECT  SYS_GUID()
        FROM    t_even
        )

运行但几2秒钟，因为它甚至不尝试评估SYS_GUID()（尽管*是参数COUNT(*)）
