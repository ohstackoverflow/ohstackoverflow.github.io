解答

欢迎来到非规范化浮点数 的世界！他们会对性能造成严重破坏！！！

非正规（或次正规）数字是一种黑客手段，可以从浮点表示中获得一些非常接近于零的额外值。非规范化浮点运算可能比规范化浮点运算 慢数十到数百倍。 这是因为许多处理器不能直接处理它们，必须使用微代码捕获和解析它们。

如果您在 10,000 次迭代后打印出这些数字，您将看到它们收敛到不同的值，具体取决于是否使用0或。0.1

这是在 x64 上编译的测试代码：

int main() {

    double start = omp_get_wtime();

    const float x[16]={1.1,1.2,1.3,1.4,1.5,1.6,1.7,1.8,1.9,2.0,2.1,2.2,2.3,2.4,2.5,2.6};
    const float z[16]={1.123,1.234,1.345,156.467,1.578,1.689,1.790,1.812,1.923,2.034,2.145,2.256,2.367,2.478,2.589,2.690};
    float y[16];
    for(int i=0;i<16;i++)
    {
        y[i]=x[i];
    }
    for(int j=0;j<9000000;j++)
    {
        for(int i=0;i<16;i++)
        {
            y[i]*=x[i];
            y[i]/=z[i];
#ifdef FLOATING
            y[i]=y[i]+0.1f;
            y[i]=y[i]-0.1f;
#else
            y[i]=y[i]+0;
            y[i]=y[i]-0;
#endif

            if (j > 10000)
                cout << y[i] << "  ";
        }
        if (j > 10000)
            cout << endl;
    }

    double end = omp_get_wtime();
    cout << end - start << endl;

    system("pause");
    return 0;
}

请注意在第二次运行中数字如何非常接近于零。

非规范化数字通常很少见，因此大多数处理器不会尝试有效地处理它们。

为了证明这与非规范化数字有关，如果我们通过将以下代码添加到代码开头来将 非规范化数清零：

_MM_SET_FLUSH_ZERO_MODE(_MM_FLUSH_ZERO_ON);

然后带有的版本0不再慢 10 倍，实际上变得更快。（这需要在启用 SSE 的情况下编译代码。）

这意味着我们不是使用这些奇怪的低精度几乎为零的值，而是四舍五入为零。

时序：Core i7 920 @ 3.5 GHz：

//  Don't flush denormals to zero.
0.1f: 0.564067
0   : 26.7669

//  Flush denormals to zero.
0.1f: 0.587117
0   : 0.341406

最后，这真的与它是整数还是浮点数无关。or被0转换0.1f/存储到两个循环之外的寄存器中。所以这对性能没有影响。


更多建议

使用gcc差异并将其应用于生成的程序集只会产生以下差异：

73c68,69

81d76

一个cvtsi2ssq确实慢了 10 倍。

显然，该float版本使用从内存加载的XMM寄存器，而该int版本将实际int值 0 转换为float使用该cvtsi2ssq指令，需要花费大量时间。传递-O3给 gcc 没有帮助。（海湾合作委员会版本 4.2.1。）

（使用doubleinstead offloat无关紧要，除非它将 the 更改cvtsi2ssq为 a cvtsi2sdq。）

更新

一些额外的测试表明它不一定是cvtsi2ssq指令。一旦消除（使用 aint ai=0;float a=ai;和使用a而不是0），速度差异仍然存在。所以@Mysticial 是对的，非规范化的浮点数有所不同。0这可以通过测试和之间的值看出0.1f。上面代码的转折点大约在0.00000000000000000000000000000001，此时循环突然需要 10 倍的时间。
