接口与基类

Arya

2023-02-17 12:12

什么时候应该使用接口，什么时候应该使用基类？

如果我不想实际定义方法的基本实现，它是否应该始终是一个接口？

如果我有狗和猫课。为什么我要实施 IPet 而不是 PetBase？我可以理解为 ISheds 或 IBarks (IMakesNoise?) 提供接口，因为它们可以一个接一个地放在宠物身上，但我不知道将哪个用于普通宠物。



解答

让我们以 Dog 和 Cat 类为例，并使用 C# 进行说明：

狗和猫都是动物，特别是四足哺乳动物（动物太笼统了）。让我们假设您有一个抽象类 Mammal，用于它们：

public abstract class Mammal

这个基类可能会有默认方法，例如：

  • 喂养
  • 伴侣

所有这些都是在两个物种之间或多或少具有相同实现的行为。要定义它，您将拥有：

public class Dog : Mammal
public class Cat : Mammal

现在让我们假设还有其他哺乳动物，我们通常会在动物园看到它们：

public class Giraffe : Mammal
public class Rhinoceros : Mammal
public class Hippopotamus : Mammal

这仍然有效，因为它是功能的核心Feed()，并且Mate()仍然是相同的。

然而，长颈鹿、犀牛和河马并不是可以用来当宠物的动物。这就是接口有用的地方：

public interface IPettable
{
    IList<Trick> Tricks{get; set;}
    void Bathe();
    void Train(Trick t);
}

猫狗之间的上述合同的执行情况将不同；将它们的实现放在抽象类中进行继承将是一个坏主意。

您的 Dog 和 Cat 定义现在应该如下所示：

public class Dog : Mammal, IPettable
public class Cat : Mammal, IPettable

理论上你可以从更高的基类覆盖它们，但本质上一个接口允许你只添加你需要的东西到一个类中而不需要继承。

因此，因为你通常只能从一个抽象类继承（在大多数静态类型的 OO 语言中......例外包括 C++）但能够实现多个接口，它允许你严格按照需要构造 对象 。


更多建议

好吧，Josh Bloch 在Effective Java 2d中说：

优先使用接口而不是抽象类

一些要点：

  • 可以很容易地改造现有的类来实现新的接口 。您所要做的就是添加所需的方法（如果它们尚不存在）并在类声明中添加一个 implements 子句。

  • 接口是定义 mixin 的理想选择 。笼统地说，mixin 是类可以在其“主要类型”之外实现的一种类型，以声明它提供了一些可选行为。例如，Comparable 是一个混合接口，它允许一个类声明其实例相对于其他相互可比较的对象是有序的。

  • 接口允许构建非分层类型框架 。类型层次结构非常适合组织某些事物，但其他事物并不能整齐地归入严格的层次结构中。

  • *接口通过包装类习惯用法 *实现安全、强大的功能增强。 如果你使用抽象类来定义类型，你会让想要添加功能的程序员别无选择，只能使用继承。

此外，您可以通过提供一个抽象的骨架实现类来结合接口和抽象类的优点，以配合您导出的每个重要接口。

另一方面，接口很难进化。如果您向接口添加一个方法，它将破坏它的所有实现。
