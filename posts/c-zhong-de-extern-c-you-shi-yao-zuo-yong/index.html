C++中的extern "C"有什么作用？

Arya

2023-02-16 23:00

extern "C"放入C++ 代码到底有什么作用？

例如：

extern "C" {
   void foo();
}



解答

extern "C"使 C++ 中的函数名称具有 C 链接（编译器不会破坏名称），以便客户端 C 代码可以使用仅包含函数声明的 C 兼容头文件链接到（使用）您的函数。您的函数定义包含在二进制格式中（由您的 C++ 编译器编译），然后客户端 C 链接器将使用 C 名称链接到该格式。

由于 C++ 有函数名的重载而 C 没有，C++ 编译器不能只使用函数名作为链接的唯一 id，因此它通过添加有关参数的信息来破坏名称。AC 编译器不需要破坏名称，因为您不能在 C 中重载函数名称。当您在extern "C"C++ 中声明函数具有链接时，C++ 编译器不会将参数/参数类型信息添加到用于链接的名称中。

如您所知，您可以明确指定extern "C"与每个单独的声明/定义的链接，或者使用块将一系列声明/定义分组以具有特定链接：

extern "C" void foo(int);
extern "C"
{
   void g(char);
   int i;
}

如果您关心技术细节，它们在 C++03 标准的第 7.5 节中列出，这里是一个简短的摘要（强调extern "C"）：

  • extern "C"是链接规范
  • 每个编译器都 需要 提供“C”链接
  • 链接规范应仅出现在名称空间范围内
  • 所有的函数类型、函数名和变量名都有语言链接 见理查德的评论： 只有具有外部链接的函数名和变量名才有语言链接
  • 具有不同语言链接的两个函数类型是不同的类型，即使在其他方面相同
  • Linkage specs嵌套，inner one决定最终的linkage
  • extern "C"被类成员忽略
  • 至多一个具有特定名称的函数可以具有“C”链接（无论命名空间如何）
  • extern "C" 强制函数具有外部链接（不能使其成为静态的） 参见 Richard 的评论： inside is valid; 如此声明的实体具有内部链接，因此没有语言链接 static``extern "C"
  • 从 C++ 到用其他语言定义的对象的链接以及从其他语言到用 C++ 定义的对象的链接是实现定义的和语言相关的。只有两种语言实现的对象布局策略足够相似，才能实现这种联动

只是想添加一些信息，因为我还没有看到它发布。

你会经常在 C 头文件中看到这样的代码：

#ifdef __cplusplus
extern "C" {
#endif

// all of your legacy C code here

#ifdef __cplusplus
}
#endif

这样做的目的是允许您将 C 头文件与您的 C++ 代码一起使用，因为宏__cplusplus将被定义。但是您 仍然可以将它与 未定义 宏的遗留 C 代码一起使用，因此它不会看到独特的 C++ 构造。

虽然，我也看到过 C++ 代码，例如：

extern "C" {
#include "legacy_C_header.h"
}

我想这会完成同样的事情。

不确定哪种方式更好，但我都见过。
