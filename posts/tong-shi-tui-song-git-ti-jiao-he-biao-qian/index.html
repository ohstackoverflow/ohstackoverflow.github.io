同时推送 git 提交和标签

Arya

2023-02-18 05:39

我知道这git push --tags是与普通 old 单独操作的原因git push。推送标签应该是一个有意识的选择，因为您不想意外地推送一个标签。没关系。但是有没有办法把两者结合起来呢？（除了git push && git push --tags。）



解答

2020 年 8 月更新

正如SoBeRich在这个答案中最初提到的，在我自己的答案中，从git 2.4.x开始

git push --atomic origin <branch name> <tag>

（注意：这实际上只适用于 Git 2.24 的 HTTPS）

2015 年 5 月更新

从git 2.4.1开始，你可以做

git config --global push.followTags true

如果设置为 true，则默认启用 --follow-tags 选项。
您可以通过指定 --no-follow-tags 在推送时覆盖此配置。

正如Matt Rogers 在回答 Wes Hurd 的帖子中所指出的：

--follow-tags 只推送带注释的标签 。

git tag -a -m "I'm an annotation" <tagname>

这将被推送（而不是git tag <tagname>轻量级标签，它不会被推送，正如我在这里提到的那样）

2013 年 4 月更新

从git 1.8.3（2013 年 4 月 22 日）开始， 您不再需要执行 2 个命令来推送分支，然后推送标签 ：

新的“ --follow-tags”选项告诉“ git push” 在推出分支时推出相关的注释标签 。

您现在可以在推送新提交时尝试：

git push --follow-tags

不过，这不会推送 所有 本地标签，只会推送由git push.

Git 2.4.1+（2015 年第 2 季度）将引入该选项push.followTags：请参阅“ How to make “ git push” include tags within a branch? ”。

原始答案，2010 年 9 月

核选项将是git push --mirror，这将推动所有参考refs/。

你也可以在你当前的分支提交中只推送一个标签：

git push origin : v1.0.0

您可以将该--tags选项与 refspec 结合使用，例如：

git push origin --tags :

（因为--tags意味着： _ 除了在命令行上明确列出的 refspecs 之外refs/tags，所有 refs 都被推送） _ ****

您还有此条目“使用单个“git push”调用推送分支和标签”

Zoltán Füzesi刚刚在Git 邮件列表中发布了一个有用的提示：

我用来.git/config解决这个问题：

[remote "origin"]
    url = ...
    fetch = +refs/heads/*:refs/remotes/origin/*
    push = +refs/heads/*
    push = +refs/tags/*

添加这些行将git push origin上传您的所有分支和标签。如果你只想上传其中的一些，你可以枚举它们。

我自己还没有尝试过，但看起来它可能会有用，直到将同时推送分支和标签的其他方式添加到 git push。
另一方面，我不介意输入：

$ git push && git push --tags

当心 ，正如Aseem Kishore评论的那样

push = +refs/heads/*将 强制推动你所有的分支机构 。

刚才这让我很反感，所以仅供参考。

René Scheibe添加了这个有趣的评论：

该--follow-tags参数具有误导性，因为只.git/refs/tags考虑下面的标签。
如果git gc运行，标记将从移动.git/refs/tags到.git/packed-refs。之后git push --follow- tags ...不再按预期工作。


更多建议

@从 Git 2.4 开始

git push --atomic origin <branch name> <tag>
