为什么使用 pip 而不是 easy_install？

Arya

2023-02-17 08:25

一条推文写道：

不要使用 easy_install，除非你喜欢用刀捅自己的脸。使用点子。

为什么使用 pip 而不是 easy_install？问题不主要在于 PyPI 和包作者吗？如果作者将垃圾源 tarball（例如：缺少文件，没有 setup.py）上传到 PyPI，那么 pip 和 easy_install 都会失败。除了外观上的差异，为什么 Python 用户（如上面的推文）似乎更 喜欢 pip 而不是 easy_install？

（假设我们正在讨论由社区维护的 Distribute 包中的 easy_install）



解答

这里的许多答案在 2015 年已经过时（尽管Daniel Roseman 最初接受的答案不是）。这是目前的情况：

  • 二进制包现在作为轮子（文件）分发.whl——不仅在 PyPI 上，而且在第三方存储库中，如Christoph Gohlke 的 Windows 扩展包。pip可以处理轮子；easy_install不能。
  • 虚拟环境（内置于 3.4，或者可以添加到 2.6+/3.1+ virtualenv）已经成为一个非常重要和突出的工具（并在官方文档中推荐）；它们pip开箱即用，但甚至无法与easy_install.
  • 不再维护distribute包含的包。easy_install它的改进setuptools被合并回setuptools. 尝试安装distribute只会安装setuptools。
  • easy_install本身只是准维护。
  • 所有过去pip不如easy_install从解压的源代码树、从 DVCS 存储库安装等的情况都早已不复存在；你可以pip install .，pip install git+https://。
  • pip附带来自 python.org 的官方 Python 2.7 和 3.4+ 包，pip如果您从源代码构建，则默认包含引导程序。
  • Python Packaging User Guide取代了有关安装、使用和构建包的各种不完整文档。Python 自己的关于安装 Python 模块的文档现在遵循本用户指南，并明确称为pip“首选安装程序”。
  • pip多年来，其他新功能已被添加到easy_install. 例如，pip通过构建需求文件然后在每一侧使用单个命令安装它，可以轻松克隆站点包。或者将您的需求文件转换为本地存储库以用于内部开发。等等。

据我所知，在 2015 年使用的唯一充分理由easy_install是在 OS X 10.5-10.8 中使用苹果预装的 Python 版本的特殊情况。从 10.5 开始，Apple 已经包含easy_install，但从 10.10 开始他们仍然不包含pip。对于 10.9+，你仍然应该只使用get-pip.py，但是对于 10.5-10.8，这有一些问题，所以更容易sudo easy_install pip。（一般来说，easy_install pip这是个坏主意；你只希望在 OS X 10.5-10.8 上执行此操作。）另外，10.5-10.8 以readline一种easy_install知道如何拼凑但pip不知道如何拼凑的方式包含，所以你也想sudo easy_install readline如果你想升级它。
