如何修复 Hibernate“对象引用未保存的瞬态实例 - 在刷新前保存瞬态实例”错误

Arya

2023-02-17 11:34

使用 Hibernate 保存对象时收到以下错误

object references an unsaved transient instance - save the transient instance before flushing



解答

您应该在集合映射中包含cascade="all"（如果使用 xml）或（如果使用注释）。cascade=CascadeType.ALL

发生这种情况是因为您的实体中有一个集合，并且该集合包含一个或多个不存在于数据库中的项目。通过指定上述选项，您告诉 hibernate 在保存它们的父级时将它们保存到数据库中。

设计函数 f(f(n)) == -n

Arya

2023-02-17 11:34

我在上次面试中遇到的一个问题：

设计一个函数f，使得：

f(f(n)) == -n

哪里n是一个 32 位 有符号整数 ；你不能使用复数算术。

如果您不能为整个数字范围设计这样的函数，请将其设计为尽可能大的范围。

有任何想法吗？



解答

怎么样：

f(n) = sign(n) - (-1)ⁿ * n

在 Python 中：

def f(n): 
    if n == 0: return 0
    if n >= 0:
        if n % 2 == 1: 
            return n + 1
        else: 
            return -1 * (n - 1)
    else:
        if n % 2 == 1:
            return n - 1
        else:
            return -1 * (n + 1)

Python 自动将整数提升为任意长度的 long。在其他语言中，最大的正整数会溢出，因此它适用于除那个整数之外的所有整数。

要使其适用于实数，您需要将(-1)ⁿ 中的 n{ ceiling(n) if n>0; floor(n) if n<0 }替换为。

在 C# 中（适用于任何 double，溢出情况除外）：

static double F(double n)
{
    if (n == 0) return 0;

    if (n < 0)
        return ((long)Math.Ceiling(n) % 2 == 0) ? (n + 1) : (-1 * (n - 1));
    else
        return ((long)Math.Floor(n) % 2 == 0) ? (n - 1) : (-1 * (n + 1));
}

计数 (*) 与计数 (1) - SQL Server

Arya

2023-02-17 11:33

只是想知道你们中是否有人使用Count(1)过Count(*)，性能上是否存在明显差异，或者这是否只是过去的遗留习惯？

具体数据库为SQL Server 2005.



解答

没有区别。

原因：

在线书籍说“ COUNT ( { [ [ ALL | DISTINCT ] expression ] | * } )”

“1”是一个非空表达式：所以它与COUNT(*). 优化器认识到它的本质：微不足道。

EXISTS (SELECT * ...与或相同EXISTS (SELECT 1 ...

例子：

SELECT COUNT(1) FROM dbo.tab800krows
SELECT COUNT(1),FKID FROM dbo.tab800krows GROUP BY FKID

SELECT COUNT(*) FROM dbo.tab800krows
SELECT COUNT(*),FKID FROM dbo.tab800krows GROUP BY FKID

相同的 IO，相同的计划，有效

编辑，2011 年 8 月

DBA.SE 上的类似问题。

编辑，2011 年 12 月

COUNT(*)在ANSI-92中特别提到（寻找“ Scalar expressions 125”）

案件：

a) 如果指定了 COUNT(*)，则结果是 T 的基数。

也就是说，ANSI 标准将其视为显而易见的意思。 由于COUNT(1)这种迷信，已被 RDBMS 供应商优化掉。否则它将根据 ANSI 进行评估 __

b) 否则，令 TX 为单列表，它是将 应用于 T 的每一行并消除空值的结果。如果消除了一个或多个空值，则会引发完成条件：warning-


更多建议

在 SQL Server 中，这些语句产生相同的计划。

与流行的观点相反，在 Oracle 中他们也是如此。

SYS_GUID()在 Oracle 中是计算密集型函数。

在我的测试数据库中，t_even是一个包含1,000,000行的表

这个查询：

SELECT  COUNT(SYS_GUID())
FROM    t_even

运行几48秒钟，因为该函数需要评估每个SYS_GUID()返回的值以确保它不是NULL.

但是，此查询：

SELECT  COUNT(*)
FROM    (
        SELECT  SYS_GUID()
        FROM    t_even
        )

运行但几2秒钟，因为它甚至不尝试评估SYS_GUID()（尽管*是参数COUNT(*)）
