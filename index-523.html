如何使用现有自定义主题在 XML 中隐藏活动的标题栏

Arya

2023-02-17 09:10

我想隐藏我的一些活动的标题栏。问题是我对我的所有活动都应用了一种样式，因此我不能简单地将主题设置为@android:style/Theme.NoTitleBar.

使用 NoTitleBar 主题作为我的样式的父主题将从我的所有活动中删除标题栏。

我可以在某处设置无标题样式项目吗？



解答

我现在做了以下事情。

我声明了一种样式，它继承了我的一般样式的所有内容，然后禁用了 titleBar。

<style name="generalnotitle" parent="general">
    <item name="android:windowNoTitle">true</item>
</style>

现在我可以将此样式设置到每个我想隐藏标题栏的活动，覆盖应用程序范围的样式并继承所有其他样式信息，因此样式代码中没有重复。

要将样式应用于特定的 Activity，请打开AndroidManifest.xml并将以下属性添加到activity标记中；

<activity
    android:theme="@style/generalnotitle">


更多建议

您可以修改您的AndroidManifest.xml：

<activity android:name=".MainActivity"
          android:label="@string/app_name"
          android:theme="@android:style/Theme.Black.NoTitleBar.Fullscreen">

android:theme="@android:style/Theme.Black.NoTitleBar"或者在不需要全屏活动时使用。

注意： 如果您之前使用过“默认”视图，您可能还应该将父类从更改AppCompatActivity为Activity.

错误 java.lang.OutOfMemoryError：超出 GC 开销限制

Arya

2023-02-17 09:09

我在执行JUnit测试时收到此错误消息：

java.lang.OutOfMemoryError: GC overhead limit exceeded

我知道 anOutOfMemoryError是什么，但是 GC 开销限制是什么意思？我该如何解决这个问题？



解答

此消息意味着由于某种原因，垃圾收集器占用了过多的时间（默认情况下占进程所有 CPU 时间的 98%），并且在每次运行中回收的内存很少（默认情况下占堆的 2%）。

这实际上意味着您的程序停止任何进展，并且一直忙于只运行垃圾收集。

为了防止您的应用程序在没有完成任何事情的情况下占用 CPU 时间，JVM 会抛出此消息，Error以便您有机会诊断问题。

我见过这种情况的罕见情况是某些代码在内存已经非常受限的环境中创建大量临时对象和大量弱引用对象。

查看 Java GC 调优指南，它适用于各种 Java 版本并包含有关此特定问题的部分：

  • Java 11 调优指南针对不同的垃圾收集器提供了关于过度 GC 的专门章节：
    • 对于并行收集器
    • 用于并发标记扫描 (CMS) 收集器
    • 没有提及垃圾优先 (G1) 收集器的这种特定错误情况。
  • Java 8 调优指南及其过度 GC 部分
  • Java 6 调优指南及其过度 GC 部分。


更多建议

引用自 Oracle 的文章 “Java SE 6 HotSpot[tm] Virtual Machine Garbage Collection Tuning” ：

GC 时间过长和 OutOfMemoryError

如果在垃圾收集上花费了太多时间，并行收集器将抛出一个 OutOfMemoryError：如果超过 98% 的总时间花在了垃圾收集上并且回收了不到 2% 的堆，将抛出一个 OutOfMemoryError。此功能旨在防止应用程序长时间运行而由于堆太小而进展甚微或根本没有进展。-XX:-UseGCOverheadLimit如有必要，可以通过将选项添加到命令行来禁用此功能。

跳过 Git 提交钩子

Arya

2023-02-17 09:09

我正在查看一个 Git 挂钩，它在 Python 代码中查找打印语句。如果找到打印语句，它会阻止 Git 提交。

我想覆盖这个钩子，有人告诉我有一个命令可以这样做。我没能找到它。有什么想法吗？



解答

也许（来自git commit手册页）：

git commit --no-verify -m "commit message"
           ^^^^^^^^^^^
-n  
--no-verify

此选项绕过 pre-commit 和 commit-msg 挂钩。另见githooks(5)。

正如Blaise评论的那样，-n对于某些命令可以有不同的作用。
例如，git push -n实际上是一个试运行推送。
只会git push --no-verify跳过钩子。

注意：Git 2.14.x/2.15 改进了--no-verify行为：

请参阅Kevin Willford (`)的提交 680ee55（2017 年 8 月 14 日）。（由Junio C Hamano合并——提交c3e034f，2017 年 8 月 23 日）gitster`

commit: 如果没有pre-commithook ，则跳过丢弃索引

" git commit" 用于丢弃索引并从文件系统重新读取，以防钩子pre- commit在中间更新它；当我们知道我们不运行钩子时，这已经被优化了pre-commit。

Davi Lima在评论中指出不git cherry-pick支持 --no -verify。
因此，如果 cherry-pick 触发了预提交挂钩，您可能会像这篇博文中那样，必须以某种方式评论/禁用该挂钩，以便您的 git cherry-pick 继续。

git rebase --continue在合并冲突解决后的情况下，将需要相同的过程。

使用 Git 2.36（2022 年第 2 季度），调用者run_commit_hook()可以了解它“成功”是因为挂钩成功还是因为没有任何挂钩。

请参阅Ævar Arnfjörð Bjarmason ( )的提交 a8cc594（已修复提交 4369e3a1）、提交 9f6e63b（2022 年 3 月 7 日）。（由Junio C Hamano合并——提交7431379，2022年 3 月 16 日）avar
gitster

hooks：修复一个晦涩的 TOCTOU “我们刚刚运行了一个钩子吗？” 种族

签字人：Ævar Arnfjörð Bjarmason

修复680ee55中添加的代码中的 检查时间到使用时间 ( TOCTOU )竞争（“ ：如果没有预提交挂钩，则跳过丢弃索引”，2017-08-14，Git v2.15.0- rc0——合并在第 3 批中列出）。commit

如果我们运行“ pre-commit”挂钩并且它修改了索引，但hook_exists()稍后返回 false（例如，因为挂钩本身消失，目录变得不可读等），就会发生这种模糊的竞争条件。
那么我们不会discard_cache()在我们应该打电话的时候打电话。

竞争条件本身可能无关紧要，用户在实践中不太可能遇到它。在讨论680ee55
时，这个问题已在列表中注明，但尚未修复。

让我们也为 push-to-checkout 钩子改变这个。
现在，我们将始终尝试推送结帐，而不是检查挂钩是否存在并进行推送结帐或推送部署。
如果挂钩不存在，我们将退回到推送部署。
与之前相同的行为，没有 TOCTOU 比赛。
参见0855331 (" receive-pack: support push-to-checkout hook", 2014-12-01, Git v2.4.0-rc0 -- merge ) 对前面行为的介绍。

这使得 ofhook_exists()在两个重要的地方使用。
“reference-transaction”签入refs.c，参见6754159（“ refs：实施参考交易挂钩”，2020-06-19，Git v2.28.0-rc0 -- merge listed in batch

7），以及“prepare-commit-msg " 钩子，参见66618a5（" sequencer：运行 'prepare-commit-

msg' 钩子"，2018 年 1 月 24 日，Git v2.17.0-rc0 --合并列在第 2 批次中）。

在这两种情况下，我们都通过不为挂钩准备数据来节省 CPU 时间，如果没有挂钩，我们将什么都不做。
所以"invoked_hook"在这些情况下使用这种模式没有意义。

" reference-transaction" 和 " prepare-commit-msg" 钩子也不活泼。
在这些情况下，如果我们在添加新钩子的情况下进行比赛，我们将跳过钩子运行，而在此处修复的 TOCTOU 比赛中，我们错误地跳过了所需的钩子后逻辑。


更多建议

有comment没有no verify任何进一步的问题：

git commit -m "Some comments" --no-verify
