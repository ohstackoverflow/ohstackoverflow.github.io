如何在项目提交历史中找到已删除的文件？

Arya

2023-02-17 00:23

曾几何时，我的项目中有一个文件，我现在希望能够获取该文件。

问题是：我不知道我什么时候删除了它以及它在哪条路径上。

当文件存在时，如何找到它的提交？



解答

如果您不知道您可以使用的确切路径

git log --all --full-history -- "**/thefile.*"

如果你知道文件所在的路径，你可以这样做：

git log --all --full-history -- <path-to-file>

这应该显示触及该文件的所有分支中的提交列表。然后，你可以找到你想要的文件的版本，并显示它......

git show <SHA> -- <path-to-file>

或者将其恢复到您的工作副本中：

git checkout <SHA>^ -- <path-to-file>

请注意插入符号 ( )，它会 先于^已识别的文件进行检出，因为在提交时文件已被删除，我们需要查看上一次提交以获取已删除文件的内容 __<SHA>


更多建议

获取已删除文件的列表并复制已删除文件的完整路径

git log --diff-filter=D --summary | grep delete

执行下一个命令以查找该提交的提交 ID 并复制提交 ID

git log --all -- FILEPATH

显示已删除文件的差异

git show COMMIT_ID -- FILE_PATH

>请记住，您可以使用like将输出写入文件

git show COMMIT_ID -- FILE_PATH > deleted.diff

插入到 ... 值 ( SELECT ... FROM ... )

我正在尝试INSERT INTO使用来自另一个表的输入来创建一个表。尽管这对于许多数据库引擎来说是完全可行的，但我似乎总是难以记住当时引擎SQL（MySQL、Oracle、SQL Server、Informix和DB2）的正确语法。

是否有来自 SQL 标准（例如， SQL-92 ）的银弹语法允许我插入值而不用担心底层数据库？



解答

尝试：

INSERT INTO table1 ( column1 )
SELECT  col1
FROM    table2

这是标准的 ANSI SQL，应该适用于任何 DBMS

它绝对适用于：

  • 甲骨文
  • 微软SQL服务器
  • MySQL
  • Postgres
  • SQLite v3
  • 天睿数据
  • 数据库
  • 数据库
  • 垂直
  • 数据库
  • 氢气
  • 红移
  • SAP HANA
  • 谷歌扳手


更多建议

Claude Houle 的回答：应该可以正常工作，您也可以有多个列和其他数据：

INSERT INTO table1 ( column1, column2, someInt, someVarChar )
SELECT  table2.column1, table2.column2, 8, 'some string etc.'
FROM    table2
WHERE   table2.ID = 7;

我只在 Access、SQL 2000/2005/Express、MySQL 和 PostgreSQL 中使用过这种语法，因此应该涵盖这些内容。它也应该与 SQLite3 一起工作。

我如何检查对象是否具有 JavaScript 中的特定属性？

Arya

2023-02-17 00:22

我如何检查对象是否具有 JavaScript 中的特定属性？

考虑：

x = {'key': 1};
if ( x.hasOwnProperty('key') ) {
    //Do this
}

这是最好的方法吗？



解答

2022 更新

Object.hasOwn()

Object.hasOwn()建议不要使用，Object.hasOwnProperty()因为它适用于使用Object.create(null)重写继承hasOwnProperty()方法的对象创建的对象。虽然可以通过调用Object.prototype.hasOwnProperty()外部对象来解决这些问题，Object.hasOwn()但更直观。

例子
const object1 = {
  prop: 'exists'
};

console.log(Object.hasOwn(object1, 'prop'));
// expected output: true

原答案

我真的对给出的答案感到困惑——其中大部分都是完全错误的。当然，您可以拥有具有未定义、空值或假值的对象属性。因此，简单地将属性检查减少到typeof this[property]or，甚至更糟，x.key会给您带来完全误导的结果。

这取决于您要查找的内容。如果你想知道一个对象是否在物理上包含一个属性（并且它不是来自原型链上的某个地方）那么就是object.hasOwnProperty要走的路。所有现代浏览器都支持它。（它在旧版本的 Safari - 2.0.1 及更早版本中缺失 - 但这些浏览器版本很少再使用。）

如果你正在寻找的是一个对象是否有一个可迭代的属性（当你迭代对象的属性时，它会出现）然后做：会给你想要的效果prop in object。

由于 usinghasOwnProperty可能是您想要的，并且考虑到您可能需要一种后备方法，因此我向您提供以下解决方案：

var obj = {
    a: undefined,
    b: null,
    c: false
};

// a, b, c all found
for ( var prop in obj ) {
    document.writeln( "Object1: " + prop );
}

function Class(){
    this.a = undefined;
    this.b = null;
    this.c = false;
}

Class.prototype = {
    a: undefined,
    b: true,
    c: true,
    d: true,
    e: true
};

var obj2 = new Class();

// a, b, c, d, e found
for ( var prop in obj2 ) {
    document.writeln( "Object2: " + prop );
}

function hasOwnProperty(obj, prop) {
    var proto = obj.__proto__ || obj.constructor.prototype;
    return (prop in obj) &&
        (!(prop in proto) || proto[prop] !== obj[prop]);
}

if ( Object.prototype.hasOwnProperty ) {
    var hasOwnProperty = function(obj, prop) {
        return obj.hasOwnProperty(prop);
    }
}

// a, b, c found in modern browsers
// b, c found in Safari 2.0.1 and older
for ( var prop in obj2 ) {
    if ( hasOwnProperty(obj2, prop) ) {
        document.writeln( "Object2 w/ hasOwn: " + prop );
    }
}

上面是一个有效的、跨浏览器的解决方案hasOwnProperty()，有一个警告：它无法区分原型和实例上的相同属性的情况——它只是假设它来自原型。您可以根据自己的情况将其更改为更宽松或更严格，但这至少应该更有帮助。


更多建议

使用Underscore.js或（甚至更好）Lodash：

_.has(x, 'key');

调用Object.prototype.hasOwnProperty，但 (a) 更短于键入，并且 (b) 使用“对”的安全引用hasOwnProperty（即即使hasOwnProperty被覆盖它也能工作）。

特别地，Lodash 定义_.has为：

function has(object, key) {
  return object ? hasOwnProperty.call(object, key) : false;
}
// hasOwnProperty = Object.prototype.hasOwnProperty
