如何将本地 .jar 文件依赖添加到 build.gradle 文件？

Arya

2023-02-17 14:01

我试图将我的本地 .jar 文件依赖项添加到我的 build.gradle 文件中：

apply plugin: 'java'

sourceSets {
    main {
        java {
            srcDir 'src/model'
        }
    }
}

dependencies {
    runtime files('libs/mnist-tools.jar', 'libs/gson-2.2.4.jar')
    runtime fileTree(dir: 'libs', include: '*.jar')
}

您可以看到我将 .jar 文件添加到此处的 referencedLibraries 文件夹中： https: //github.com/WalnutiQ/wAlnut/tree/version-2.3.1/referencedLibraries

但问题是，当我在命令行上运行命令：gradle build 时，出现以下错误：

error: package com.google.gson does not exist
import com.google.gson.Gson;

这是我的整个回购协议：https://github.com/WalnutiQ/wAlnut/tree/version-2.3.1



解答

如果你真的需要从本地目录中获取那个 .jar，

在您的模块 gradle 旁边添加（不是应用程序 gradle 文件）：

repositories {
   flatDir {
       dirs("libs")
   }
}


dependencies {
   implementation("gson-2.2.4")
}

但是，作为实际 Maven 存储库中的标准 .jar，您为什么不试试这个呢？

repositories {
   mavenCentral()
}
dependencies {
   implementation("com.google.code.gson:gson:2.2.4")
}

collections.defaultdict 是如何工作的？

Arya

2023-02-17 14:00

我已经阅读了 python 文档中的示例，但仍然无法弄清楚此方法的含义。有人可以帮忙吗？这是 python 文档中的两个示例

>>> from collections import defaultdict

>>> s = 'mississippi'
>>> d = defaultdict(int)
>>> for k in s:
...     d[k] += 1
...
>>> d.items()
[('i', 4), ('p', 2), ('s', 4), ('m', 1)]

和

>>> s = [('yellow', 1), ('blue', 2), ('yellow', 3), ('blue', 4), ('red', 1)]
>>> d = defaultdict(list)
>>> for k, v in s:
...     d[k].append(v)
...
>>> d.items()
[('blue', [2, 4]), ('red', [1]), ('yellow', [1, 3])]

参数int是list什么？



解答

KeyError通常，如果您尝试获取具有当前不在字典中的键的项目，Python 字典会抛出一个。相比之下defaultdict，只会创建您尝试访问的任何项目（当然前提是它们尚不存在）。要创建这样一个“默认”项目，它会调用您传递给构造函数的函数对象（更准确地说，它是一个任意的“可调用”对象，其中包括函数和类型对象）。对于第一个示例，默认项是使用创建的int()，它将返回整数对象0。对于第二个示例，默认项目是使用创建的list()，它返回一个新的空列表对象。


更多建议

defaultdict意味着如果在字典中找不到键，则不会KeyError抛出一个键，而是创建一个新条目。这个新条目的类型由 defaultdict 的参数给出。

例如：

somedict = {}
print(somedict[3]) # KeyError

someddict = defaultdict(int)
print(someddict[3]) # print int(), thus 0

拉取请求与合并请求

Arya

2023-02-17 14:00

拉取请求和合并请求有什么区别？

在 GitHub 中，它是一个 Pull Request，而在 GitLab 中，它是一个 Merge Request。那么，这两者之间有区别吗？



解答

GitLab 的“合并请求”功能相当于 GitHub 的“拉取请求”功能。两者都是从另一个分支或分支中拉取更改到您的分支并将更改与现有代码合并的方法。它们是代码审查和变更管理的有用工具。

GitLab 的一篇文章讨论了命名功能的差异：

合并或拉取请求在 git 管理应用程序中创建，并要求指定人员合并两个分支。GitHub 和 Bitbucket 等工具选择名称拉取请求，因为第一个手动操作是拉取功能分支。GitLab 和 Gitorious 等工具选择名称合并请求，因为这是请求受让人的最终操作。在本文中，我们将它们称为合并请求。

“合并请求”不应与git merge命令混淆。“拉取请求”也不应与git pull命令混淆。这两个git命令都在拉取请求和合并请求的幕后使用，但是合并/拉取请求指的是比这两个命令更广泛的主题。


更多建议

在我看来，它们意味着相同的活动，但从不同的角度来看：

想一想，Alice 在存储库 A 上做了一些提交，它是从 Bob 的存储库 B 分叉出来的。

当 Alice 想要将她的更改“合并”到 B 中时，她实际上是想让 Bob 从 A 中“拉取”这些更改。

因此，从 Alice 的角度来看，这是一个“合并请求”，而 Bob 则将其视为“拉取请求”。
