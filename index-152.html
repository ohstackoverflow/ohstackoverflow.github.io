删除不再在远程的跟踪分支

Arya

2023-02-16 23:14

有没有一种简单的方法可以删除所有远程等效项不再存在的跟踪分支？

例子：

分支机构（本地和远程）

  • 掌握
  • 起源/主人
  • 起源/错误修复-a
  • 起源/错误修复-b
  • 起源/错误修复-c

在本地，我只有一个 master 分支。现在我需要处理 bug-fix-a ，所以我检查它，处理它，然后将更改推送到远程。接下来我对 bug-fix- b 做同样的事情。

分支机构（本地和远程）

  • 掌握
  • 错误修复-a
  • 错误修复-b
  • 起源/主人
  • 起源/错误修复-a
  • 起源/错误修复-b
  • 起源/错误修复-c

现在我有本地分支 master ， bug-fix-a ， bug-fix-b 。Master 分支维护者会将我的更改合并到 master 中，并删除他已经合并的所有分支。

所以现在的状态是：

分支机构（本地和远程）

  • 掌握
  • 错误修复-a
  • 错误修复-b
  • 起源/主人
  • 起源/错误修复-c

现在我想调用一些命令来删除分支（在本例中为 bug-fix-a ， bug-fix-b ），它们不再出现在远程存储库中。

它会像现有的命令git remote prune origin，但更像git local prune origin。



解答

git remote prune origin修剪跟踪分支不在远程。

git branch --merged列出已合并到当前分支的分支。

xargs git branch -d删除标准输入中列出的分支。

小心删除 列出的分支git branch --merged。该列表可能包括master或您不想删除的其他分支。

为了让自己有机会在删除分支之前编辑列表，您可以在一行中执行以下操作：

git branch --merged >/tmp/merged-branches && \
  vi /tmp/merged-branches && xargs git branch -d </tmp/merged-branches


更多建议

最安全的方法是使用git for-each- ref带有插值变量的“管道”命令%(upstream:track)，这将[gone]在分支不再位于远程时发生：

git fetch -p && for branch in $(git for-each-ref --format '%(refname) %(upstream:track)' refs/heads | awk '$2 == "[gone]" {sub("refs/heads/", "", $1); print $1}'); do git branch -D $branch; done

这种方法比使用“瓷器”命令更安全，因为不存在意外匹配部分提交消息的风险。这是一个使用“porcelain”git 命令的版本：

git fetch -p && for branch in $(git branch -vv | grep ': gone]' | awk '{print $1}'); do git branch -D $branch; done

它的工作方式是在命令之后

git fetch -p

删除远程引用，当你运行

git branch -vv

它将显示“消失”作为远程状态。例如，

$ git branch -vv
  master                 b900de9 [origin/master: behind 4] Fixed bug
  release/v3.8           fdd2f4e [origin/release/v3.8: behind 2] Fixed bug
  release/v3.9           0d680d0 [origin/release/v3.9: behind 2] Updated comments
  bug/1234               57379e4 [origin/bug/1234: gone] Fixed bug

这就是脚本迭代的内容。

如何补救“当前不会命中断点。没有为该文档加载任何符号。” 警告？

Arya

2023-02-16 23:13

AC# 桌面应用程序（在 Visual Studio Express 版本上）可以运行，但 5 秒后就不能运行了。

我尝试了以下内容：

  • 确保在所有程序集中设置调试配置、调试标志和完整的调试信息。
  • 从我的整个机器上删除所有与项目相关的 bin 和 obj 文件夹以及所有 DLL 文件。
  • 从头开始重新创建导致问题的项目。
  • 重启。

我在解决方案中有两个 Windows 窗体项目。其中一个加载调试信息，一个不加载。它们都引用了我试图在项目文件中以完全相同的方式获取调试信息的程序集。有任何想法吗？

我想在这里补充一点，主要是为了我自己，当我回来复习这个问题时，直到加载程序集才加载符号，直到需要时才加载程序集。如果断点位于仅在主程序集中的一个函数中使用的库中，则在调用该函数之前不会加载符号（并且它将显示断点未命中）。



解答

开始调试，一旦到达断点或使用Debug > Break All，请使用Debug > Windows > Modules。您将看到加载到进程中的所有程序集的列表。找到您要获取其调试信息的那个。右键单击它并选择符号加载信息。您将看到一个对话框，其中列出了它在其中查找程序集的 .pdb 文件的所有目录。根据实际的 .pdb 位置验证该列表。确保它找不到旧的。

在正常项目中，程序集及其 .pdb 文件应该始终由 IDE 复制到与 .exe 相同的文件夹中，即项目的 bin\Debug 文件夹中。如果您一直在使用它，请确保从 GAC 中删除它。


更多建议

检查以确保您未处于发布状态，而是处于调试状态。

调试时：

首先尝试通过右键单击项目>重建来重建项目如果这不起作用，请尝试清理项目（右键单击项目>清理）

如果那不起作用，请检查以下内容：

  1. 鼠标右键单击您的 项目
  2. 选择 【属性】
  3. 选择 [构建] 选项卡
  4. 确保选中 [Define DEBUG constant] 和 [Define TRACE constant]
  5. 确保未选中 [优化代码]
  6. 点击Build标签页底部的 【Advanced】按钮
  7. 确保 [Debug Info:] 设置为 [full]
  8. 单击 [确定] 并重建 项目 ;-)

（第 7 步生成 .pdb 文件，这些是调试符号）
