按多列对数据框行进行排序（排序）

Arya

2023-02-17 02:04

我想按多列对数据框进行排序。例如，对于下面的数据框，我想按“z”列（降序）排序，然后按“b”列（升序）排序：

dd <- data.frame(b = factor(c("Hi", "Med", "Hi", "Low"), 
      levels = c("Low", "Med", "Hi"), ordered = TRUE),
      x = c("A", "D", "A", "C"), y = c(8, 3, 9, 9),
      z = c(1, 1, 1, 2))
dd
    b x y z
1  Hi A 8 1
2 Med D 3 1
3  Hi A 9 1
4 Low C 9 2



解答

您可以直接使用该order()功能，而无需求助于附加工具——请参阅这个更简单的答案，它使用了代码顶部的一个技巧example(order)：

R> dd[with(dd, order(-z, b)), ]
    b x y z
4 Low C 9 2
2 Med D 3 1
1  Hi A 8 1
3  Hi A 9 1

2 年多后编辑： 刚刚有人问如何按列索引执行此操作。答案是简单地将所需的排序列传递给函数order()：

R> dd[order(-dd[,4], dd[,1]), ]
    b x y z
4 Low C 9 2
2 Med D 3 1
1  Hi A 8 1
3  Hi A 9 1
R>

而不是使用列的名称（并且with()更容易/更直接地访问）。

<select>使用 jQuery 从 JavaScript 对象向 a 添加选项的最佳方法是什么？

我正在寻找不需要插件的东西，但我也对现有的插件感兴趣。

这就是我所做的：

selectValues = { "1": "test 1", "2": "test 2" };

for (key in selectValues) {
  if (typeof (selectValues[key] == 'string') {
    $('#mySelect').append('<option value="' + key + '">' + selectValues[key] + '</option>');
  }
}

一个干净/简单的解决方案：

这是matdumsa 的清理和简化版本：

$.each(selectValues, function(key, value) {
     $('#mySelect')
          .append($('<option>', { value : key })
          .text(value));
});

matdumsa 的更改：(1) 删除了 append() 内选项的关闭标记，以及 (2) 将属性/特性移动到映射中作为 append() 的第二个参数。



解答

与其他答案一样，以 jQuery 方式：

$.each(selectValues, function(key, value) {   
     $('#mySelect')
         .append($("<option></option>")
                    .attr("value", key)
                    .text(value)); 
});


更多建议

var output = [];

$.each(selectValues, function(key, value)
{
  output.push('<option value="'+ key +'">'+ value +'</option>');
});

$('#mySelect').html(output.join(''));

通过这种方式，您只需“触摸 DOM”一次。

我不确定最新的行是否可以转换为 $('#mySelect').html(output.join('')) 因为我不知道 jQuery 内部结构（也许它在 html() 中做了一些解析方法）

并发性和并行性有什么区别？

Arya

2023-02-17 02:03

并发性和并行性有什么区别？



解答

并发是指两个或多个任务可以在重叠的 时间段内 开始、运行和完成。这并不一定意味着它们会 同时 运行。例如，在单核机器上 进行多任务处理。

并行 是指任务 实际上 同时运行，例如，在多核处理器上。

引用Sun 的 多线程编程指南 ：

  • 并发：当至少有两个线程取得进展时存在的条件。一种更通用的并行形式，可以将时间分片作为一种虚拟并行形式包括在内。

  • 并行性：当至少有两个线程同时执行时出现的情况。


更多建议

为什么存在混乱

存在混淆是因为这两个词的字典含义几乎相同：

  • 并发 ：同时存在、发生或完成（dictionary.com）
  • 并行 ：非常相似并且经常同时发生（韦氏词典）。

然而，它们在计算机科学和编程中的使用方式却大不相同。这是我的解释：

  • 并发性 ：可中断性
  • 并行性 ：独立性

那么上面的定义是什么意思呢？

我将用现实世界的类比来澄清。假设您必须在一天内完成 2 项非常重要的任务：

  1. 拿到护照
  2. 完成演示

现在，问题是任务 1 要求你去一个极其官僚的政府办公室，让你排队等候 4 个小时才能拿到护照。同时，task-2是你办公室需要的，是一个critical task。两者都必须在特定的一天完成。

案例 1：顺序执行

通常情况下，你会开车去护照办公室 2 小时，排队等候 4 小时，完成任务，开车回来 2 小时，回家，再睡 5 小时并完成演示。

案例 2：并发执行

但你很聪明。你提前计划。您随身携带一台笔记本电脑，在排队等候时开始处理您的演示文稿。这样一来，您回到家后只需多工作 1 小时，而不是 5 小时。

在这种情况下，这两项任务都由您完成，只是部分完成。您在排队等候时中断了护照任务并开始进行演示。当您的号码被呼叫时，您中断了演示任务并切换到护照任务。由于这两项任务的可中断性，基本上可以节省时间。

并发，IMO，可以理解为ACID中的“隔离”属性。如果子事务可以以每个和任何交错方式执行，并且最终结果与两个任务按顺序完成相同，则两个数据库事务被认为是隔离的。请记住，对于护照和出示任务， 您是唯一的刽子手 。

案例 3：并行执行

现在，你这么聪明，明明是上级，还有个助理。所以，在你离开开始护照任务之前，你打电话给他并告诉他准备演示文稿的初稿。你花了一整天的时间完成护照任务，回来查看你的邮件，然后你找到了演示文稿草稿。他的工作非常扎实，经过 2 小时的一些编辑，您完成了它。

现在，你的助手和你一样聪明，他可以 独立 完成它，而不需要不断地要求你澄清。 因此，由于任务的独立性，它们是由两个不同的刽子手 同时执行的。

还在我这儿？ 好吧...

案例 4：并发但不并行

还记得你的护照任务，你必须在那里排队等候吗？因为是 你的 护照，所以你的助理不能排队等候你。因此，护照任务具有 可中断性 （您可以在排队时停止它，稍后在呼叫您的号码时恢复它），但没有 独立性 （您的助手不能代替您等待）。

案例 5：并行但不并发

假设政府办公室有进入场所的安全检查。在这里，你必须移除所有电子设备并将它们提交给官员，他们只会在你完成任务后归还你的设备。

在这种情况下，护照任务既不可 独立 也 不可中断 。即使您在排队等候，您也无法从事其他工作，因为您没有必要的设备。

同样，假设演示文稿在本质上是高度数学化的，您需要至少 5 小时 100% 的注意力。即使您随身携带笔记本电脑，也不能在排队等候护照任务时这样做。

在这种情况下，演示任务是 独立的 （您或您的助手可以投入 5 小时的专注力），但不可 中断 。

案例 6：并发和并行执行

现在，假设除了分配您的助手进行演示之外，您还随身携带一台笔记本电脑来执行护照任务。在排队等候时，您看到您的助手已在共享幻灯片中创建了前 10 张幻灯片。您发送对他的工作的评论并进行一些更正。稍后，当你回到家时，你只需要 15 分钟而不是 2 小时来完成草稿。

这是可能的，因为演示任务具有 独立性 （你们中的任何一个都可以完成）和可 中断性 （您可以停止它并稍后恢复）。因此，您同时执行了 这两个 任务，并并行执行了演示任务。

比方说，除了过于官僚之外，政府办公室还很腐败。因此，你可以出示你的身份证明，输入它，开始排队等候你的号码被叫，贿赂一名警卫和其他人让你在队伍中保持你的位置，偷偷溜出去，在你的号码被叫之前回来，然后继续等待你自己。

在这种情况下，您可以同时并行执行护照和出示任务。你可以偷偷溜出去，你的位置由你的助手担任。然后你们俩都可以处理演示文稿等。

回到计算机科学

在计算世界中，以下是每种情况的典型示例场景：

  • 案例一： 中断处理。
  • 情况 2： 当只有一个处理器时，但所有正在执行的任务由于 I/O 而有等待时间。
  • 案例 3： 经常在我们谈论 map-reduce 或 hadoop 集群时看到。
  • 案例四： 我觉得案例四比较少见。任务并发但不并行的情况并不常见。但它 有可能 发生。例如，假设您的任务需要访问只能通过处理器 1 访问的特殊计算芯片。因此，即使处理器 2 空闲并且处理器 1 正在执行其他任务，特殊计算任务也无法在处理器 2 上进行。
  • 情况 5： 也很少见，但不如情况 4 少见。非并发代码可以是受互斥体保护的关键区域。一旦启动，它 必须 执行到完成。但是，两个不同的关键区域可以在两个不同的处理器上同时进行。
  • 案例 6： IMO，大多数关于并行或并发编程的讨论基本上都是在谈论案例 6。这是并行和并发执行的混合和匹配。

并发和围棋

如果你明白为什么 Rob Pike 说并发性更好，你就必须明白原因是什么。你有一个非常长的任务，其中有多个等待期，你等待一些外部操作，如文件读取、网络下载。在他的演讲中，他所说的只是，“把这个长时间的顺序任务打散，这样你就可以在等待的时候做一些有用的事情。” 这就是为什么他谈论不同的组织和不同的地鼠。

现在 Go 的优势来自于使用go关键字和渠道使这种突破变得非常容易。此外，在运行时有很好的底层支持来调度这些 goroutines。
