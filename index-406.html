将 Git 子模块更新为源上的最新提交

Arya

2023-02-17 06:07

我有一个带有 Git 子模块的项目。它来自 ssh://... URL，并且在提交 A 上。提交 B 已被推送到该 URL，我希望子模块检索提交并更改为它。

现在，我的理解是git submodule update应该这样做，但事实并非如此。它什么都不做（没有输出，成功退出代码）。这是一个例子：

$ mkdir foo
$ cd foo
$ git init .

$ git submodule add ssh://user@host/git/mod mod
Cloning into mod...
user@host's password: hunter2
remote: Counting objects: 131, done.
remote: Compressing objects: 100% (115/115), done.
remote: Total 131 (delta 54), reused 0 (delta 0)
Receiving objects: 100% (131/131), 16.16 KiB, done.
Resolving deltas: 100% (54/54), done.
$ git commit -m "Hello world."
[master (root-commit) 565b235] Hello world.
 2 files changed, 4 insertions(+), 0 deletions(-)
 create mode 100644 .gitmodules
 create mode 160000 mod
# At this point, ssh://user@host/git/mod changes; submodule needs to change too.
$ git submodule init
Submodule 'mod' (ssh://user@host/git/mod) registered for path 'mod'
$ git submodule update

$ echo $?

$ git submodule update mod
$ ...

我也试过git fetch mod，它似乎进行了提取（但不可能，因为它没有提示输入密码！），但git log否认git show存在新的提交。到目前为止，我一直在rm-ing 模块并重新添加它，但这在原则上是错误的，在实践中也很乏味。



解答

该git submodule update命令实际上告诉 Git，您希望您的子模块每次检查已在超级项目索引中指定的提交。如果你想将你的子模块 更新 到远程可用的最新提交，你需要直接在子模块中执行此操作。

所以总结一下：

# Get the submodule initially
git submodule add ssh://bla submodule_dir
git submodule init

# Time passes, submodule upstream is updated
# and you now want to update

# Change to the submodule directory
cd submodule_dir

# Checkout desired branch
git checkout master

# Update
git pull

# Get back to your project root
cd ..

# Now the submodules are in the state you want, so
git commit -am "Pulled down update to submodule_dir"

或者，如果您是一个忙碌的人：

git submodule foreach git pull origin master


更多建议

Git 1.8.2 提供了一个新选项 ，--remote它将完全启用此行为。跑步

git submodule update --remote --merge

将从每个子模块中的上游获取最新更改，将它们合并，并检查子模块的最新版本。正如文档所说：

- 偏僻的

该选项仅对更新命令有效。不要使用超级项目记录的 SHA-1 来更新子模块，而是使用子模块的远程跟踪分支的状态。

这相当于在每个子模块中运行git pull <remote> <default_branch>（通常git pull origin master是 or git pull origin main），这通常正是您想要的。

INNER JOIN ON 与 WHERE 子句

Arya

2023-02-17 06:07

为简单起见，假设所有相关字段都是NOT NULL。

你可以做：

SELECT
    table1.this, table2.that, table2.somethingelse
FROM
    table1, table2
WHERE
    table1.foreignkey = table2.primarykey
    AND (some other conditions)

要不然：

SELECT
    table1.this, table2.that, table2.somethingelse
FROM
    table1 INNER JOIN table2
    ON table1.foreignkey = table2.primarykey
WHERE
    (some other conditions)

这两个在中以相同的方式工作吗MySQL？



解答

INNER JOIN是您应该使用的 ANSI 语法。

它通常被认为更具可读性，尤其是当您连接大量表时。

OUTER JOIN它也可以在需要时轻松替换。

语法WHERE更面向关系模型。

两个表 ed 的结果JOIN是应用过滤器的表的笛卡尔积，该过滤器仅选择连接列匹配的那些行。

使用语法更容易看出这一点WHERE。

对于您的示例，在 MySQL（通常在 SQL 中）这两个查询是同义词。

另外，请注意 MySQL 也有一个STRAIGHT_JOIN子句。

使用此子句，您可以控制JOIN顺序：外循环中扫描哪个表，内循环中扫描哪个表。

您无法使用语法在 MySQL 中控制它WHERE。


更多建议

其他人指出这INNER JOIN有助于提高人类的可读性，我同意这是重中之重。
让我试着解释一下 为什么 join 语法更具可读性。

一个基本的SELECT查询是这样的：

SELECT stuff
FROM tables
WHERE conditions

该SELECT子句告诉我们要返回 什么； 该FROM子句告诉我们我们从 哪里 得到它，而该WHERE子句告诉我们我们得到了哪些 。

JOIN是关于表的声明，它们是如何绑定在一起的（概念上，实际上，成一个表）。

任何控制表的查询元素——我们从哪里获取东西——在语义上都属于子句FROM（当然，这就是JOIN元素所在的位置）。将连接元素放入子句中会将 which 和 where-fromWHERE混为一谈，这就是首选语法的原因。 * *JOIN

git reset --mixed、--soft 和 --hard 之间有什么区别？

Arya

2023-02-17 06:06

我正在寻找拆分提交但不确定要使用哪个重置选项。

我在看页面用简单的英语，“git reset”是做什么的？，但我意识到我并不真正理解 git 索引或暂存区是什么，因此解释没有帮助。

--mixed此外，在该答案中，和的用例--soft对我来说是一样的（当你想修复并重新提交时）。有人可以进一步分解吗？我意识到--mixed可能是可以选择的选项，但我想知道 为什么 。最后呢--hard？

有人可以给我一个工作流程示例，说明如何选择 3 个选项吗？



解答

当您修改存储库中的文件时，更改最初是未暂存的。为了提交它，您必须暂存它——即将它添加到索引——使用git add. 当您进行提交时，提交的更改是那些已添加到索引中的更改。

git reset至少更改当前分支 ( HEAD) 指向的位置。--mixed和之间的区别--soft在于您的索引是否也被修改。所以，如果我们在master这一系列提交的分支上：

- A - B - C (master)

HEAD指向C并且索引匹配C。

当我们运行时git reset --soft B，master（因此HEAD）现在指向B，但索引仍然有变化C；git status将显示它们已上演。因此，如果我们git commit此时运行，我们将获得一个新的提交，其更改与C.

好的，那么从这里重新开始：

- A - B - C (master)

现在让我们开始吧git reset --mixed B。（注意：--mixed是默认选项）。再次，master并HEAD指向 B，但这次索引也被修改为匹配B。如果我们git commit在此时运行，因为索引匹配，所以什么也不会发生HEAD。我们仍然在工作目录中进行更改，但由于它们不在索引中，因此git status将它们显示为未暂存。要提交它们，您需要git add像往常一样提交。

最后，与（它更改您的和索引）--hard相同，除了它还修改了您的工作目录。如果我们在并运行，那么在 中添加的更改以及您拥有的任何未提交的更改都将被删除，并且您的工作副本中的文件将与提交相匹配。由于您可能会以这种方式永久丢失更改，因此您应该始终在进行硬重置之前先运行，以确保您的工作目录是干净的，或者您可以接受丢失未提交的更改。--mixed``HEAD``--hard``C``git reset --hard B``C``B``git status

更多建议

用最简单的话来说：

  • --soft : 取消提交 更改，更改被暂存（ index ）。
  • --mixed （默认） ： 取消提交 + 取消暂存 更改，更改保留在 工作树 中。
  • --hard : uncommit + unstage + delete 更改，什么都不剩。
