为什么 Java Vector（和 Stack）类被认为已过时或已弃用？

Arya

2023-02-17 16:15

为什么 Java Vector 被视为遗留类、已过时或已弃用？

使用并发时它的使用是否有效？

如果我不想手动同步对象而只想使用线程安全集合而不需要制作底层数组的新副本（就像那样CopyOnWriteArrayList），那么可以使用吗Vector？

怎么样Stack，它是 的子类Vector，我应该用什么来代替它？



解答

Vector在每个单独的操作上同步。那几乎从来都不是您想要做的。

通常，您希望同步整个操作 序列。 同步单个操作既不太安全（例如，如果你迭代 a Vector，你仍然需要取出一个锁以避免其他人同时更改集合，这会导致ConcurrentModificationException迭代线程中的 a ）但也更慢（一次就够了，为什么要反复取出一把锁）？

当然，即使您不需要，它也有锁定的开销。

基本上，在大多数情况下，这是一种非常有缺陷的同步方法。正如 Brian Henk 先生 指出的那样，您可以使用调用来装饰集合，例如-将“调整大小的数组”集合实现与“同步每个操作”位结合起来的Collections.synchronizedList事实是糟糕设计的另一个例子；Vector装饰方法提供了更清晰的关注点分离。

至于Stack等价物 - 我会看看Deque/ArrayDeque开始。


更多建议

Vector 是 1.0 的一部分——最初的实现有两个缺点：

1. 命名： 向量实际上只是可以作为数组访问的列表，所以它应该被调用ArrayList（这是 Java 1.2 Collections 的替代品Vector）。

2. 并发： 所有的get(),set()方法都是synchronized，所以你不能对同步进行细粒度的控制。

ArrayList和之间没有太大区别Vector，但你应该使用ArrayList.

来自 API 文档。

从 Java 2 平台 v1.2 开始，这个类被改造以实现 List 接口，使其成为 Java Collections Framework 的成员。与新的集合实现不同，Vector 是同步的。

Array 与 List<T>：何时使用哪个？

Arya

2023-02-17 16:14 

MyClass[] array;
List<MyClass> list;

当一种情况比另一种更可取时，有哪些情况？为什么？



解答

实际上，您很少会想要使用数组。List<T>任何时候你想添加/删除数据时一定要使用 a ，因为调整数组的大小很昂贵。 如果您知道数据是固定长度的，并且出于某些非常具体的 原因（在基准测试之后）想要进行微优化，那么数组可能会有用。

List<T>提供比数组更多的功能（尽管 LINQ 使它更均匀一点），并且几乎总是正确的选择 。params当然，争论除外。;-p

作为计数器 -List<T>是一维的；where-as you have rectangular (etc) arrays like int[,]or string[,,]- 但是还有其他方法可以在对象模型中对此类数据进行建模（如果需要）。

也可以看看：

  • 如何/何时放弃在 c#.net 中使用数组？
  • 数组，有什么意义？

也就是说，我在我的protobuf-net项目中 大量 使用了数组；完全为了性能：

  • 它做了很多位移，所以 abyte[]对于编码来说非常重要；
  • 我使用本地滚动byte[]缓冲区，在向下发送到底层流（和 vv）之前填充它；比等快BufferedStream；
  • 它在内部使用基于数组的对象模型（Foo[]而不是List<Foo>），因为大小一旦构建就固定了，并且需要非常快。

但这绝对是个例外；对于一般的业务处理，List<T>每次都赢。

“超级”在 Python 中做什么？- super().__init__() 和显式超类 __init__() 之间的区别

Arya

2023-02-17 16:14

有什么区别：

class Child(SomeBaseClass):
    def __init__(self):
        super(Child, self).__init__()

和：

class Child(SomeBaseClass):
    def __init__(self):
        SomeBaseClass.__init__(self)

我已经看到super在只有单一继承的类中使用了很多。我明白你为什么要在多重继承中使用它，但不清楚在这种情况下使用它有什么好处。



解答

单继承的好处super()很小——主要是，您不必将基类的名称硬编码到使用其父方法的每个方法中。

但是，如果没有 . ，使用多重继承几乎是不可能的super()。这包括常见的习语，如混合、接口、抽象类等。这扩展到以后扩展你的代码。如果以后有人想写一个扩展的类Child和一个 mixin，他们的代码将无法正常工作。
