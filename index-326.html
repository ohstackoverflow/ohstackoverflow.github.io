如何使用 PHP 清理用户输入？

Arya

2023-02-17 04:06

是否有一个 catchall 函数可以很好地为 SQL 注入和 XSS 攻击清理用户输入，同时仍然允许某些类型的 HTML 标记？



解答

用户输入可以被过滤是一个常见的误解。PHP 甚至有一个（现已弃用）“功能”，称为magic- quotes，它建立在这个想法之上。这是胡说八道。忘掉过滤（或清洁，或人们所说的任何东西）。

为避免出现问题，您应该做的非常简单：无论何时在外部代码中嵌入一段数据，都必须根据该代码的格式规则处理它。但您必须明白，此类规则可能过于复杂，无法尝试手动遵循它们。例如，在 SQL 中，字符串、数字和标识符的规则都是不同的。为了您的方便，在大多数情况下，有一个专门用于此类嵌入的工具。例如，当您需要在 SQL 查询中使用 PHP 变量时，您必须使用准备好的语句，它将处理所有正确的格式/处理。

另一个例子是 HTML：如果在 HTML 标记中嵌入字符串，则必须使用htmlspecialchars. 这意味着每个echoorprint语句都应该使用htmlspecialchars.

第三个示例可能是 shell 命令：如果您要将字符串（例如参数）嵌入到外部命令中，并使用 调用它们，exec那么您必须使用escapeshellcmdand escapeshellarg。

此外，一个非常引人注目的例子是 JSON。规则如此众多和复杂，以至于您永远无法手动遵循它们。这就是为什么你永远不应该手动创建 JSON 字符串，而应该始终使用专用函数来json_encode()正确格式化每一位数据。

等等等等 ...

您需要主动过滤数据的唯一情况是您接受预先格式化的输入 。 例如，如果您让您的用户发布您计划在网站上显示的 HTML 标记。但是，您应该明智地不惜一切代价避免这种情况，因为无论您如何过滤它，它始终是一个潜在的安全漏洞。


更多建议

不要试图通过清理输入数据来防止 SQL 注入。

相反， 不允许在创建 SQL 代码时使用数据 。使用使用绑定变量的准备语句（即在模板查询中使用参数）。这是唯一可以防止 SQL 注入的方法。

有关防止 SQL 注入的更多信息，请参阅我的网站http://bobby-tables.com/。

Windows 版 Git 中的文件名太长

Arya

2023-02-17 04:05

我正在使用Git-1.9.0-preview20140217Windows。据我所知，此版本应解决文件名过长的问题。但不适合我。

当然我做错了什么：我做了git config core.longpaths true然后git add .然后git commit。一切顺利。但是当我现在做一个时git status，我得到一个文件列表Filename too long，例如：

node_modules/grunt-contrib-imagemin/node_modules/pngquant- bin/node_modules/bin- wrapper/node_modules/download/node_modules/request/node_modules/form- data/node_modules/combined-stream/node_modules/delayed- stream/test/integration/test-handle-source-errors.js: Filename too long

对我来说重现非常简单：只需使用 Angular 生成器（“yo angular”）创建一个Yeomannode_modules Web 应用程序并从.gitignore文件中删除。然后重复前面提到的 Git 命令。

我在这里错过了什么？



解答

Git 的文件名限制为 4096 个字符，但在 Windows 上，当 Git 使用 msys 编译时除外。它使用旧版本的 Windows API，文件名有 260 个字符的限制。

据我所知，这是 msys 的限制，而不是 Git 的限制。您可以在此处阅读详细信息：https ://github.com/msysgit/git/pull/110

您可以通过在 Windows 上使用另一个 Git 客户端或设置为如其他答案中所述来规避此core.longpaths问题true。

git config --system core.longpaths true

Git 是作为脚本和编译代码的组合构建的。通过上述更改，某些脚本可能会失败。这就是默认情况下不启用 core.longpaths 的原因。

Windows 文档位于https://learn.microsoft.com/en-us/windows/win32/fileio/maximum- file-path-limitation?tabs=cmd#enable-long-paths-in-windows-10-version- 1607 及更高版本有更多信息：

从 Windows 10 版本 1607 开始，MAX_PATH 限制已从常见的 Win32 文件和目录函数中删除。但是，您必须选择加入新行为。

注册表项允许您启用或禁用新的长路径行为。要启用长路径行为，请将注册表项设置为 HKLM\SYSTEM\CurrentControlSet\Control\FileSystem\LongPathsEnabled（类型：REG_DWORD）


更多建议

您应该能够运行命令

git config --system core.longpaths true

或手动将其添加到您的 Git 配置文件之一以打开此功能，一旦您使用受支持的 Git 版本。看起来可能是 1.9.0 及之后的版本。

master 分支和'origin/master' 有分歧，如何'undiverge' branches'？

Arya

2023-02-17 04:05

不知何故，我master和我的origin/master分支机构出现了分歧。
我其实不想让他们分道扬镳。

我如何查看这些差异并 合并 它们？



解答

您可以使用以下方法查看差异：

git log HEAD..origin/main

# old repositories
git log HEAD..origin/master

在拉取它之前（获取 + 合并）（另请参阅“如何让 git 始终从特定分支中拉取？”）

注意：从Git 2.28 (Q3 2020)开始，默认分支是可配置的，现在（2021+）设置为main，不再是master。
其余答案反映了最近的惯例。

当您收到如下消息时：

“你的分支和‘origin/main’已经分歧，#并且分别有 1 和 1 个不同的提交。”

检查是否需要更新origin。
如果origin是最新的，那么origin当您在本地进行自己的提交时，一些提交已被推送到另一个仓库。

... o ---- o ---- A ---- B  origin/main (upstream work)
                   \
                    C  main(your work)

您基于提交C进行提交，因为那是您当时从中A获取的最新工作。upstream

但是，在您尝试推回之前origin，其他人推送了提交B。
发展历史已经分叉成不同的路径。

然后您可以合并或变基。有关详细信息，请参阅Pro Git：Git 分支 - 变基。

合并

使用git merge命令：

$ git merge origin/main

# old repositories
$ git merge origin/master

这告诉 Git 将更改集成origin/main到您的工作中并创建合并提交。
历史图表现在看起来像这样：

... o ---- o ---- A ---- B  origin/main (upstream work)
                   \      \
                    C ---- M  main (your work)

新的合并 commitM有 两个 父项，每个父项代表一条导致存储在该提交中的内容的开发路径。

请注意，背后的历史M现在是非线性的。

变基

使用git rebase命令：

$ git rebase origin/main

# old repositories
$ git rebase origin/master

这告诉 Git 重播提交C（你的工作），就好像你是基于提交B而不是A. CVS 和 Subversion 用户在提交前更新时，
通常会在工作之上重新设置本地更改的基础。Git 只是在提交和变基步骤之间添加了明确的分隔。upstream

历史图表现在看起来像这样：

... o ---- o ---- A ---- B  origin/main (upstream work)
                          \
                           C'  main (your work)

CommitC'是由命令创建的新提交git rebase。它在两个方面
有所不同：C

  1. 它有不同的历史：B而不是A.
  2. 它的内容说明了 和 的B变化C；M它与合并示例中的相同。

请注意，后面的历史C'仍然是线性的。
我们选择（目前）只允许cmake.org/cmake.git.
这种方法保留了以前使用的基于 CVS 的工作流程，并且可以简化过渡。
尝试推C'送到我们的存储库将起作用（假设您有权限并且在您变基时没有人推送）。

该git pull命令提供了一种fetch从origin和rebase本地工作的简写方式：

$ git pull --rebase

这将上述内容fetch和rebase步骤合并为一个命令。


更多建议

即使在阅读了上述回复之后，我也遇到了这个问题，并且对造成它的原因感到困惑。我的解决方案是

git reset --hard origin/master

然后，这只是将我的（本地）master 副本（我假设它搞砸了）重置为正确的点，如（远程）origin/master 所示。

警告 ：您将丢失所有尚未推送到的更改origin/master。
